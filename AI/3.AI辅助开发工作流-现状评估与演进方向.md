# AI 辅助开发工作流 — 现状评估与演进方向

## 一、当前流程概述

我们采用 Issues-Driven Development 模式，基于 Claude Code Skills 构建了三阶段标准化流程：

```
/plan <需求>  →  /plan-to-issues <plan路径>  →  /execute-issues <csv路径>
     ↓                    ↓                           ↓
 多轮讨论收敛         Issues CSV                  TDD 连续执行
 .dev/plans/*.md       [人工审批]                  [人工验收]
  [人工审批]                                     人工 git commit
```

核心原则：**结构化文件驱动 AI 开发，每一步人工验收，测试先行。**

---

## 二、现有优势

### 2.1 CSV 即合同 — 简单、透明、可编辑

CSV 是当前设计的核心亮点。12 列结构清晰，人人能看、能改、能用 Excel 打开。`acceptance_criteria` + `test_cases` 字段在 Plan 阶段就锁定了测试边界，比自然语言对话可靠得多。

相比 JSON + Markdown + JSONL 的多文件组合方案，CSV 的认知成本更低，协作门槛也更低。

### 2.2 三次人工卡点 — 控制感强

Plan 审批 → CSV 审批 → 交付验收，每个阶段都可以打回重来，人始终掌握决策权。更自动化的流程虽然效率高，但如果 Agent 管线某一步跑偏，发现时可能已经做了不少无用功。当前的三卡点设计在可控性和效率之间取了合理的平衡。

### 2.3 中断恢复

通过 CSV 中的 `dev_state` / `test_state` 字段实现断点续跑，重新执行 `/execute-issues` 时自动跳过已完成的 issue。这是务实的工程设计，避免了因会话中断导致的重复劳动。

### 2.4 Plan 阶段的双向讨论

明确要求"AI 会 challenge 你的想法，你也应该 challenge 它的方案"，并且测试策略在 Plan 阶段就要敲定。Plan 不是让 AI 单方面出方案，而是真正的技术讨论。这比直接让 AI 生成计划然后人工审批的单向流程更能暴露问题。

### 2.5 轻量级，易上手

整个流程仅 3 个 Skill + 2 个辅助脚本，没有复杂的配置和依赖。新成员只需读一遍 workflow-guide 即可参与，不需要学习框架概念。

## 三、现有不足

### 3.1 上下文注入是被动的

当前所有项目规范写在 CLAUDE.md 中，依赖 AI 主动读取。但 AI 可能忽略其中的某些规范，尤其在长对话中容易"上下文漂移"。

**问题本质**：规范是"可选阅读"而非"强制注入"，AI 的行为因此缺乏确定性。

**改进方向**：利用 Claude Code 的 Hook 机制（`SessionStart`、`PreToolUse`），在会话开始和关键操作前自动注入相关规范和项目状态。把"AI 应该看"变成"AI 必须看"。

### 3.2 没有跨会话记忆

每次新对话，AI 都是从零开始。上一次会话中讨论过的决策、遇到的坑、发现的模式全部丢失。

**问题本质**：会话是一次性的，知识不能跨会话累积。

**改进方向**：在 `.dev/` 下建立 journal 机制，每次会话结束时记录关键决策和发现。下次会话开始时通过 Hook 自动加载，AI 不再"失忆"。

### 3.3 质量门禁不够强制

`/execute-issues` 阶段执行 pytest，但如果测试失败，AI 记个 notes 就可以跳过继续下一条。没有强制机制确保测试必须通过。

**问题本质**：测试是"建议通过"而非"必须通过"，质量把关靠 AI 自觉。

**改进方向**：引入 `SubagentStop` Hook 实现强制质量门禁 — 测试不通过则阻止 Agent 标记完成，强制继续修复。可设置最大重试次数和超时时间，避免死循环。

### 3.4 上下文加载是全量的

CLAUDE.md 是 all-in-one 文件，AI 每次都读取所有内容。随着项目规模增长，这个文件会越来越大，但每次任务实际只需要其中一部分。

**问题本质**：没有分层上下文机制，全量加载浪费上下文窗口。

**改进方向**：
- 将规范拆分为独立文件，按领域组织（如 `.dev/spec/backend/`、`.dev/spec/frontend/`）
- 用索引文件记录每个任务需要加载哪些 spec
- 扩展 CSV 的 `refs` 字段，不仅引用代码文件，也引用规范文档

### 3.5 单一 Agent 全程执行

当前是同一个 Agent 贯穿调研、编码、测试全过程。好处是简单，但在复杂任务中，同一个上下文里混合了多种行为，容易上下文漂移。

**问题本质**：一个 Agent 承担多个角色，缺乏职责隔离。

**改进方向**：对复杂任务考虑引入专业化子 Agent（如 research Agent 只读不写、implement Agent 禁止 git、check Agent 专注代码审查），各有明确边界。简单任务仍可单 Agent 执行，按复杂度动态选择。

### 3.6 规范缺少系统化的更新流程

CLAUDE.md 中有"常见陷阱与约束"，但没有机制鼓励在开发过程中将新发现的模式和踩过的坑写回规范。

**问题本质**：规范是静态的，不能随开发实践自动演进。

**改进方向**：增加 `/update-spec` 类型的命令，开发完成后引导将经验写回 spec 目录。一个人的最佳实践应该惠及全团队。

## 四、演进路线建议

按投入产出比排序：

| 优先级 | 改进项 | 投入 | 预期收益 |
|--------|--------|------|----------|
| **P0** | 引入 SessionStart Hook，自动注入项目状态和核心规范 | 小 | AI 行为确定性显著提升 |
| **P0** | 强化质量门禁，测试不通过不允许标记完成 | 小 | 交付质量提升 |
| **P1** | 增加会话记忆（journal 机制） | 中 | 跨会话连续性，减少重复沟通 |
| **P1** | 规范拆分 + 分层上下文加载 | 中 | 上下文窗口利用率提升 |
| **P2** | Spec 目录 + 更新流程 | 中 | 团队知识沉淀 |
| **P2** | 复杂任务的多 Agent 分工 | 大 | 长任务执行质量提升 |

---

## 五、总结

当前流程作为初步探索，已经抓住了核心思路：**结构化文件驱动 + 人工卡点 + 测试先行**。CSV 合同机制、三次人工审批、双向讨论的 Plan 阶段都是合理的设计。

主要短板集中在**自动化上下文管理**和**质量强制机制**两个方向。优先引入 Hook 注入和质量门禁（P0），就能以较小的投入获得显著的效果提升。后续的会话记忆、分层上下文、规范演进可以逐步推进，不需要一步到位。
