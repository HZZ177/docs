# AI 辅助开发工作流

## 定位

一套基于 Claude Code Skills 的标准化开发流程。核心思路：**用结构化文件驱动 AI 开发，每一步人工验收，测试先行。**

AI 负责调研、编码、自测；人负责需求定义、方案决策、最终自行验收和提交。

## 流程总览

```
/plan <需求描述>  →  /plan-to-issues <plan路径>  →  /execute-issues <csv路径>
      ↓                      ↓                            ↓
  多轮讨论收敛           Issues CSV                  TDD 连续执行
  .dev/plans/*.md         [人工审批]                    [人工验收]
   [人工审批]                                        人工 git commit
```

三个阶段，三次人工卡点。任何一步不满意都可以打回重来。

## 快速上手

### 第一步：规划（/plan）

```
/plan 实现新的数据导出功能，支持 Excel 和 CSV 两种格式
```

触发后 AI 会：
1. 调研项目代码，了解现状
2. 汇报发现，提出方案和测试思路，向你提问
3. **和你展开多轮讨论**——会 challenge 你的想法，你也应该 challenge 它的方案
4. 讨论中会一起敲定测试策略和用例
5. 双方达成共识后，你说「落盘」，AI 生成 Plan 文件到 `.dev/plans/`

**要点**：Plan 模式是讨论，不是让 AI 单方面出方案。测试方案在这一步就要定下来。

### 第二步：拆任务（/plan-to-issues）

```
/plan-to-issues .dev/plans/2026-02-13_data-export.md
```

AI 把 Plan 拆分为 Issues CSV，写入 `.dev/issues/`。

**你需要检查 CSV**：用 Excel 或文本编辑器打开，确认：
- 任务拆分粒度是否合理
- `acceptance_criteria` 验收标准是否明确
- `test_cases` 测试用例是否完整
- 优先级和分工是否正确

可以直接修改 CSV 内容，也可以让 AI 重新生成。

### 第三步：执行（/execute-issues）

```
/execute-issues .dev/issues/2026-02-13_data-export.csv
```

AI 按 CSV 连续执行所有 issue，每条遵循 TDD 流程：
1. 根据 `test_cases` 写测试 → 跑测试确认失败（红）
2. 写实现代码 → 跑测试确认通过（绿）
3. 自查后标记完成，继续下一条

全部完成后输出整体交付报告。

**你需要验收**：检查代码变更和测试结果，满意后自行 `git commit`。

## 中间产物说明

### 目录结构

```
.dev/
  plans/       Plan 文件（Markdown）
  issues/      Issues CSV（任务合同）
  reports/     调研报告（可选）

.claude/
  skills/
    plan/                  /plan Skill
    plan-to-issues/        /plan-to-issues Skill + 校验脚本
    execute-issues/        /execute-issues Skill + 进度脚本
```

### Plan 文件

Markdown 格式，带 YAML 头部，包含：任务概述、现状分析、关键决策、**测试策略**、执行计划、风险。

### Issues CSV

12 列，核心字段：

| 字段 | 说明 |
|------|------|
| `id` | 唯一标识，如 `EXP-010` |
| `priority` | `P0` / `P1` / `P2` |
| `title` | 一句话标题 |
| `description` | 做什么（不写怎么做） |
| `acceptance_criteria` | 验收标准，必须可测试可验证 |
| `test_cases` | 测试用例，Plan 阶段讨论确定，执行时据此写测试代码 |
| `area` | `backend` / `frontend` / `both` / `infra` |
| `refs` | 关联代码 `file:line`，多个用 `;` 分隔 |
| `dev_state` | `未开始` / `进行中` / `已完成` |
| `review_state` | `未开始` / `进行中` / `已完成` |
| `owner` | 负责人 |
| `notes` | 备注（阻塞原因、完成时间等） |

## 辅助工具

```bash
# 校验 CSV 格式是否合规
python .claude/skills/plan-to-issues/scripts/validate-csv.py .dev/issues/xxx.csv

# 查看当前执行进度
python .claude/skills/execute-issues/scripts/check-states.py .dev/issues/xxx.csv
```

## 常见场景

**执行中断了怎么办？**
重新运行 `/execute-issues <同一个csv路径>`，已完成的 issue 会自动跳过。

**验收时发现某条 issue 有问题？**
告诉 AI：「XXX-020 有问题，需要改一下 xxx」，AI 会单独重新执行那条。

**测试用例不够？**
直接修改 CSV 的 `test_cases` 字段，然后让 AI 重新执行对应 issue。

**想全部重来？**
告诉 AI「全部重跑」，所有状态重置后从头执行。

## 设计原则

- **测试先行（TDD）**：测试在 Plan 阶段就确定，执行时先写测试再写实现，测试定义需求边界
- **人工卡点**：每个阶段都有人工审批，AI 不自动提交代码
- **CSV 即合同**：结构化、可校验、可追溯，比自然语言对话可靠
- **支持中断恢复**：通过状态字段实现断点续跑
- **KISS**：先跑通再优化，流程本身也会随使用迭代
