# BI-Agent 持久记忆方案调研与分析

---

## 一、调研背景

通过档案系统 + 热启动的设计（详见《BI-Agent-热启动与档案系统设计》），我们已经规划了 **"AI 认识业务"** 的能力 —
预计算项目档案、组织结构树注入、按需加载指标数据。

但这只解决了一半问题。AI 仍然不认识**用户** — 不知道老板关注哪些项目、不记得上次讨论过什么、不了解老板的分析偏好和沟通风格。同时，Agent
在长期服务过程中积累的业务经验（如某类问题的常见归因模式、某个项目的历史分析结论）也无法沉淀和复用。

**档案系统解决"AI 对业务的认知"，持久记忆系统解决"AI 对用户的认知"和"AI 的经验积累"。**

---

## 二、记忆需求分层

| 层级                  | 内容                    | 状态             |
|---------------------|-----------------------|----------------|
| **Layer 1: 业务知识记忆** | 项目档案、集团报告、组织树         | 档案系统已规划，不在本次范围 |
| **Layer 2: 用户偏好记忆** | 关注的项目、常问的维度、决策偏好      | **本次调研重点**     |
| **Layer 3: 追踪事项记忆** | "帮我盯着项目A的利润"、"下周三出月报" | 建议结构化表独立设计     |
| **Layer 4: 交互历史记忆** | 跨会话的对话摘要和关键结论         | **本次调研重点**     |

---

## 三、多流派记忆框架

| 流派         | 代表产品                   | 核心思路                       |
|------------|------------------------|----------------------------|
| **智能提取派**  | Mem0、PowerMem、Memobase | LLM 从对话中提取事实 → 向量存储 → 语义召回 |
| **自编辑记忆派** | Letta (原 MemGPT)       | Agent 自主读写分层记忆，借鉴 OS 虚拟内存  |
| **知识图谱派**  | Graphiti (Zep)、Cognee  | 实体-关系图谱 + 时序推理             |
| **框架原生派**  | LangMem                | 与 LangGraph 深度绑定           |

经过初步筛选，**Mem0、PowerMem、LangMem** 三个方案最适合我们的技术栈和需求场景，以下做重点分析。

## 四、三个重点方案分析

### 4.1 Mem0 — 社区最成熟的方案

> 47.3k Stars | v1.0.3 稳定 | 2025.10 获 2400 万美元融资 | 默认 LLM: gpt-4.1-nano

#### 工作原理

**存记忆（add）**：

```
对话内容
  → [LLM调用1] 事实提取：从对话中抽取原子事实
      例: "我最近关注华东区项目A的利润" → ["用户关注华东区项目A的利润"]
  → [向量搜索] 对每条事实，在已有记忆中搜索语义相似条目
      找到: "用户关注项目A"（相似度0.87）
  → [LLM调用2] 记忆决策：将新事实 + 已有记忆交给 LLM 判断
      - ADD:    全新信息 → 创建新记忆
      - UPDATE: 补充/修正 → 合并更新（如 "关注项目A" → "关注华东区项目A的利润"）
      - DELETE: 信息矛盾 → 删除旧记忆
      - NOOP:   重复信息 → 跳过
  → [写入] 生成 embedding → 写入向量库 → 记录审计日志
```

每次 `add()` 固定 **2 次 LLM 调用**，与对话长度无关。可用 `infer=False` 跳过 LLM 直接存储原文。

**取记忆（search）**：

```
查询文本 → 向量化 → cosine similarity 近邻搜索 + metadata 过滤 → [可选 Reranking] → 返回结果
```

取记忆**不需要 LLM 调用**，只做向量搜索，速度很快。

#### 记忆生命周期管理

| 能力        | 机制                                       | 是否自动  |
|-----------|------------------------------------------|-------|
| **创建**    | `add()` 内部 LLM 提取事实后 ADD                 | ✅ 自动  |
| **去重**    | 向量相似度 + LLM 判断 → NOOP                    | ✅ 自动  |
| **更新**    | 新旧事实对比 → UPDATE（保留更完整的版本）                | ✅ 自动  |
| **矛盾解决**  | 已有"坐商务舱" + 新"改经济舱" → UPDATE 为"从商务舱改为经济舱" | ✅ 自动  |
| **衰减/过期** | ❌ 无内置衰减，记忆只增不减                           | 需手动清理 |
| **版本追踪**  | 所有变更记录到 SQLite 审计表，可查看完整变更链              | ✅ 自动  |

#### 粒度控制

- **提取粒度**：通过自定义 `FACT_RETRIEVAL_PROMPT` 精确控制从对话中提取什么类型的信息（如"
  只提取项目关注、分析偏好、约定事项，忽略闲聊"）
- **存储粒度**：metadata 标签打标，检索时按标签过滤，支持 eq/ne/gt/in/AND/OR 等操作符
- **隔离维度**：user_id / agent_id / session_id 三个维度，可组合使用

#### 注入时机

完全由开发者控制，三种模式：

- **被动注入**：在 ContextProvider 中每次对话启动时自动检索 Top-N 记忆注入 system prompt
- **主动调用**：封装为 LangChain Tool，Agent 自主决定何时检索
- **混合模式（推荐）**：被动注入高频记忆 + 主动工具深入检索

#### 优势与局限

**优势**：社区最大（47k stars）、文档完善、25+ 种向量库后端可选、API 简洁（add/search/update/delete）、LOCOMO 基准准确率比 OpenAI
Memory 高 26%、延迟低 91%、Token 省 90%+。

**局限**：**无记忆衰减** — 记忆只增不减，长期使用后召回精度下降，需自行清理；无用户画像；无知识图谱；仅单路向量搜索。

#### 代码示例

```python
from mem0 import Memory

# 初始化（可配置向量库、LLM等，此处使用默认配置）
config = {
    "llm": {"provider": "openai", "config": {"model": "gpt-4.1-nano"}},
    "vector_store": {"provider": "qdrant", "config": {"host": "localhost", "port": 6333}},
}
memory = Memory.from_config(config)

# 存记忆 — 自动提取事实、去重、合并
memory.add(
    messages=[{"role": "user", "content": "我最近重点关注华东区项目A的利润情况"}],
    user_id="boss_001",
    metadata={"category": "project_focus"},
)

# 取记忆 — 纯向量搜索，无 LLM 调用
results = memory.search("项目A的利润", user_id="boss_001", limit=5)
for r in results["results"]:
    print(r["memory"], r["score"])  # "用户重点关注华东区项目A的利润情况" 0.92

# 自定义提取 prompt — 控制提取什么类型的信息
custom_prompt = """从对话中仅提取以下类型的信息：
1. 用户关注的项目和指标
2. 用户的分析偏好和沟通风格
3. 用户约定的事项
忽略闲聊和无关信息。"""
memory.add(messages=messages, user_id="boss_001", prompt=custom_prompt)
```

### 4.2 PowerMem — 认知科学驱动的新锐

> 427 Stars | v0.5.2 | OceanBase 团队出品 | Apache 2.0
>
> LOCOMO 基准：准确率 78.70%（比 Mem0 高 48.77%），P95 延迟 1.44s，Token 消耗 0.9k

#### 工作原理

**存记忆（add）** — 在 Mem0 的基础管道上增加了额外步骤：

```
对话内容
  → [LLM调用1] 事实提取 ← 与 Mem0 相同
  → [向量搜索] 去重/冲突检测 ← 与 Mem0 相同
  → [LLM调用2] 行动决策 ADD/UPDATE/DELETE/NOOP ← 与 Mem0 相同
  → ===== 以下是 PowerMem 独有 =====
  → [重要性评估] LLM 对每条事实打分 0.0~1.0（考虑独特性、对用户画像的价值、时效性）
  → [三级分层] 按重要性分配层级：
        importance >= 0.8 → 长期记忆（标准衰减）
        importance 0.5~0.8 → 短期记忆（1.5x 加速衰减）
        importance < 0.5 → 工作记忆（2x 加速衰减）
  → [写入] 生成 embedding + 稀疏向量 → 注入 retention_score=1.0, access_count=0 → 存入向量库
  → [可选] 同步更新知识图谱 / 用户画像
```

**取记忆（search）** — 四路混合检索：

```
查询文本
  → [四路并行]
      ├── 密集向量搜索 — 语义相似度
      ├── 全文搜索 (BM25 + jieba分词) — 精确关键词匹配
      ├── 稀疏向量搜索 — 关键词语义混合
      └── 图检索 — 知识图谱多跳遍历 [可选]
  → [RRF 融合] Reciprocal Rank Fusion 合并四路结果
  → [可选 Reranking] Qwen / Jina / Zhipu AI 重排序
  → [艾宾浩斯衰减加权]
      combined_score = similarity_score × retention_score
      被访问的记忆触发强化（reinforcement_factor = 1.2x）
      retention < 0.3 的记忆 → 自动删除
  → 返回最终结果
```

与 Mem0 的关键区别：Mem0 的 search 是单路向量搜索；PowerMem 是四路混合 + 衰减加权。

#### 艾宾浩斯遗忘曲线 — 核心差异化能力

衰减公式：`retention = e^(-decay_rate × hours_elapsed)`

| 时间  | 保留率  | 说明                |
|-----|------|-------------------|
| 0分钟 | 100% | 刚存入               |
| 1小时 | 44%  |                   |
| 1天  | 33%  |                   |
| 7天  | 25%  | 未被访问的低重要性记忆接近清理阈值 |
| 31天 | 21%  | 低于 0.3 阈值 → 自动删除  |

**触发机制**：不是后台定时任务，是**懒计算（on-access）** — 每次 `search()`
时顺便计算衰减、更新访问计数、删除过期记忆。这意味着长时间未被查询的记忆不会被主动清理，只在下次被搜到时才会被判定过期。

**强化机制**：被反复检索到的记忆衰减会变慢，高频访问的记忆自动沉淀为长期记忆。

#### 内置用户画像系统（UserMemory）

独立于普通记忆的画像模块，可从对话中自动提取结构化用户画像（职业背景、个人偏好、行为模式等），并基于画像自动增强搜索查询。例如搜索"
项目分析"时，如果画像记录了"用户关注成本控制"，会自动增强为"项目利润和成本分析"。

**注意**：完整的 UserMemory 和图存储功能仅 OceanBase 后端支持，PG/SQLite 功能受限。

#### 粒度控制

与 Mem0 类似：自定义提取 prompt + metadata 标签 + 过滤器。额外支持高级嵌套逻辑过滤（AND/OR 组合）和 Sub Stores 数据分区（仅
OceanBase）。原生提供 AsyncMemory 异步接口。

#### 优势与局限

**优势**：记忆自动衰减不会无限膨胀、四路混合检索召回率高、jieba 分词中文友好、内置用户画像、LOCOMO 准确率最高。

**局限**：**项目太年轻**（v0.5，427 stars），生产可靠性未验证；完整功能**绑定 OceanBase**；社区规模小，生产案例不明确。

#### 代码示例

```python
from powermem import AsyncMemory

# 初始化（异步接口，原生支持 async/await）
memory = AsyncMemory(config={
    "llm": {"provider": "openai", "config": {"model": "gpt-4.1-nano"}},
    "db": {"provider": "postgresql", "config": {"connection_string": "postgresql://..."}},
})

# 存记忆 — 自动提取 + 重要性评分 + 三级分层
await memory.add(
    messages=[{"role": "user", "content": "帮我持续关注项目A的利润率，低于5%要告诉我"}],
    user_id="boss_001",
    infer=True,  # 开启 LLM 推理提取
)

# 取记忆 — 四路混合检索 + 艾宾浩斯衰减加权
results = await memory.search(
    query="项目A利润",
    user_id="boss_001",
    limit=5,
    filters={"AND": [{"category": {"eq": "project_focus"}}]},  # 嵌套逻辑过滤
)
for r in results["results"]:
    print(r["memory"], r["score"], r["retention_score"])

# 用户画像 — 自动从对话中提取结构化画像（仅 OceanBase 完整支持）
from powermem import UserMemory

user_mem = UserMemory(config=config)
await user_mem.add(messages=messages, user_id="boss_001")
profile = await user_mem.get_profile(user_id="boss_001")
# {"role": "集团高管", "focus": ["利润率", "成本控制"], "style": "偏好简洁汇报"}
```

### 4.3 LangMem — LangGraph 原生方案（停止更新-最新更新在4个月前，且与 1.x langchain存在兼容性问题）

> 1.3k Stars | LangChain 官方出品 | 与 LangGraph Store 深度绑定

#### 工作原理

LangMem 采用两层架构：

- **核心 API 层（无状态）**：纯函数，转换记忆状态，不产生副作用
- **有状态集成层**：依赖 LangGraph 的 `BaseStore` 接口自动持久化

记忆提取通过 trustcall 库实现**并行工具调用** — LLM 可以在一次调用中同时执行多个 insert/update/delete
操作。提取前会先用向量搜索已有记忆进行比对，LLM 在上下文中看到已有记忆后决定是创建新的、更新已有的、还是删除过时的。

#### 核心差异 — 三种记忆类型

LangMem 是三个框架中**唯一区分记忆类型**的（Mem0/PowerMem 只有第一种）：

**语义记忆（Semantic）** — "知道什么"：

存储事实、知识、用户偏好。有两种子模式：

- **集合模式**：无限量知识存储，适合持续积累的用户偏好
- **档案模式**：单文档严格 Schema，适合追踪最新用户画像状态

**情景记忆（Episodic）** — "经历过什么"：

捕获完整的交互经验和推理链，而非仅存事实。每条情景记忆包含：观察（用户问了什么）→ 思考（AI 的推理过程）→ 行动（采取了什么方法）→
结果（效果如何）。

实际价值：当老板再次问类似问题时，Agent 可以参考过去成功的分析经验，而不是每次从零推理。例如记住"
上次分析项目A时用环比对比效果好，用同比对比老板不满意"。

**程序性记忆（Procedural）** — "怎么做"：

LangMem 独有的能力。通过 `create_prompt_optimizer` 实现 — 输入对话轨迹 + 效果评分，自动优化 system prompt，让 Agent
从反馈中内化行为规则。三种优化策略：

| 策略              | LLM调用 | 深度 | 适用场景 |
|-----------------|-------|----|------|
| `prompt_memory` | 1次    | 基础 | 快速迭代 |
| `metaprompt`    | 2-5次  | 良好 | 均衡选择 |
| `gradient`      | 4-10次 | 最佳 | 离线优化 |

这三种记忆类型恰好对应我们的需求：

- 语义记忆 → Layer 2 用户偏好
- 情景记忆 → Layer 4 交互历史（比 Mem0 更结构化）
- 程序性记忆 → Agent 的经验积累（Mem0/PowerMem 完全不具备）

#### 记忆形成的两条路径

**热路径（对话中）**：Agent 在推理过程中主动调用 `manage_memory` 工具存取记忆，记忆写入 Store 后继续响应。

**后台路径（对话后）**：`ReflectionExecutor` 在对话结束后异步提取记忆，不阻塞用户响应。内置**防抖机制** —
如果在延迟期间收到新消息，旧任务被取消，用完整对话上下文重新调度，避免连续消息重复处理。

#### 记忆生命周期管理

| 能力        | 机制                                            | 是否自动 |
|-----------|-----------------------------------------------|------|
| **创建**    | LLM 通过 trustcall 并行工具调用创建                     | ✅ 自动 |
| **去重**    | 提取前搜索相似记忆，LLM 决定创建新的还是更新已有                    | ✅ 自动 |
| **更新**    | trustcall 使用 JSON Patch **增量更新**（非全量重写）       | ✅ 自动 |
| **矛盾解决**  | LLM 判断后 update 覆盖或 delete（需开启 enable_deletes） | ✅ 自动 |
| **衰减/过期** | ❌ 无内置衰减，与 Mem0 一样                             | 无    |
| **版本追踪**  | 无内置审计                                         | 无    |

#### 粒度控制

通过 **Pydantic BaseModel** 精确定义记忆结构 — 比 Mem0/PowerMem 的 metadata 标签更强，可以定义字段类型、描述、验证规则。LLM
提取时必须按 Schema 结构化输出。

命名空间为 tuple 结构，支持模板变量和多层隔离（如 `("memories", "{topgroup_id}", "{user_id}")`），可按记忆类型分命名空间存取。

#### 存储后端

直接使用 LangGraph 的 `BaseStore` 接口：

- `InMemoryStore` — 开发/测试
- `AsyncPostgresStore` — **生产推荐**
- LangGraph Platform 内置 Store — 平台部署
- 可自定义适配器对接其他存储

**Checkpointer vs Store**：Checkpointer 保存单个会话的对话状态（短期记忆），Store 保存跨会话的长期记忆。LangMem 使用 Store。

#### 优势与局限

**优势**：LangGraph 原生零额外存储依赖、三种记忆类型（情景记忆和程序性记忆是独有能力）、Pydantic Schema 类型安全、内置后台异步处理+防抖、JSON
Patch 增量更新。

**局限**：**无记忆衰减**（与 Mem0 一样）；**仅单路向量搜索**（无全文/混合检索）；紧耦合 LangGraph；无用户画像系统；无审计日志；LOCOMO
基准延迟较高（p95: 59.82s，但此基准由 Mem0 团队发布，可能有偏）。

#### 代码示例

```python
from pydantic import BaseModel, Field
from langmem import create_memory_manager, create_manage_memory_tool, create_search_memory_tool
from langgraph.store.memory import InMemoryStore


# 1. 用 Pydantic 定义记忆结构（类型安全，LLM 必须按 Schema 输出）
class UserPreference(BaseModel):
    """用户偏好记忆"""
    topic: str = Field(description="关注的主题，如项目名称或指标类型")
    detail: str = Field(description="具体偏好描述")
    importance: str = Field(description="重要程度: high/medium/low")


# 2. 创建记忆管理器（语义记忆 - 集合模式）
store = InMemoryStore(index={"dims": 1536, "embed": embeddings})
manager = create_memory_manager(
    "openai:gpt-4.1",
    schemas=[UserPreference],
    instructions="提取用户关注的项目、指标偏好和分析风格",
    enable_deletes=True,
)

# 3. 作为 Tool 暴露给 Agent（主动调用模式）
memory_tools = [
    create_manage_memory_tool(namespace=("memories", "{user_id}")),
    create_search_memory_tool(namespace=("memories", "{user_id}")),
]
# 将 memory_tools 加入 Agent 的工具列表即可

# 4. 后台异步提取（对话后路径，不阻塞用户响应）
from langmem import ReflectionExecutor

executor = ReflectionExecutor(manager, store=store, delay=5.0)  # 5秒防抖
executor.submit(namespace=("memories", "boss_001"), messages=conversation_messages)

# 5. 程序性记忆 — Prompt 自优化（LangMem 独有）
from langmem import create_prompt_optimizer

optimizer = create_prompt_optimizer(
    "openai:gpt-4.1",
    kind="metaprompt",  # 均衡策略，2-5次 LLM 调用
)
updated_prompt = optimizer.invoke(
    current_prompt=system_prompt,
    trajectories=[(conversation, score)],  # 对话轨迹 + 效果评分
)
```

### 4.4 三框架综合对比

## 五、记忆系统设计

参考《从零构建一个超越ChatGPT和Claude客户端的长期记忆能力系统》一文：

- **ChatGPT 三层记忆**：User Memory（结构化特征）+ Recent Conversations（会话摘要 RAG）+ User Insight（隐性画像）
- **Claude 的 Agentic RAG**：将记忆搜索作为 Tool 暴露给模型，由模型自主决定何时检索 — 与我们 Agent 的工具化架构天然契合
- **双通道检索**：被动 RAG（自动注入）+ 主动 RAG（工具调用）
- **异步存储**：不阻塞用户体验


### 对于ChatGPT
对话时加载的上下文的结构大致如下：

系统定义层

[0] System Instructions

[1] Developer Instructions

[2] Session Metadata (ephemeral)

长期记忆层

[3] User Memory (long-term facts)

[4] Recent Conversations Summary (past chats, titles + snippets)

当前会话层

[5] Current Session Messages (this session)

[6] Your latest message

- 层级0和1包含了系统级别的配置，如系统参数级别的描述、防injection指令、工具定义等
- 层级2则包含临时的会话元信息，如用户当前时间、IP所在地区等
- user memory（层级3）和recent conversations summary（层级4）构成了所谓的长期记忆

ChatGPT提供了一个名为bio的工具。当模型检测到当前用户信息中包含与用户档案相关的信息时，它会主动调用这个工具来存储记忆（不好控制，社区提出走一个单独的异步任务，每一条消息作为单独任务判断是否存入记忆或者提取核心摘要）。这些存储的记忆条目可以在设置界面中查看和管理，形成了一个用户可见、可编辑的记忆列表。

实际的实现比描述的更加复杂。ChatGPT并不是每次都将所有的user memory全部注入上下文，而是使用RAG进行检索匹配。也就是说，系统会根据当前的用户问题/信息，从记忆库中检索出最相关的若干条memory记录（相关性，时间衰减等多种因素），然后注入到上下文中。

除此之外，还发现有一些两阶段的进阶RAG策略：
1. 首先通过RAG匹配相关的bio记录，这些bio带有时间区间标记
2. 然后在这些时间区间内，再次使用RAG检索top-k个相关的conversation摘要（rerank？）
3. 同时，无论是否匹配到相关bio，都会加入固定数量的最近session摘要

**User Insight隐性系统**

类似于用户画像，更高维度的定期刷新，记载用户的性格，偏好，语言风格，接受的回复风格，关注特点等等


### 对于claudecode
相比ChatGPT，Claude的记忆系统采用了截然不同的设计哲学。

Claude的最大特点是完全放弃了传统的RAG自动注入方式，转而采用agentic RAG——也就是将会话搜索和记忆搜索作为工具直接暴露给模型，由模型自主决定何时、如何检索记忆。

它高度依赖模型的能力——模型需要准确判断何时需要检索、使用什么关键词检索、如何整合检索结果。如果模型能力不足，或者在本地部署时使用较弱的模型，效果可能会大打折扣。

其次，由于没有自动注入相关的会话摘要，完全依赖模型主动检索，可能会存在信息遗漏的风险。毕竟RAG的语义搜索能力是经过验证的，而模型的主动检索更多依赖prompt理解和关键词提取。
