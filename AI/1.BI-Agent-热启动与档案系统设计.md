# BI-Agent 热启动与档案系统设计

---

## 一、场景与边界

### 1.1 当前面临的问题

1. **Token 爆炸**：每次对话都实时调用多个接口聚合大量数据，Token 大量消耗在"搬运数据"上，尤其是涉及多项目分析时
2. **热启动缺失**：AI 对业务目标零感知，每次对话从零开始，任何具体项目问题都要先经过一大圈检索才能回复
3. **数据维度结构性缺失**：没有项目级别的各维度汇总数据供 AI 做"一眼粗了解"，很多指标藏在 Tool 里，Master 感知不到，只能按通用常识拆分任务

### 1.2 两条路径的边界

本设计覆盖的是**档案系统 + 渐进式信息披露**这条路径，适用于：

| 路径 | 适用场景 | 数据来源 |
|------|----------|----------|
| **档案披露路径（本设计）** | 概览性问题、项目情况了解、少量项目对比、异常识别、集团总览 | 预计算的项目档案 + 集团报告 |
| **宽表 + SubAgent 路径** | 深度分析、详细指标分析、所有实时数据查询、大范围项目/车场的情况调查、横向对比、情况分析、原因归因 | 宽表实时查询 + SubAgent 工具 |

两条路径协同工作：Master 基于档案快速回答概览性问题，当需要深入分析、实时数据、或大范围调查时，派发 SubAgent 走宽表路径。

## 二、热启动机制

### 2.1 核心思路

对话开始时自动注入关键信息，让 AI 具备基本的业务认知：

1. **组织结构树**（带关键标签）— AI 知道集团有哪些组织、哪些项目
2. **当前时间** — 用于相对时间计算

不初始注入完整项目档案，理由：
- 50 个项目 × 200 字 = 1 万字初始 context，大部分浪费
- 组织树上的标签已覆盖"最基本了解"的需求
- 需要更多细节时再按需加载，节省初始 Token

### 2.2 按需加载策略

AI 自主决定加载哪些项目档案：

```
用户提问 → AI 看到问题 + 组织树 → 判断涉及哪些项目 → 调用工具加载档案 → 回答
```

为什么不用代码预匹配项目名：
- "最近有什么异常项目" — 没有项目名，但 AI 看组织树上的标记就知道该加载谁
- "华东区情况" — AI 看树结构直接知道华东区有哪些项目
- AI 理解用户意图，能做更准确的判断

## 三、组织结构树设计

### 3.1 结构树内容

```
集团：XX停车管理集团（ID: G001）

华东区
├── 宝龙广场项目 (P1001) [承包/非金融] [运营中] 💰+12万
├── 万达停车场 (P1002) [分成] [运营中] ⚠️-3万
└── 银泰中心 (P1003) [承包/金融] [运营中断]

华南区  
├── 深圳湾项目 (P2001) [管理] [运营中] 💰+8万
└── 前海车场 (P2002) [分成] [运营中] 💰+5万
```

### 3.2 节点信息

每个项目节点包含：
- 项目名称 + 项目 ID
- 项目性质（承包/分成/管理，是否金融）
- 运营状态
- 上月盈亏标记（💰 盈利 / ⚠️ 亏损）

### 3.3 设计考量

- 50 个项目约 1500-2000 字符，可接受的初始 context
- 项目 ID 必须带上，后续加载档案和调用查询都需要
- 盈亏标记让 AI 一眼识别异常项目，不用先查询
- 项目性质影响业务规则判断（如亏损判断逻辑因项目类型而异）

## 四、档案库设计

### 4.1 整体结构

```
档案库/
├── 集团报告          ← 集团级汇总，对齐集团看板维度
└── 项目档案/
    └── 每个项目/
        ├── 基础信息    ← 几乎不变
        ├── 静态档案    ← 事件驱动更新
        └── 动态档案    ← 每日更新
```

### 4.2 集团报告

每日预计算的集团级汇总报告，数据维度对齐集团总览看板：

| 维度 | 内容 |
|------|------|
| 经营总览 | 集团整体归属收入、成本、利润（本月/上月/同比/环比） |
| 预算情况 | 集团预算达成率、偏差率 |
| 项目分布 | 盈利/亏损项目数量分布、运营状态分布 |
| 关键排行 | 利润 TOP5 / BOTTOM5、收入增长 TOP5 / 下滑 TOP5 |
| 异常信号 | 本月新增亏损项目、状态变更为异常的项目、预算达成率骤降项目 |
| 重点指标 | 集团整体车位利用率、异常放行损失总额、未关闭告警总数等我们认为重要的集团级指标 |
| AI 摘要 | 集团整体经营状况的简短总结 |

集团报告的定位：当用户问"集团整体情况"时，Master 加载这一份报告就能回答，不需要加载所有项目档案。

### 4.3 项目档案分层

| 层级 | 内容 | 更新频率 |
|------|------|----------|
| 基础信息 | 项目 ID、名称、地区、性质、绑定车场 | 几乎不变 |
| 静态档案 | 合同情况、开业时间、重大事件、历史转折点 | 事件驱动 |
| 动态档案 | 财务/运营/管理各维度的数据汇总 + AI 摘要 | 每日更新 |

### 4.4 动态档案的数据维度

| 域 | 指标 |
|---|------|
| 财务 | 归属收入、成本、利润、预算达成率、回款 |
| 运营 | 车位利用率、异常放行损失、月租占比、停车时长分布 |
| 管理 | 告警情况、工单情况、近期项目事件 |
| 摘要 | AI 生成的简短总结（~100字）、异常信号标记 |

### 4.5 预计算策略

**代码聚合为主，LLM 总结为辅**：

- 每日定时任务，代码层调接口/宽表聚合数据（纯计算，无 LLM）
- 输出结构化的指标数据
- LLM 只做最后一步：生成简短的 AI 摘要

## 五、存储设计

### 5.1 为什么选择数据库

| 方案 | 问题 |
|------|------|
| 文件系统 | Docker 部署有状态问题、多实例不共享 |
| OSS | 列目录/树结构查询不方便 |
| **数据库** | 天然支持碎片化查询、多集团隔离 |

### 5.2 核心设计：按指标域拆分存储

不是"每个项目一个文档"，而是"每个项目 × 每个指标域 × 每个指标"一条记录。

好处：
- 查"所有项目的利润"只需要筛选 metric_key = profit，不用加载完整档案
- 跨项目对比高效，一条查询拿到 50 个项目的同一指标
- 灵活扩展，新增指标只需要插入新记录

### 5.3 数据结构

核心字段：
- topgroup_id — 集团隔离
- project_id — 项目 ID（集团报告用特殊标识如 "GROUP"）
- metric_domain — 指标域（financial / operation / management / summary）
- metric_key — 指标键（income / profit / utilization / alerts / ai_digest ...）
- metric_value — JSON 格式的指标值
- report_date — 报告日期

### 5.4 metric_value 的 JSON 结构

根据指标类型，JSON 结构不同：

- 单值型：`{value: 450000, unit: "元"}`
- 带对比：`{current: 450000, last: 520000, change: -0.13}`
- 趋势型：`{values: [42, 45, 48], labels: ["1月", "2月", "3月"]}`
- 列表型：`{items: [{name: "设备维修", status: "逾期", days: 3}]}`
- 摘要型：`{text: "上月利润12万...", signals: ["utilization_low"]}`

## 六、Tool 查询设计

### 6.1 核心查询工具

一个万能查询工具，支持不同粒度：

**参数**：project_ids（项目 ID 列表）、domain（指标域，可选）、metrics（具体指标列表，可选）

**查询行为矩阵**：

|  | 单项目 | 多项目 |
|--|--------|--------|
| 不传域+不传键 | 完整档案 | 只返回 AI 摘要 |
| 传域+不传键 | 该域全部指标 | 跨项目域对比 |
| 传域+传键 | 单指标详情 | 跨项目指标对比 |

多项目 + 不传域不传键 强制降级到只返回摘要，避免 Token 爆炸。

### 6.2 快捷工具

| 工具 | 用途 |
|------|------|
| get_group_report | 获取集团报告（整体经营概览） |
| compare_projects_metric | 跨项目单指标对比 |
| get_group_anomalies | 获取集团异常项目列表 |
| get_group_ranking | 获取集团指标排行 |

### 6.3 Master 工具集总览

| 工具 | 说明 |
|------|------|
| query_project_metrics | 万能查询（单项目/多项目、全量/按域/按指标） |
| get_group_report | 集团报告 |
| compare_projects_metric | 快捷对比 |
| get_group_anomalies | 快捷异常 |
| get_group_ranking | 快捷排行 |
| calculator | 数学计算 |

6 个工具覆盖绝大部分场景。

## 七、输出格式设计

### 7.1 设计原则

对 AI 来说，交互形式是"浏览项目档案库"，输出格式要强化这种"档案文档"的感觉：

- 层级感 — 用标题层级体现档案结构
- 图标辅助 — 📁 档案、📊 数据、💡 摘要、⚠️ 警告
- 表格优先 — 多指标用表格，比列表更紧凑
- ID 始终可见 — 每次提到项目都带 ID，方便后续查询

### 7.2 集团报告格式

```
📊 集团经营报告（2026-02-06）

整体经营
| 指标 | 本月累计 | 上月 | 环比 | 预算达成 |
| 归属收入 | 1200万 | 1350万 | -11% | 85% |
| 利润 | 180万 | 220万 | -18% | 78% |

项目分布：运营中 42个 / 运营中断 3个 / 解约中 1个
盈利 38个 / 亏损 4个

⚠️ 异常信号
- 万达停车场 (P1002) 本月转亏
- 银泰中心 (P1003) 运营中断

💡 集团本月收入环比下降11%，主要受华东区3个项目拖累...
```

### 7.3 单项目完整档案格式

```
📁 宝龙广场项目档案

基础信息
- 项目ID: P1001 | 华东区 | 承包（非金融）| 运营中

📊 财务数据（截至 2026-02-05）
| 指标 | 本月累计 | 上月 | 环比 | 预算达成 |
| 归属收入 | 45万 | 52万 | -13% | 87% |
| 利润 | 7万 | 12万 | -42% | 70% |

📊 运营数据
| 指标 | 当前值 | 上月均值 | 变化 |
| 车位利用率 | 45% | 48% | ↓3pp |
| 异常放行损失 | 2.1万 | 1.6万 | ↑31% |

📊 管理数据
- 未关闭告警: 0 | 逾期工单: 1（设备维修）

💡 上月利润12万，环比+8%。车位利用率偏低，异常放行损失增加31%值得关注。
```

### 7.4 跨项目对比格式

```
📊 利润对比（3个项目）

| 项目 | 区域 | 性质 | 本月利润 | 上月利润 | 环比 | 状态 |
| 宝龙广场 (P1001) | 华东 | 承包 | 7万 | 12万 | -42% | 盈利 |
| 万达停车场 (P1002) | 华东 | 分成 | -3万 | 2万 | - | ⚠️亏损 |
| 深圳湾项目 (P2001) | 华南 | 管理 | 8万 | 7万 | +14% | 盈利 |
```

## 八、交互流程总结

### 8.1 对话启动

1. 用户发起对话
2. 自动加载组织结构树（带标签）注入 prompt
3. Master 已"认识"所有项目，准备就绪

### 8.2 问题处理

```
用户提问
    ↓
Master 分析问题 + 组织树
    ↓
    ├→ 简单问题（从树标签可答）→ 直接回答
    ├→ 集团概览 → 加载集团报告 → 回答
    ├→ 项目相关 → 加载项目档案 → 基于档案回答
    └→ 深度分析/实时数据/大范围调查/详细指标 → 派发 SubAgent 走宽表路径
```

### 8.3 典型场景

| 场景 | 处理路径 |
|------|----------|
| 哪些项目亏损？ | 看组织树 ⚠️ 标记，直接回答 |
| 集团整体经营情况？ | 加载集团报告，回答 |
| 项目A 情况怎么样？ | 加载项目A完整档案，回答 |
| 华东区项目利润对比（少量） | 识别华东区项目，加载利润指标对比，回答 |
| 项目A 为什么亏损？ | 加载档案看到亏损事实，派发 SubAgent 深度分析归因 |
| 所有项目的车位利用率排行 | 大范围调查，派发 SubAgent 走宽表查询 |
| 集团所有车场异常放行情况分析 | 大范围情况分析，派发 SubAgent 走宽表查询 |
| 项目A 上月各科目收入明细 | 详细指标分析，派发 SubAgent 走宽表查询 |
| 今天的实时收入数据 | 实时数据查询，派发 SubAgent 走宽表查询 |

## 九、关键设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 初始注入内容 | 只注入组织树（带标签） | 平衡认知广度和 Token 消耗 |
| 档案加载方式 | AI 自主决定 | 比代码预匹配更准确 |
| 存储方案 | 数据库，按指标域拆分 | 支持碎片化查询 |
| 预计算方式 | 代码聚合 + LLM 简短总结 | 成本低、结果稳定 |
| 集团报告 | 独立预计算，对齐看板维度 | 集团概览不需要加载所有项目 |
| 输出格式 | 档案文档风格 | 强化渐进式披露的交互感 |
| 路径分工 | 档案路径 vs 宽表+SubAgent 路径 | 概览走档案，深入走宽表 |
