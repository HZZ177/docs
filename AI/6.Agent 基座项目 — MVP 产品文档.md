# Agent 基座项目 — MVP 产品文档

## 项目背景

本项目正在经历从**初期探索**到**企业级 Agent 基座**的关键转型。

**一期（现状）** 是一个面向单业务线（永策Pro BI）的硬编码多 Agent 系统。Agent 定义、Prompt、工具绑定、SubAgent 编排全部写死在 Python 代码中（装饰器注册 + 类属性配置），验证了 LangChain + LangGraph + DeepAgents 技术栈的可行性，跑通了 MainAgent 协调 → SubAgent 执行 → 工具调用 → 流式响应的完整链路。但硬编码架构意味着：每新增一个 Agent 或调整一条 Prompt 都需要改代码、重新部署，无法支撑多业务线复用。

**MVP（目标）** 将一期的硬编码架构重构为**数据库配置驱动 + 管理页面**的通用基座平台。核心变更：

| 维度 | 一期（硬编码） | MVP（配置化） |
|------|------------|------------|
| Agent 定义 | Python 类 + 装饰器（`agent.py`） | DB 记录，页面创建编辑 |
| Prompt 管理 | Python 字符串常量（`prompt.py`） | DB 字段，页面实时编辑，热更新 |
| Agent 编排 | `get_subagents()` 硬编码返回 | Scene 页面可视化编排主从关系 |
| 工具绑定 | `get_tools()` 硬编码返回 | Scene 页面勾选内置工具 + MCP 远程工具 |
| 业务工具 | `@tool` 装饰器进程内定义 | 内置工具库 + MCP Server 远程接入 |
| 记忆能力 | 无 | 五层 Prompt 记忆架构（PowerMem + Checkpointer） |
| 配置生效 | 改代码 → 重新部署 | 页面修改 → 下次请求自动生效 |
| 多业务线 | 仅支持单业务线 | 各业务线独立部署同一份基座代码，库级隔离 |

转型后，各业务线拿到同一份基座代码，通过管理页面完成 Agent 定义、编排、工具接入和记忆配置，**零代码**即可搭建自己的多 Agent 系统。

**技术栈：**

| 层级       | 技术选型                                  |
|----------|---------------------------------------|
| 前端       | Vue                                   |
| 后端 Web   | FastAPI                               |
| Agent 框架 | LangChain + LangGraph + DeepAgents    |
| 记忆模块     | PowerMem                              |

| 数据库      | SeekDB（OceanBase AI 原生数据库，MySQL 协议兼容） |

**数据库部署模式：**

- **云端统一部署**：公司级别部署一套 SeekDB 实例
- **业务线隔离**：各业务线单独部署基座服务，通过连接地址接入云端数据库，**一个业务线对应一个独立库**
- **库内数据分表**：
    - 后台配置（agents / skills / scenes / mcp_servers / memory_configs 等）
    - 会话映射（sessions —— user_id ↔ session_id 对应关系）
    - 对话历史（checkpoints / checkpoint_writes —— LangGraph Checkpointer 自动管理）
    - 记忆系统（PowerMem 自动创建的表 —— 长期记忆 / 用户画像）
    - 定时任务（scheduled_tasks / task_execution_logs）
- **自研适配**：LangGraph Checkpointer 需自行实现 `SeekDBCheckpointSaver`（官方无 SeekDB 适配）

## 一、产品概述

### 1.1 产品定位

Agent 基座是一个**各业务线单独部署**的 AI Agent 开发与管理平台。每个业务线拿到同一份基座代码，通过管理页面完成业务配置，独立部署、独立运行。

### 1.2 核心能力

| 能力            | 说明                                       |
|---------------|------------------------------------------|
| Agent 全生命周期管理 | 页面创建、编辑、停用 Agent，配置模型与 Prompt            |
| Skill 管理      | 在线上传 skill.md 自动解析，或页面直接创建编辑             |
| Scene 可视化编排   | 在页面上配置 MainAgent + SubAgent 主从关系，绑定工具和技能 |
| MCP 工具接入      | 页面配置 MCP Server，刷新获取工具列表，按 Server 分组导入   |
| 记忆系统          | 两层记忆架构（Layer 1 场景记忆按 scene_id+user_id 隔离 + Layer 2 用户画像跨 Scene 共享）+ 对话上下文（Checkpointer 自动管理），MVP 全部启用 |
| 调试能力          | 选择任意 Agent 或 Scene 直接对话调试，查看完整调用链路       |
| 定时任务          | Scene 中的 Agent 可创建定时任务，系统调度执行并按配置通知结果    |
| 三方接入          | WebSocket 协议统一接入（chat/cancel/session 管理），不另设 HTTP Chat API |

> **用户体系说明**：本平台不自建用户体系。所有"用户"信息来自三方系统传入的 `user_id`（必传）及 `extra_params`（业务线自定义透传参数）字段，仅用于：
> - **MCP 权限透传**：将 `user_id` + `extra_params` 透传给 MCP Server，由 MCP 层自行通过这些参数获取 token 并鉴权
> - **对话历史归属**：按三方 `user_id` 隔离会话和消息记录
> - **记忆隔离**：按三方 `user_id` 隔离长期记忆和用户画像
> - **定时任务身份**：创建定时任务时自动持久化 `user_id` + `extra_params`，任务唤起时传给 Agent 用于工具调用
>
> **参数透传设计**：三方系统需传入 `user_id` + `extra_params`（如集团 ID、车场 ID 等），这些参数的组合必须能唯一定位业务场景下的确定用户。框架向后透传这些参数给 MCP 层，MCP Server 自行用 `user_id` + 业务参数获取 token 完成鉴权。这样做的原因是定时任务被唤起时没有实时 chat 场景，不存在有效 token，预存的 token 也会过期，因此将 token 获取下沉到 MCP 层。
>
> **兼容场景**：如果 chat 场景下业务线直接传入了 `token`，框架也会存入上下文变量（`request_context`），工具调用时能获取到 token 就透传，方便 MCP 层直接鉴权，跳过自行获取 token 的步骤。

## 二、核心概念与术语

### 2.1 概念定义

| 概念             | 定义             | 说明                                        |
|----------------|----------------|-------------------------------------------|
| **Agent**      | 一个 AI 智能体单元    | 拥有名称、模型配置、系统提示词，本身不具有"主/从"属性              |
| **Skill**      | Agent 的结构化复合能力 | 由执行指令（Prompt）+ 依赖工具组成，粒度高于单个 Tool         |
| **Scene**      | 一组 Agent 的编排配置 | 定义 MainAgent + SubAgent 的主从关系，以及工具和技能的绑定  |
| **MCP Server** | 业务线部署的工具服务     | 遵循 MCP 协议，提供业务工具给 Agent 调用                |
| **Tool**       | 一个原子函数调用       | 分为内置工具（框架提供）和 MCP 工具（业务线提供）               |
| **Memory**     | 记忆系统           | 两层架构：Layer 1 场景记忆（PowerMem Memory，按 scene_id+user_id 隔离）、Layer 2 用户画像（PowerMem UserMemory，跨 Scene 共享）+ 对话上下文（LangGraph Checkpointer 自动管理） |
| **MainAgent**  | 协调型 Agent      | 在某个 Scene 中担任主角色，负责意图理解和任务分派              |
| **SubAgent**   | 执行型 Agent      | 在某个 Scene 中担任从角色，专注某个业务领域的任务执行            |
| **定时任务**       | Agent 创建的周期性任务 | 由 Scene 中的 MainAgent 通过工具创建，系统按 cron 调度执行 |

### 2.2 概念关系

```
Agent（能力单元，无固有主从属性）
  │
  │  通过 Scene 编排确定角色
  │
  ▼
Scene（编排配置）
  │
  ├── main_agent ──→ 某个 Agent 充当 MainAgent
  ├── sub_agents ──→ 若干 Agent 充当 SubAgent
  │
  ├── tool_bindings ──→ 为每个 Agent 绑定工具
  │   ├── 内置工具（框架提供）
  │   └── MCP 工具（从 MCP Server 导入）
  │
  └── skill_bindings ──→ 为每个 Agent 绑定技能
```

**核心原则：定义与编排分离**

- 同一个 Agent 可以在 Scene A 中作为 MainAgent，在 Scene B 中作为 SubAgent
- 工具和技能的绑定由 Scene 决定，不写死在 Agent 定义中

### 2.3 关键区分

| 对比   | Tool               | Skill                      |
|------|--------------------|----------------------------|
| 本质   | 一个函数调用             | 结构化的执行指令                   |
| 粒度   | 原子操作（查一个数据）        | 复合任务流程（诊断分析）               |
| 来源   | 内置工具库 / MCP Server | 上传 skill.md / 页面创建         |
| 执行方式 | Agent 直接调用         | 工具模式（Prompt 注入）或脚本模式（沙箱运行） |
| 触发方式 | Agent 推理时自动调用      | 手动触发 / LLM 自动触发 / 定时触发     |

**表单字段定义：**

| 字段       | 类型           | 必填 | 校验规则             | 说明                   |
|----------|--------------|----|------------------|----------------------|
| 名称       | 文本输入         | 是  | 1-50 字符，不可重复     | Agent 的显示名称          |
| 描述       | 多行文本         | 是  | 1-200 字符         | Agent 的功能描述          |
| 模型       | 下拉选择         | 是  | 从可用模型列表中选择       | 关联的 LLM 模型           |
| 温度       | 数字输入         | 否  | 0 ≤ 值 ≤ 1，步长 0.1 | LLM temperature 参数   |
| 最大 Token | 数字输入         | 否  | 512 ≤ 值 ≤ 128000 | LLM max_tokens 参数    |
| 超时       | 数字输入         | 否  | 10 ≤ 值 ≤ 600，单位秒 | LLM 调用超时时间           |
| 系统提示词    | Markdown 编辑器 | 是  | 不可为空             | 支持 `{variable}` 模板变量 |

**模板变量说明：**

系统提示词中支持使用 `{variable}` 占位符，框架在运行时自动替换：

| 变量               | 说明       | 来源       |
|------------------|----------|----------|
| `{name}`         | Agent 名称 | Agent 定义 |
| `{user_id}`      | 当前用户 ID  | 请求上下文    |
| `{current_time}` | 当前时间     | 运行时生成    |

**交互逻辑：**

1. **创建流程**：点击"创建 Agent" → 填写表单 → 点击"保存" → 校验通过 → 调用 `POST /api/agents` → 成功后跳转到列表页
2. **编辑流程**：列表页点击"编辑" → 加载已有数据填充表单 → 修改 → 点击"保存" → 调用 `PUT /api/agents/:id` → 成功后跳转到列表页

#### 3.3.2 Skill 上传解析流程

上传 skill.md 是 **辅助填写** 手段，**不会直接创建**
Skill。上传的文件经后端解析后，自动填入创建页面的表单字段，用户确认或修改后通过"保存"按钮完成创建。

```
用户点击 [上传 skill.md]
    │
    ▼
选择本地 .md 文件
    │
    ▼
前端调用 POST /api/skills/parse（multipart/form-data）
    │
    ▼
后端解析文件（仅解析，不入库）
    │
    ├── 解析 YAML frontmatter（--- 之间的头部）
    │   ├── 提取 name, description, mode, user_invocable,
    │   │   model_invocable, schedule, argument_hint, requires
    │   │
    │   └── 校验格式合法性（字段类型、枚举值范围）
    │
    ├── 解析 Markdown body（--- 之后的内容）
    │   └── 提取为 instruction 字段
    │
    ├── 解析成功 → 返回结构化 JSON（不入库）
    │   └── 前端携带解析结果跳转到 **创建页**
    │       → 所有表单字段自动预填
    │       → 用户可检查、修改
    │       → 点击"保存"才真正调用 POST /api/skills 创建
    │
    └── 解析失败 → 返回错误信息
        └── 前端弹出提示（如："frontmatter 格式错误"或"缺少必填字段 name"）
```

> **设计原则**：创建页是新建 Skill 的唯一入口。上传解析只是"文件 → 表单"的快捷通道，确保所有 Skill 都经过人工确认后才写入系统。

#### 3.3.2.1 skill.md 标准格式

skill.md 文件由两部分组成：**YAML Frontmatter**（元数据头部）和 **Markdown Body**（执行指令）。

**完整格式模板：**

```markdown
---
# ====== 基础信息（必填） ======
name: "项目健康诊断"                    # Skill 名称，1-50 字符
description: "对指定项目进行全方位健康诊断分析"  # Skill 描述，1-200 字符

# ====== 调用控制（必填 mode，其余可选） ======
mode: tool                             # 执行模式：tool（工具模式）| script（脚本模式）
user_invocable: true                   # 用户可否手动 /命令 触发，默认 true
model_invocable: true                  # LLM 可否根据上下文自动触发，默认 true
schedule: null                         # cron 定时表达式，null 表示不启用定时触发
argument_hint: "<项目ID>"               # 参数提示，告知用户需提供什么参数，可选

# ====== 依赖声明（可选） ======
requires:
  tools:                               # 该 Skill 执行时依赖的工具列表
    - get_project_profit
    - get_operation_data

# ====== 脚本模式专属（仅 mode: script 时需要） ======
# script_content 放在 frontmatter 之后的第二个 --- 分隔区域
# 详见下方"脚本模式的完整格式"
---

# 执行指令（Markdown 格式，作为 instruction 字段存储）

你正在执行【项目健康诊断】任务。

## 执行步骤

1. 调用 get_project_profit 获取项目财务数据
2. 调用 get_operation_data 获取运营数据
3. 综合分析，输出健康评分和关键发现

## 输出格式

- 健康评分（0-100）
- 关键发现（最多 5 条）
- 改进建议

## 约束

- $ARGUMENTS 为项目 ID，如果用户未提供则主动询问
- 数据精确，不估算不猜测
```

**字段说明：**

| 字段                | 位置          | 必填 | 类型          | 说明                                                           |
|-------------------|-------------|----|-------------|--------------------------------------------------------------|
| `name`            | frontmatter | 是  | string      | Skill 名称，对应表单"名称"字段                                          |
| `description`     | frontmatter | 是  | string      | Skill 描述，对应表单"描述"字段                                          |
| `mode`            | frontmatter | 是  | enum        | `tool` 或 `script`，对应表单"执行模式"                                 |
| `user_invocable`  | frontmatter | 否  | bool        | 默认 `true`，对应表单"用户手动触发"                                       |
| `model_invocable` | frontmatter | 否  | bool        | 默认 `true`，对应表单"LLM 自动触发"                                     |
| `schedule`        | frontmatter | 否  | string/null | cron 表达式，非空时启用定时触发                                           |
| `argument_hint`   | frontmatter | 否  | string      | 参数提示文本，对应表单"参数提示"                                            |
| `requires.tools`  | frontmatter | 否  | list        | 依赖工具名称列表，对应表单"依赖工具"                                          |
| `script_path`     | frontmatter | 条件 | string      | 已废弃，平台不提供文件系统。脚本内容直接在页面代码编辑器中编写，存储在 DB 的 `script_content` 字段 |
| *(body)*          | markdown    | 是  | markdown    | `---` 之后的全部内容，对应表单"执行指令"                                     |

**frontmatter → 表单字段映射关系：**

```
skill.md                         创建页表单
─────────────                    ─────────────
name                      →     名称
description                →     描述
mode                       →     执行模式
user_invocable             →     用户手动触发（开关）
model_invocable            →     LLM 自动触发（开关）
schedule (非null)           →     定时触发（开关=开）+ 定时表达式
argument_hint              →     参数提示
requires.tools             →     依赖工具（标签列表）
script_path                →     （已废弃，脚本内容在页面代码编辑器编写）
Markdown body              →     执行指令（Markdown 编辑器）
```

**解析校验规则：**

| 校验项             | 规则                       | 失败提示                                                 |
|-----------------|--------------------------|------------------------------------------------------|
| frontmatter 存在性 | 文件必须包含 `---` 包裹的 YAML 头部 | "文件缺少 YAML frontmatter 头部（--- 包裹的元数据区域）"             |
| name            | 必填，1-50 字符               | "缺少必填字段 name" 或 "name 长度须在 1-50 字符之间"                |
| description     | 必填，1-200 字符              | "缺少必填字段 description" 或 "description 长度须在 1-200 字符之间" |
| mode            | 必填，值为 `tool` 或 `script`  | "缺少必填字段 mode" 或 "mode 取值须为 tool 或 script"            |
| schedule        | 若非 null，须为合法 cron 表达式    | "schedule 不是合法的 cron 表达式"                            |
| requires.tools  | 若存在，须为字符串列表              | "requires.tools 格式错误，须为字符串列表"                        |
| Markdown body   | 不可为空                     | "文件缺少执行指令内容（Markdown body）"                          |

**连接状态检测机制：**

MCP Server 的在线/离线状态通过 MCP 协议标准的 `ping` 方法检测，**不使用后台定时轮询**：

```
检测时机：
┌───────────────────────────────────────────────────────────────┐
│                                                               │
│  1. 页面加载时                                                 │
│     用户进入 /mcp-servers 页面 → 前端并发请求所有 Server 状态    │
│     → 后端对每个 Server 发送 MCP ping → 更新状态展示            │
│                                                               │
│  2. 手动操作后                                                 │
│     - 点击"刷新工具列表"时，连接 Server 的结果隐式反映状态      │
│     - 添加新 Server 后，自动触发一次 ping 检测                  │
│     - 编辑 Server 地址后，自动触发一次 ping 检测                │
│                                                               │
│  3. 不做后台定时轮询                                            │
│     状态仅在用户打开页面或操作时刷新，避免无意义的后台开销        │
│                                                               │
└───────────────────────────────────────────────────────────────┘

检测方式（MCP 协议标准）：
┌───────────────────────────────────────────────────────────────┐
│                                                               │
│  请求:  POST {server_url}                                     │
│         Content-Type: application/json                        │
│         {"jsonrpc":"2.0","id":1,"method":"ping"}              │
│                                                               │
│  成功 → 返回 {"jsonrpc":"2.0","id":1,"result":{}}             │
│       → 状态: 在线 ✓                                          │
│                                                               │
│  失败 → 连接超时（3s）/ 连接拒绝 / 非预期响应                   │
│       → 状态: 离线 ✗                                          │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

> **注意**：运行时（Agent 执行中）调用 MCP 工具时是按需连接的，使用 MCP 的 Streamable HTTP 传输协议。页面上的"在线"状态仅表示 MCP Server 可达，不代表持有长连接。

**交互逻辑：**

1. **切换调试目标**：切换 Agent/Scene 后自动新建会话，清空对话和调试面板
2. **WebSocket 连接**：进入调试页面后建立 WebSocket 长连接，用于消息收发和调试事件推送
3. **调试事件流**：后端通过 WebSocket 推送所有调试事件（`tool_start/end`、`subagent_start/end`、`llm_request/response`、

4. **消息发送**：用户输入消息后，通过 WebSocket 发送 `{"action": "chat", "data": {...}}`，同 三方调用协议
5. **中断操作**：发送 `{"action": "cancel", "data": {"session_id": "..."}}` → Agent 停止执行

**一期的关键工作机制：**

**SubAgent 调度 —— 由 LangChain 框架自动管理，不需要自定义 orchestrator：**

```
                    一期 SubAgent 调度机制

MainAgent 代码：
    get_subagents() → [FinancialManager(), ProjectManager(), ...]
                                    ↓
AgentFactory.create_deep_agent()：
    遍历 SubAgent 列表 → 逐个 create_agent() → 包装为 CompiledSubAgent
                                    ↓
DeepAgents 框架 create_deep_agent()：
    自动将 SubAgent 列表注册为名为 task 的内置工具
                                    ↓
运行时：
    LLM 根据用户问题 + 各 SubAgent 的 description，自主决定调用哪个 task
    框架自动管理：SubAgent 执行、checkpoint_ns 上下文隔离、结果回注
                                    ↓
event_handler.py：
    通过 event_name == "task" 识别 SubAgent 事件
    通过 checkpoint_ns 区分不同 SubAgent 的事件流
```

> **要点**：MainAgent 调度 SubAgent 的本质是 LLM 调用 `task` 工具，和调用普通 Tool 没有本质区别。
> 框架负责将 SubAgent 包装为 `task` 工具、管理执行上下文。
> 不存在"主从调度器"或"挂起-恢复"逻辑 —— 这些都是框架内部实现，对我们透明。

**一期的硬编码点（即 MVP 需要配置化的部分）：**

| 硬编码位置       | 一期写法                        | MVP 目标                    |
|-------------|-----------------------------|---------------------------|
| Agent 定义    | Python 类 + 装饰器（`agent.py`）  | DB 记录，页面创建                |
| Prompt      | Python 字符串常量（`prompt.py`）   | DB 字段，页面编辑                |
| SubAgent 列表 | `get_subagents()` 硬编码返回     | Scene 配置的 `sub_agents` 字段 |
| Tool 绑定     | `get_tools()` 硬编码返回         | Scene 配置的 `tool_bindings` |
| LLM 配置      | 全局 `.env` 或类属性 `model_name` | 每个 Agent 独立配置，DB 存储       |
| 动态上下文       | Python 类 `ContextProvider`  | DB 配置 + 通用 Provider       |
| 业务工具        | `@tool` 装饰器进程内定义            | 内置工具库 + MCP Server 远程工具   |

### 4.3 各层职责

#### 4.3.1 API 层

| `assembler`        | Scene 配置 → 可执行 Agent 实例（详见 4.4）                              | 新增（替代一期 `registry.get_agent()`）    |
| `agent_factory`    | 创建 LangChain Agent / DeepAgent 实例，LLM 缓存复用                   | 一期 `agent/factory.py` 迁入           |
| `event_handler`    | 处理 `astream_events()` 事件流，识别 `task` 工具 → SubAgent 事件，转 WS 消息 | 一期 `services/event_handler.py` 迁入  |
| `context_builder`  | Prompt 模板渲染 + 记忆召回注入 + 用户画像注入 + Skill 索引注入                   | 新增（一期 `render_prompt_template` 演进） |
| `checkpoint_saver` | SeekDBCheckpointSaver：LangGraph 状态持久化到 SeekDB（自研适配）          | 新增（替代一期 `MemorySaver` 内存实现）        |
| `skill_executor`   | Skill 触发判断（手动/自动/定时）、工具模式：注入 instruction 到 Prompt；脚本模式：沙箱执行  | 新增                                 |
| `mcp_hub`          | MCP Client 连接池、Schema 缓存（从 DB 读取）、认证透传（注入 user_id + extra_params + token?） | 新增                                 |

**为什么 event_handler 从 services/ 迁到 engine/：**
`event_handler` 本质是 Agent 执行管线的一部分（处理 `astream_events()` 的输出流），而非业务服务。
它与 `assembler`、`agent_factory` 紧密耦合（需要知道 SubAgent 的 checkpoint_ns 结构），放在 engine/ 更内聚。

**engine/ 层不包含 orchestrator：**
一期验证了 SubAgent 调度完全由 DeepAgents 框架通过 `task` 工具自动管理。
LLM 根据各 SubAgent 的 `description` 自主选择调用，框架处理执行隔离和结果回注。
我们不需要、也不应该在框架之上再叠加一层自定义调度器。

#### 4.3.4 记忆模块

记忆模块采用**两层结构**，基于 PowerMem 框架封装。对话上下文由 LangGraph Checkpointer 自动管理。

**两层记忆架构：**

```
┌─ Layer 1: 场景记忆（Scene Memory）─── MVP 启用 ──────────────────────┐
│                                                                       │
│  隔离维度：scene_id + user_id                                         │
│  存储后端：PowerMem Memory（metadata filter: scene_id）               │
│  特征：每个 Scene × 每个用户独立的记忆空间                              │
│  用途：记录该用户在该场景中的交互事实、关注点、历史结论                    │
│  示例："用户在【BI分析】场景中关注华东区项目利润率"                       │
│                                                                       │
│  召回方式：                                                            │
│  ├── 语义召回：Memory.search(query, user_id, filters={scene_id})      │
│  └── 时间召回：Memory.search(user_id, filters={scene_id, 时间范围})   │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘

┌─ Layer 2: 用户画像（User Profile）─── MVP 启用 ──────────────────────┐
│                                                                       │
│  隔离维度：user_id（跨 Scene 共享）                                    │
│  存储后端：PowerMem UserMemory                                        │
│  特征：全局用户画像，所有 Scene 的交互都会更新此画像                     │
│  用途：用户偏好、沟通风格、常用指标、关注领域等持久化画像信息              │
│  示例："该用户偏好简洁回复，常关注利润指标，主要管理华东区"               │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**模块职责：**

| 模块                 | 职责                                            | PowerMem API | MVP 状态 |
|--------------------|-----------------------------------------------|-------------|--------|
| `powermem_client`  | PowerMem 实例管理：初始化 `Memory` + `UserMemory` 实例，统一配置（llm / embedder / vector_store） | — | 启用 |
| `scene_memory`     | 场景记忆：按 `scene_id + user_id` 隔离，语义召回（`Memory.search` + query + scene_id filter）、时间召回（`Memory.search` + 时间 filter）、session 结束后抽取存储（`Memory.add(infer=True, metadata={scene_id})`） | `Memory` | **MVP 启用** |
| `user_profile`     | 用户画像：会话启动时加载（`UserMemory.search`）、session 结束后异步更新（`UserMemory.add`），跨 Scene 共享 | `UserMemory` | **MVP 启用** |

> **与一期的关键差异**：
> - 一期没有记忆模块
> - MVP 采用两层结构：场景记忆（scene_id 隔离）+ 用户画像（跨 scene 共享），替代一期无记忆的状态
> - 对话历史的存储/加载由 LangGraph Checkpointer（`SeekDBCheckpointSaver`）自动管理，记忆抽取由 PowerMem `Memory.add(infer=True)` 内置完成
> - 框架只管理 `sessions` 表中 `user_id ↔ session_id ↔ scene_id` 的映射关系，用于加载用户会话列表和关联 checkpointer 的 thread_id

### 4.4 Agent 组装流程（assembler 核心逻辑）

assembler 是从一期 `AgentFactory` + `AgentRegistry` 演进而来的核心模块。
一期通过 Python 代码硬编码 Agent 定义，MVP 改为从 DB 加载 Scene 配置后动态组装。

```
输入: scene_id, user_id, session_id, extra_params, token(可选)
    │
    ▼
┌─ Step 1: 加载 Scene 配置 ─────────────────────────────────────┐
│                                                                │
│  scene       = db.query(Scene).get(scene_id)                  │
│  main_def    = db.query(Agent).get(scene.main_agent_id)       │
│  sub_defs    = db.query(Agent).filter(id in scene.sub_agents) │
│  bindings    = db.query(AgentToolBinding + AgentSkillBinding)  │
│                  .filter(scene_id)                             │
│  memory_cfg  = db.query(MemoryConfig).get(scene_id)  # 按 Scene 维度 │
│                                                                │
└────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Step 2: 创建 SubAgent 实例 ──────────────────────────────────┐
│                                                                │
│  for sub_def in sub_defs:                                     │
│                                                                │
│    # 2a. 解析工具绑定                                           │
│    tools = []                                                  │
│    tools += builtin_tools(bindings)    # 内置工具直接引用        │
│    tools += mcp_hub.get_tools(bindings) # MCP 工具通过 Hub 获取 │
│                                                                │
│    # 2b. 构建上下文，渲染 Prompt                                │
│    context = context_builder.build(sub_def, extra_params, token)  │
│    prompt = render_prompt(sub_def.system_prompt, context)      │
│                                                                │
│    # 2c. 注入 Skill 索引（tool 模式 Skill 的 name+description）│
│    skill_index = skill_executor.build_index(bindings)          │
│    prompt += skill_index                                       │
│                                                                │
│    # 2d. 创建 LangChain Agent                                 │
│    llm = agent_factory.get_or_create_llm(sub_def.model_config)│
│    sub_agent = agent_factory.create_agent(prompt, tools, llm)  │
│                                                                │
│    # 2e. 包装为 CompiledSubAgent（供 DeepAgents 框架使用）      │
│    compiled = CompiledSubAgent(                                │
│        name=sub_def.agent_type,                                │
│        description=sub_def.description,                        │
│        runnable=sub_agent                                      │
│    )                                                           │
│                                                                │
└────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Step 3: 创建 MainAgent 实例 ─────────────────────────────────┐
│                                                                │
│  # 3a. MainAgent 自己的工具 + Skill 索引                       │
│  main_tools = resolve_tools(bindings[main_def.id])             │
│  main_prompt = render_prompt(main_def.system_prompt, context)  │
│  main_prompt += skill_executor.build_index(bindings)           │
│                                                                │
│  # 3b. 调用 DeepAgents 创建，传入 SubAgent 列表                │
│  #      框架自动将 SubAgent 注册为 task 工具                    │
│  #      LLM 运行时自主决定调用哪个 SubAgent                     │
│  main_llm = agent_factory.get_or_create_llm(main_def.config)  │
│  main_agent = create_deep_agent(                               │
│      model=main_llm,                                           │
│      tools=main_tools,                                         │
│      subagents=compiled_subagents,  # ← 框架处理调度           │
│      system_prompt=main_prompt,                                │
│      checkpointer=checkpointer                                 │
│  )                                                             │
│                                                                │
└────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Step 4: 注入记忆层上下文 ────────────────────────────────────┐
│                                                                │
│  # Layer 2 用户画像（MVP 启用，跨 Scene 共享）                   │
│  profile = user_profile.load(user_id)                          │
│                                                                │
│  # Layer 1 场景记忆：语义召回（MVP 启用，按 scene_id 隔离）       │
│  recall = scene_memory.recall(                                 │
│      user_id, query, scene_id=scene_id                         │
│  )                                                             │
│                                                                │
│  # Layer 1 场景记忆：时间近邻召回（MVP 启用，按 scene_id 隔离）   │
│  recent = scene_memory.recall_recent(                          │
│      user_id, scene_id=scene_id, days=3                        │
│  )                                                             │
│                                                                │
│  # 将 L2 画像 + L1 场景记忆注入 system_prompt 尾部              │
│  prompt = context_builder.inject_memory(                       │
│      prompt, profile, recall, recent                           │
│  )                                                             │
│                                                                │
│  # L5 对话上下文：checkpointer 自动管理                         │
│  # → 传入 session_id 作为 thread_id                            │
│  # → checkpointer 自动加载历史 messages                         │
│  # → 超阈值时触发上下文压缩（LLM 摘要替换旧消息）                │
│                                                                │
└────────────────────────────────────────────────────────────────┘
    │
    ▼
输出: (main_agent 实例, session_id 作为 thread_id)  → 交给 chat_service 执行，checkpointer 自动管理对话历史
```

**与一期的对比：**

| 步骤             | 一期实现                                           | MVP 实现                                    |
|----------------|------------------------------------------------|-------------------------------------------|
| 获取 Agent 定义    | `registry.get_agent(agent_type)` → Python 类实例化 | `db.query(Agent)` → 从 DB 加载配置             |
| 获取 SubAgent 列表 | `agent_def.get_subagents()` → 硬编码返回            | `scene.sub_agents` → 从 Scene 配置读取         |
| 获取工具列表         | `agent_def.get_tools()` → 硬编码返回                | `tool_bindings` → 从 DB 绑定关系解析             |
| 获取动态上下文        | `context_provider.safe_fetch()` → Python 类     | `context_builder.build()` → 通用构建器         |
| 创建 Agent       | `factory.create_deep_agent()`                  | `agent_factory.create_deep_agent()`（逻辑相同） |
| 记忆/画像注入        | 无                                              | PowerMem（Layer 1 场景记忆 + Layer 2 用户画像）+ Checkpointer（对话上下文） |
| Skill 索引       | 无                                              | `skill_executor.build_index()` 注入         |

### 4.5 热更新机制

MVP 的核心设计原则之一是"**DB 修改后下次请求自动生效，无需重启**"。这不是通过复杂的通知机制实现的，而是源于一个简单的架构决策：
**assembler 每次请求都从 DB 读取最新配置，不做内存缓存**。

#### 4.5.1 热更新原理

```
管理员修改配置
    │
    ▼
管理页面 → API → Service → DB（写入新配置）
                                    │
                                    │  ← 此时运行中的对话不受影响（已组装的 Agent 实例不变）
                                    │
下一次用户对话请求到达 ─────────────────┘
    │
    ▼
chat_service → assembler.assemble(scene_id, ...)
    │
    ▼
assembler 从 DB 读取最新配置 ← 读到的就是修改后的数据
    │
    ▼
用新配置组装 Agent 实例 → 执行对话
```

**关键设计**：Agent 实例是**请求级别**的（每次对话请求创建新实例），不是应用级别的单例。只有 LLM 实例做了缓存（因为创建开销大且配置变化少）。

#### 4.5.2 各资源的热更新行为

| 资源类型                       | 修改方式              | 生效时机         | 实现机制                                                            |
|----------------------------|-------------------|--------------|-----------------------------------------------------------------|
| Agent 定义（Prompt、描述等）       | 管理页面编辑            | 下次对话请求       | assembler 从 DB 读取                                               |
| Agent 的 LLM 配置             | 管理页面编辑            | 下次对话请求       | agent_factory 检测 LLM 缓存 key 变化，自动创建新实例                          |
| Scene 编排（增减 SubAgent）      | 管理页面编辑            | 下次对话请求       | assembler 从 DB 读取 scenes.sub_agent_ids                          |
| 工具绑定（增减工具）                 | 管理页面编辑            | 下次对话请求       | assembler 从 DB 读取 agent_tool_bindings                           |
| Skill 内容（instruction / 脚本） | 管理页面编辑            | 下次对话请求       | assembler 重建 Skill 索引 + skill_executor 读取最新内容                   |
| MCP 工具列表                   | 管理页面点击"刷新 Schema" | 刷新完成后的下次对话请求 | mcp_service 重新调用 MCP Server `tools/list` → 更新 DB → assembler 读取 |
| 内置工具                       | 代码部署              | 重启后          | 内置工具是代码级别的，需要部署                                                 |

#### 4.5.3 MCP 工具热更新流程

MCP 工具是"远程工具"，需要额外的 Schema 同步步骤：

```
┌─ 场景 1：MCP Server 端新增/修改了工具 ──────────────────────────┐
│                                                                 │
│  1. 业务线开发者修改了 MCP Server 代码，新增了工具或调整了参数     │
│  2. MCP Server 部署上线                                         │
│  3. 管理员进入"MCP 配置"页面                                     │
│  4. 点击对应 MCP Server 的 [刷新 Schema] 按钮                   │
│  5. 后端调用 mcp_service.refresh_schema(server_id)              │
│     ├── 连接 MCP Server                                        │
│     ├── 调用 tools/list 获取最新工具列表                         │
│     ├── 对比 DB 中的 mcp_tools 记录                             │
│     ├── 新增的工具 → INSERT                                    │
│     ├── 删除的工具 → DELETE（检查无绑定引用后）                   │
│     └── 修改的工具 → UPDATE（Schema、描述等）                   │
│  6. 页面刷新后显示最新工具列表                                   │
│  7. 管理员在 Scene 编排中绑定新工具                               │
│  8. 下次对话请求 → assembler 读到新绑定 → mcp_hub 获取对应工具   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

┌─ 场景 2：管理员新增 MCP Server ─────────────────────────────────┐
│                                                                 │
│  1. 管理员在"MCP 配置"页面点击 [+ 添加]                          │
│  2. 填写 MCP Server 地址、认证方式                               │
│  3. 保存后自动触发 Schema 刷新                                   │
│  4. 刷新成功 → mcp_tools 表写入工具列表                          │
│  5. 管理员在 Scene 编排中将新工具绑定到 Agent                     │
│  6. 下次对话请求自动生效                                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 4.5.4 Skill 热更新流程

Skill 的 instruction 和 script_content 都存储在 DB 中，热更新天然支持：

```
┌─ 工具模式 Skill 热更新 ──────────────────────────────────────────┐
│                                                                   │
│  管理员在页面修改 Skill 的执行指令（instruction 字段）              │
│      ↓                                                            │
│  API → skill_service.update() → UPDATE skills SET instruction=... │
│      ↓                                                            │
│  下次对话请求：                                                    │
│  assembler → skill_executor.build_index(bindings)                 │
│      ↓                                                            │
│  从 DB 读取最新 instruction → 注入 Agent Prompt                   │
│                                                                   │
│  效果：Agent 立即获得更新后的 Skill 执行指令                       │
│                                                                   │
└───────────────────────────────────────────────────────────────────┘

┌─ 脚本模式 Skill 热更新 ──────────────────────────────────────────┐
│                                                                   │
│  管理员在页面代码编辑器修改脚本代码（script_content 字段）          │
│      ↓                                                            │
│  API → skill_service.update() → UPDATE skills SET script_content= │
│      ↓                                                            │
│  下次触发该 Skill 时：                                             │
│  execute_script 工具 → 从 DB 读取最新 script_content → 沙箱执行   │
│                                                                   │
│  效果：脚本代码立即更新，无需部署                                  │
│                                                                   │
└───────────────────────────────────────────────────────────────────┘
```

### 4.6 内置工具设计

内置工具是框架自带的工具，不依赖外部 MCP Server，直接在进程内执行。
与一期的业务工具不同，内置工具面向**框架能力**而非特定业务。

> **设计原则**：平台不提供文件系统。所有"文档"（Skill 指令、脚本代码）都存储在 DB 中。
> 因此需要提供内置工具让 Agent 能"查看 Skill 内容"和"执行脚本" —— 类似 Claude Code 查看 SKILL.md 和执行脚本的能力。

#### 4.6.1 工具清单

| 工具名                     | 说明               | 参数                                        | 返回值                    |
|-------------------------|------------------|-------------------------------------------|------------------------|
| `time_provider`         | 获取当前时间           | `format`（可选，时间格式）                         | 格式化的当前时间字符串            |
| `get_skill_content`     | 查询 Skill 的完整内容   | `skill_name`（Skill 名称）                    | Skill 的 instruction 全文 |
| `execute_script`        | 执行 Skill 的脚本代码   | `skill_name`（Skill 名称）, `arguments`（执行参数） | 脚本执行结果                 |
| `create_scheduled_task` | 创建当前 Scene 的定时任务 | `name`, `schedule`（cron）, `task_prompt`   | 创建结果（任务 ID）            |

> **注意**：`get_skill_content` 和 `execute_script` 仅在 Agent 绑定了 Skill 时才有意义，assembler 根据 skill_bindings 是否为空决定是否注入这两个工具。`create_scheduled_task` 仅在 Scene 开启了定时任务（`schedule_enabled=true`）时注入。`time_provider` 始终可用。

#### 4.6.2 time_provider — 获取当前时间

**定位**：为 Agent 提供准确的当前时间信息，避免 LLM 使用训练截止日期的错误时间。

**工具定义：**

```python
@tool
def time_provider(format: str = "%Y-%m-%d %H:%M:%S") -> str:
    """获取当前系统时间。

    Args:
        format: 时间格式（可选，默认 %Y-%m-%d %H:%M:%S）
                常用格式：
                - %Y-%m-%d          → 2025-01-15
                - %Y-%m-%d %H:%M:%S → 2025-01-15 10:30:00
                - %Y年%m月%d日       → 2025年01月15日
    Returns:
        格式化的当前时间字符串
    """
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| format | string | 否 | Python strftime 格式字符串，默认 `%Y-%m-%d %H:%M:%S` |

| 返回值 | 说明 |
|--------|------|
| string | 按指定格式返回的当前时间字符串 |

#### 4.6.3 get_skill_content — 查看 Skill 内容

**定位**：对标 Claude Code "查看 SKILL.md 获取完整执行指令"的能力。

工具模式的 Skill 在 assembler 阶段只将 `name + description` 注入 Prompt 作为索引（避免所有 Skill 的完整指令撑爆上下文）。当
LLM 决定执行某个 Skill 时，通过此工具获取完整的 `instruction` 内容。

```
Skill 触发流程（工具模式）：

1. assembler 将 Skill 索引注入 Prompt：
   "可用技能: [项目健康诊断] 对指定项目进行全方位健康诊断分析"

2. LLM 判断需要执行该 Skill → 调用 get_skill_content("项目健康诊断")

3. 工具实现：
   skill = db.query(Skill).filter(name="项目健康诊断").first()
   return skill.instruction  # 完整的 Markdown 执行指令

4. LLM 获得完整指令后，按指令步骤执行（调用依赖工具、生成输出等）
```

**工具定义：**

```python
@tool
async def get_skill_content(skill_name: str) -> str:
    """获取指定 Skill 的完整执行指令。
    当你需要执行某个技能时，先调用此工具获取该技能的详细执行步骤。

    Args:
        skill_name: 技能名称（从可用技能列表中获取）
    Returns:
        该技能的完整执行指令（Markdown 格式）
    """
    skill = await db.query(Skill).filter(name=skill_name).first()
    if not skill:
        return f"技能 '{skill_name}' 不存在"
    return skill.instruction
```

#### 4.6.4 execute_script — 执行 Skill 脚本

**定位**：脚本代码存储在 DB 的 `script_content` 字段中（仅支持 Python），由框架通过 `exec()` 在受控命名空间中执行，不依赖文件系统。

```
Skill 触发流程（脚本模式）：

1. assembler 将 Skill 索引注入 Prompt：
   "可用技能: [数据导出] 将查询结果导出为 CSV/Excel"

2. LLM 判断需要执行该 Skill → 调用 execute_script("数据导出", arguments={"project_id": "P1001"})

3. 工具实现：
   skill = db.query(Skill).filter(name="数据导出").first()
   script_code = skill.script_content  # 从 DB 读取脚本代码
   result = await execute_in_sandbox(script_code, context)  # 受控执行
   return result

4. 执行方式：
   - 通过 exec() 在受控命名空间中执行脚本代码
   - 注入 context 对象作为脚本与框架的唯一交互接口
   - 限制 __builtins__，禁止 import os / subprocess 等危险模块
   - MVP 阶段使用 exec() + 受限命名空间，后续可升级为 RestrictedPython
```

**context 对象 API（注入脚本的受控接口）：**

| API | 说明 | 示例 |
|-----|------|------|
| `context.arguments` | 用户传入的参数（JSON 解析后的 dict） | `context.arguments["project_id"]` |
| `context.call_tool(name, params)` | 调用当前 Agent 已绑定的工具 | `await context.call_tool("get_project_profit", {"project_id": "P1001"})` |
| `context.get_user_id()` | 获取当前用户 ID | `context.get_user_id()` |
| `context.get_extra_params()` | 获取透传参数 | `context.get_extra_params()` |
| `context.http_get(url)` / `context.http_post(url, data)` | 受限 HTTP 请求（仅白名单域名） | `await context.http_get("https://api.internal/data")` |

**安全约束：**

| 约束 | 说明 |
|------|------|
| 语言限制 | 仅支持 Python |
| 模块白名单 | `json`, `math`, `datetime`, `re`, `collections`, `itertools` 等纯计算模块 |
| 禁止模块 | `os`, `sys`, `subprocess`, `importlib`, `shutil`, `socket` 等系统/网络模块 |
| 文件系统 | 禁止访问 |
| 网络请求 | 仅通过 `context.http_get/post` 访问白名单域名 |
| 执行超时 | 单次执行最大 30 秒 |

**工具定义：**

```python
@tool
async def execute_script(skill_name: str, arguments: str = "") -> str:
    """执行指定 Skill 的脚本代码。
    脚本在受控环境中运行，通过 context 对象与框架交互。

    Args:
        skill_name: 技能名称
        arguments: 传递给脚本的参数（JSON 字符串）
    Returns:
        脚本执行结果
    """
    skill = await db.query(Skill).filter(name=skill_name, mode="script").first()
    if not skill:
        return f"脚本技能 '{skill_name}' 不存在"
    return await execute_in_sandbox(skill.script_content, arguments)
```

**脚本编写示例：**

```python
# 存储在 DB script_content 字段中
async def execute(context):
    """Skill 脚本入口（固定签名）"""
    project_id = context.arguments.get("project_id")

    # 调用已绑定的工具获取数据
    profit = await context.call_tool("get_project_profit", {"project_id": project_id})

    # 纯计算逻辑（不需要 LLM 推理）
    import json
    data = json.loads(profit)
    summary = f"项目 {project_id} 利润率: {data['profit_rate']}%"

    return summary
```

#### 4.6.6 Skill 三种触发方式的执行流程

Skill 支持三种触发方式，可组合使用。触发方式决定的是**谁发起调用**，与执行模式（工具/脚本）正交。

##### 1. 用户手动触发（`user_invocable = true`）

用户在对话输入框中以 `/技能名 参数` 形式发送消息，**由框架拦截处理，不经过 LLM 判断**。

```
用户输入: "/项目诊断 P1001"
    │
    ▼
┌─ 框架拦截层 ──────────────────────────────────────────────────┐
│                                                                │
│  1. chat_service 收到消息，检测到以 "/" 开头                     │
│  2. 解析命令：skill_name = "项目诊断"，arguments = "P1001"       │
│  3. 查询 DB：该 Skill 是否存在 + 是否绑定到当前 Scene             │
│     + user_invocable 是否为 true                                │
│  4. 校验失败 → 返回错误提示（"技能不存在"/"未授权"等）              │
│                                                                │
│  校验通过 ↓                                                     │
│                                                                │
│  5. 根据执行模式分流：                                           │
│     ┌─ 工具模式 ─────────────────────────────────────────────┐ │
│     │ 组装 instruction：                                      │ │
│     │   "用户触发了技能【项目诊断】，参数: P1001                  │ │
│     │    以下是该技能的完整执行指令：                              │ │
│     │    {skill.instruction 全文}"                             │ │
│     │ → 将组装后的内容作为用户消息发送给 Agent                    │ │
│     │ → Agent 按指令执行（调用依赖工具 → 生成结果）               │ │
│     └────────────────────────────────────────────────────────┘ │
│     ┌─ 脚本模式 ─────────────────────────────────────────────┐ │
│     │ 直接调用 execute_script(skill_name, arguments)           │ │
│     │ → 从 DB 读取 script_content → 沙箱执行 → 返回结果        │ │
│     │ → 将脚本执行结果作为上下文发送给 Agent 进行总结/格式化      │ │
│     └────────────────────────────────────────────────────────┘ │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

**关键设计**：

| 要点 | 说明 |
|------|------|
| 拦截时机 | 消息进入 Agent 之前，由框架层（chat_service）统一拦截 |
| 匹配规则 | 精确匹配 Skill 的 `name` 字段（不区分大小写） |
| 参数传递 | `/技能名` 后的所有文本作为 `arguments` 字符串传入 |
| 权限校验 | 必须满足：Skill 存在 + 绑定到当前 Scene + `user_invocable = true` |
| 错误处理 | 校验失败时直接返回友好提示，不发送给 Agent |

##### 2. LLM 自动触发（`llm_callable = true`）

Agent 在推理过程中根据对话上下文自主决定是否使用某个 Skill。这是 4.6.3 和 4.6.4 已描述的标准流程：

- assembler 将 Skill 的 `name + description` 注入 Prompt 作为索引
- LLM 判断需要 → 调用 `get_skill_content` 或 `execute_script`
- 该触发方式不需要框架额外处理

##### 3. 定时触发（`cron_expression` 已配置）

由 APScheduler 在 cron 时间到达时自动触发，详见 4.7 定时任务架构。

#### 4.6.7 create_scheduled_task — 创建定时任务

**定位**：仅编排场景（含 MainAgent + SubAgent 的 Scene）的 MainAgent 可用。单 Agent 场景不提供此工具。

**约束**：

- 只能为**当前 Scene** 创建定时任务，不可指定其他 Scene 或 Agent
- `scene_id` 和 `creator_id` 从 `request_context` 自动获取，不暴露给 Agent
- `notify_user_id` 自动取当前对话的三方 `user_id`——谁让 Agent 创建的就通知谁
- `extra_params`（透传参数）自动从 `request_context` 获取并持久化，定时任务唤起时作为任务描述的一部分传给 Agent
- `task_prompt` 必须自包含所有执行所需上下文（项目 ID、指标名、关注点等）
- 用户自定义的通知条件（如"占用率>80%时通知我"）由 Agent 嵌入 `task_prompt`

**工具定义：**

```python
@tool
async def create_scheduled_task(
        name: str,
        schedule: str,
        task_prompt: str,
) -> str:
    """为当前场景创建定时任务。定时到达时系统会唤醒当前场景的 Agent 并发送 task_prompt。

    task_prompt 必须包含执行所需的全部具体信息（如项目ID、指标名称等），
    不要使用模糊描述，因为定时执行时没有对话上下文。
    如果用户指定了通知条件（如"超过阈值时才通知"），必须将条件写入 task_prompt。

    Args:
        name: 任务名称（简短描述，如"每日异常项目扫描"）
        schedule: cron 表达式（如 "0 8 * * *" 表示每天8点）
        task_prompt: 自包含的任务描述，作为定时执行时发给 Agent 的消息
    Returns:
        创建结果，包含任务 ID
    """
    scene_id = get_scene_id()          # 从 request_context 获取
    user_id = get_user_id()            # 从 request_context 获取（三方 user_id）
    extra_params = get_extra_params()  # 从 request_context 获取（透传参数）
    token = get_token()                # 从 request_context 获取（可选，chat 场景可能有）

    task = ScheduledTask(
        name=name,
        scene_id=scene_id,
        schedule=schedule,
        task_prompt=task_prompt,
        notify_user_id=user_id,        # 通知对象 = 当前对话用户
        creator_id=user_id,
        creator_extra_params=extra_params,  # 持久化透传参数，定时任务唤起时使用
    )
    await db.add(task)
    await scheduler_service.refresh()  # 通知调度器加载新任务
    return f"定时任务 '{name}' 创建成功，ID: {task.id}，调度: {schedule}"
```

### 4.7 定时任务架构

定时任务是 MVP 的重要能力，但设计上保持简单：**仅支持 Agent 类型任务**（定时唤醒 Scene 的 Agent 执行），不支持 Hook/Webhook
类定时任务。

> **MVP 阶段说明**：定时任务不提供可视化管理页面。任务仅通过 Agent 对话中调用 `create_scheduled_task` 工具创建，任务数据存储在 DB 的 `scheduled_tasks` 表中。管理员可通过直接操作数据库进行任务的暂停/删除等管理操作。后续版本计划开放定时任务管理页面。

#### 4.7.1 整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│                     SchedulerService                            │
│  (APScheduler, 启动时从 DB 加载所有 status=active 的任务)         │
│                                                                 │
│  ┌──────────┐   任务 CRUD 后                                    │
│  │ JobStore  │ ←── refresh() 重新加载                            │
│  │ (DB-back) │                                                  │
│  └─────┬────┘                                                   │
│        │ cron 时间到达                                           │
│        ▼                                                        │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  TaskDispatcher                                          │   │
│  │                                                          │   │
│  │  1. 从 DB 读取任务详情（task_prompt, scene_id, creator_*） │   │
│  │  2. 设置 request_context（user_id + extra_params）        │   │
│  │  3. 调用 assembler.assemble(scene_id) 组装 Agent          │   │
│  │  4. agent.ainvoke(task_prompt) 作为用户消息执行             │   │
│  │  5. 收集执行结果 result                                    │   │
│  │  6. 写入 task_execution_logs 表                           │   │
│  │                                                          │   │
│  └────────────────────────┬─────────────────────────────────┘   │
│                           │                                     │
│                           ▼                                     │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  TaskNotifier                                            │   │
│  │                                                          │   │
│  │  1. 读取该任务的 notify_user_id（创建时自动记录）           │   │
│  │  2. 读取所属 Scene 的 notification_channel                │   │
│  │  3. 判断 Agent 执行结果是否有需通知的内容                   │   │
│  │     （Agent 在 task_prompt 中已包含用户自定义条件，          │   │
│  │      如"占用率>80%时通知"，Agent 执行时自行判断）            │   │
│  │  4. 有结果 → 调用内置通知 Tool 发送给 notify_user_id       │   │
│  │     无结果 → 仅写日志，不发送通知                           │   │
│  │                                                          │   │
│  └──────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

**关键设计决策：**

| 决策点  | 方案                     | 理由                                |
|------|------------------------|-----------------------------------|
| 任务类型 | 仅 Agent 任务             | MVP 简化；Agent 可覆盖 Skill/Webhook 场景 |
| 创建权限 | 仅编排 Scene 的 MainAgent  | 单 Agent 无需定时任务能力                  |
| 任务归属 | 只能创建当前 Scene 的任务       | 避免跨场景调度的复杂性                       |
| 通知对象 | 自动取创建时的三方 user_id     | 谁让 Agent 创建的就通知谁，无需手动配置         |
| 通知条件 | 用户自然语言描述，嵌入 task_prompt | Agent 执行时自行判断是否满足条件             |
| 通知渠道 | Scene 级配置（内置通知 Tool）     | 编排时统一选择，系统后置发送                   |
| 执行身份 | 使用创建者的 user_id + extra_params | 定时任务无实时 token，MCP 层用 user_id + extra_params 自行获取 token |

#### 4.7.2 任务生命周期

```
Agent 调用 create_scheduled_task 工具
    │
    ▼
scheduled_tasks 表写入（status=active）
    │
    ▼
scheduler_service.refresh() → APScheduler 加载新 Job
    │
    ▼
cron 时间到达 → TaskDispatcher 执行
    │
    ├── 成功 → task_execution_logs（status=success）
    │          → TaskNotifier（Agent 产出了需通知的结果 → 发送给 notify_user_id）
    │
    └── 失败 → task_execution_logs（status=failed, error_message）
               → TaskNotifier（发送失败通知给 notify_user_id）
```

**任务状态流转：**

| 状态        | 说明            |
|-----------|---------------|
| `active`  | 正常调度中         |
| `paused`  | 暂停（可通过管理页面恢复） |
| `deleted` | 逻辑删除，不再调度     |

#### 4.7.3 与现有架构的复用

定时任务执行**完全复用现有的 Agent 执行链路**，不另建执行引擎：

```
正常对话:  WebSocket → chat_service → assembler → Agent → event_handler → WS 响应
定时任务:  APScheduler → TaskDispatcher → assembler → Agent → 结果写 DB + 通知
```

差异仅在触发入口和结果出口：

| 对比   | 正常对话                     | 定时任务                     |
|------|--------------------------|--------------------------|
| 触发方式 | 用户 WebSocket 消息          | APScheduler cron 触发      |
| 输入   | 用户实时输入的消息                | task_prompt（创建时写好）       |
| 执行   | 同一个 assembler + Agent 链路 | 同一个 assembler + Agent 链路 |
| 输出   | 流式推送到 WebSocket          | 结果写入 execution_logs      |
| 通知   | 无（用户实时看到）                | 读 Scene 通知渠道 → 发送给任务的 notify_user_id |

#### 4.7.4 热更新联动

| 变更                  | 生效方式                                                       |
|---------------------|------------------------------------------------------------|
| Agent 创建新定时任务       | 写入 DB → 调 `scheduler_service.refresh()` → APScheduler 立即加载 |
| 管理页面暂停/删除任务         | 更新 DB status → 调 `scheduler_service.refresh()` → Job 移除    |
| 修改 Scene 的 Agent 配置 | 无需刷新调度器；下次任务执行时 assembler 自动读最新配置                          |
| 修改 Scene 的通知渠道      | 无需刷新调度器；下次通知时 TaskNotifier 自动读最新配置                         |

### 5.2 请求处理数据流（用户消息 → Agent 执行 → 流式响应）

```

│  └── 创建/获取 session                                                  │
│                                                                        │
└───────────────────────────────────┬────────────────────────────────────┘
                                    │
                                    ▼
┌─ 服务层 ──────────────────────────────────────────────────────────────┐
│                                                                        │
│  chat_service.py                                                       │
│  ├── session_manager.get_or_create(user_id, scene_id)                  │
│  ├── assembler.assemble(scene_id, user_id, session_id, extra_params)   │

```

### 5.3 参数透传机制

框架的认证模型是"**user_id 必传 + extra_params 透传 + token 可选兼容**"。核心原因：定时任务被唤起时没有实时 chat 场景，不存在有效 token，因此 token 获取下沉到 MCP 层。

#### 5.3.1 参数来源与存储

| 参数 | 来源 | 是否必传 | 存储位置 | 说明 |
|------|------|---------|---------|------|
| `user_id` | 三方系统 | **必传** | `request_context` + `sessions` 表 | 三方用户唯一标识，全链路使用 |
| `extra_params` | 三方系统 | 可选 | `request_context` + `scheduled_tasks` 表（持久化） | 业务线自定义参数（如 topgroup_id、park_id 等），JSON 对象 |
| `token` | 三方系统 | 可选 | `request_context`（仅内存，不持久化） | chat 场景下如传入则透传，定时任务场景下不存在 |

> **约束**：`user_id` + `extra_params` 的组合必须能唯一定位业务场景下的确定用户。例如集团 + 用户 ID、车场 + 用户 ID 等。

#### 5.3.2 两种场景下的参数流转

**场景一：实时 Chat（有 token）**

```
三方系统 → WebSocket chat 消息
    │  data: { user_id, extra_params, token, message }
    ▼
ws_router → set_request_context(user_id, extra_params, token)
    ▼
Agent 执行 → 工具调用
    ├── get_user_id()       → "zhangsan_001"
    ├── get_extra_params()  → {"topgroup_id": "group-001", "park_id": "park-123"}
    └── get_token()         → "Bearer xxx..."  ← 有值
    ▼
mcp_hub 透传给 MCP Server
    ├── user_id + extra_params （始终透传）
    └── token （有值则透传，MCP 可直接鉴权，跳过自行获取 token）
```

**场景二：定时任务唤起（无 token）**

```
APScheduler cron 触发
    ▼
TaskDispatcher
    ├── 从 DB 读取：creator_id, creator_extra_params
    ├── set_request_context(user_id=creator_id, extra_params=creator_extra_params)
    │   （注意：此时没有 token，get_token() 返回 None）
    ▼
Agent 执行 → 工具调用
    ├── get_user_id()       → "zhangsan_001"
    ├── get_extra_params()  → {"topgroup_id": "group-001", "park_id": "park-123"}
    └── get_token()         → None  ← 无值
    ▼
mcp_hub 透传给 MCP Server
    ├── user_id + extra_params （始终透传）
    └── token = None（MCP Server 需自行用 user_id + extra_params 获取 token）
```

#### 5.3.3 MCP Server 侧的鉴权策略

```
MCP Server 收到工具调用请求
    │
    ├── 检查请求中是否携带 token
    │   ├── 有 token → 直接使用，调用业务 API
    │   └── 无 token → 用 user_id + extra_params 向认证服务换取 token → 调用业务 API
    │
    └── 返回工具执行结果
```

### 5.5 记忆数据流

Agent 启动时的 Prompt 上下文由多层组成，采用**两层记忆 + 对话上下文**架构：

```
┌─ Prompt 记忆架构 ─────────────────────────────────────────────────────┐
│                                                                        │
│  ┌─ L1: 基础提示词（静态）──────────────────────────────────────────┐  │
│  │  来源：DB agents 表 system_prompt + context_builder 模板渲染      │  │
│  │  时机：assembler 组装时一次性生成                                  │  │
│  └────────────────────────────────────────────────────────────────┘  │
│                                                                        │
│  ┌─ Layer 2: 用户画像（跨 Scene 共享）──── ← MVP 启用 ────────────┐  │
│  │  来源：PowerMem UserMemory.search(user_id)                        │  │
│  │  隔离：按 user_id，所有 Scene 共享同一份画像                      │  │
│  │  内容："该用户关注华东区项目，偏好看利润指标，沟通风格偏简洁"      │  │
│  │  时机：每次会话开始时加载，注入 system_prompt 尾部                │  │
│  └────────────────────────────────────────────────────────────────┘  │
│                                                                        │
│  ┌─ Layer 1: 场景记忆（按 Scene 隔离）──── ← MVP 启用 ────────────┐  │
│  │                                                                    │  │
│  │  隔离：按 scene_id + user_id，不同 Scene 的记忆互不可见           │  │
│  │                                                                    │  │
│  │  ├─ 语义召回 ──────────────────────────────────────────────────┐  │  │
│  │  │  来源：Memory.search(query, user_id, filters={scene_id})   │  │  │
│  │  │  内容：与当前查询语义相关的该场景下的历史事实                │  │  │
│  │  │  时机：每轮用户消息到达时，基于消息内容做语义检索            │  │  │
│  │  └────────────────────────────────────────────────────────────┘  │  │
│  │                                                                    │  │
│  │  ├─ 时间召回 ──────────────────────────────────────────────────┐  │  │
│  │  │  来源：Memory.search(user_id, filters={scene_id, 时间范围})│  │  │
│  │  │  内容：该场景下近期交互的关键事实（基于时间近邻）            │  │  │
│  │  │  时机：会话开始时按时间范围加载                              │  │  │
│  │  └────────────────────────────────────────────────────────────┘  │  │
│  │                                                                    │  │
│  └────────────────────────────────────────────────────────────────┘  │
│                                                                        │
│  ┌─ 对话上下文（当前会话）──────────────── ← MVP 启用 ────────────┐  │
│  │  来源：LangGraph Checkpointer 自动管理（SeekDBCheckpointSaver）   │  │
│  │  内容：当前 session 的完整消息历史                                │  │
│  │        未压缩时 → 原始 messages 列表                             │  │
│  │        压缩后 → 历史摘要 + 最近 N 轮原始消息                     │  │
│  │  时机：checkpointer 按 thread_id(=session_id) 自动加载/存储      │  │
│  └────────────────────────────────────────────────────────────────┘  │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘
```

**两层记忆的隔离设计：**

| 层 | 隔离键 | PowerMem API | 共享范围 | 典型内容 |
|---|--------|-------------|---------|---------|
| Layer 1: 场景记忆 | `scene_id + user_id` | `Memory`（metadata filter: `scene_id`） | 仅当前 Scene 可见 | "上次分析了项目 P1001 的利润率为 12%"、"用户关注停车场A的收入异常" |
| Layer 2: 用户画像 | `user_id` | `UserMemory` | 所有 Scene 共享 | "偏好简洁回复"、"常关注华东区项目"、"管理层角色" |

**会话管理职责划分：**

```
┌─ LangGraph Checkpointer（自动管理）───────────────────────────────────┐
│                                                                        │
│  职责：对话消息的存储 / 加载 / 状态恢复                                │
│  实现：SeekDBCheckpointSaver（自研适配）                               │
│  粒度：按 thread_id（= session_id）自动持久化                          │
│  内容：完整 messages 列表、工具调用记录、Agent 中间状态                  │
│  压缩：超阈值时触发 LLM 摘要压缩，压缩结果写回 checkpointer 状态       │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘

┌─ 框架 sessions 表（自管理）────────────────────────────────────────────┐
│                                                                        │
│  职责：管理 user_id ↔ session_id ↔ scene_id 映射关系                   │
│  ├── 用户连接时 → 查 sessions 表 → 加载该用户在该 Scene 的会话列表     │
│  ├── 新建对话 → 创建 session 记录（含 scene_id）→ 生成 session_id      │
│  │   → 作为 thread_id 传给 checkpointer                               │
│  └── 恢复对话 → 用 session_id 作为 thread_id                           │
│      → checkpointer 自动恢复完整对话状态                                │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘
```

**Session 结束后异步处理：**

```
┌─ Session 关闭后（close_session / WS 断连宽限期后）──────────────────────┐
│                                                                        │
│  # 对话上下文：checkpointer 已自动持久化（无需手动操作）                │
│                                                                        │
│  # Layer 1 场景记忆抽取（MVP 启用）                                    │
│  scene_memory.store(user_id, scene_id, messages)                       │
│      → PowerMem Memory.add(messages, user_id,                          │
│            infer=True, metadata={"scene_id": scene_id})                │
│      → 内部自动：事实抽取 → 去重 → 冲突解决 → 遗忘曲线评估 → 存储     │
│      → 存储的记忆条目携带 scene_id 元数据，后续召回时按 scene_id 过滤  │
│                                                                        │
│  # Layer 2 用户画像更新（MVP 启用）                                    │
│  user_profile.update(user_id, messages)                                │
│      → PowerMem UserMemory.add(messages, user_id)                      │
│      → 内部自动提取用户偏好/关注点增量并更新全局画像                    │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘
```

## 六、数据库设计

> **数据库选型**：统一使用 **SeekDB**（OceanBase AI 原生数据库），兼容 MySQL 5.7/8.0 协议。
>
> **部署模式**：公司云端统一部署一套 SeekDB 实例，各业务线通过连接地址（`mysql+aoceanbase://user:pass@host:2881/biz_db`
> ）接入，一个业务线一个独立库。
>
> **选型理由**：
> - PowerMem 的 UserMemory（用户画像）功能要求 OceanBase 作为存储后端，SeekDB 天然满足
> - 向量搜索 + 全文检索 + 关系型 CRUD 统一在一个引擎内，无需拼接多个数据库
> - 后台配置、对话历史、记忆系统、LangGraph 状态均存储在同一个库中，运维简单
> - 各业务线库级隔离，互不影响
>
> **约束**：所有表不使用外键约束（FK），引用完整性由应用层保证，避免级联删除的隐式行为。
>
> **自研适配**：LangGraph 官方无 SeekDB Checkpointer 实现，需自行开发 `SeekDBCheckpointSaver`（继承 `BaseCheckpointSaver`
> ，参考 PostgresSaver 蓝图，将 psycopg 调用替换为 MySQL 兼容驱动）。

**连接配置（`.env`）：**

```
SEEKDB_HOST=seekdb.company.internal
SEEKDB_PORT=2881
SEEKDB_USER=root@biz_tenant
SEEKDB_PASSWORD=****
SEEKDB_DATABASE=yongce_bi          # 每个业务线独立库名
```

**库内表分类：**

| 分类           | 表                                                                                                                                   | 说明                                               |
|--------------|-------------------------------------------------------------------------------------------------------------------------------------|--------------------------------------------------|
| 后台配置         | agents, skills, scenes, mcp_servers, mcp_tools, agent_tool_bindings, agent_skill_bindings, memory_configs | 管理页面 CRUD（memory_configs MVP 不暴露）                |
| 会话映射         | sessions                                                                                                                            | user_id ↔ session_id 映射关系                        |
| 对话历史         | checkpoints, checkpoint_writes                                                                                                      | LangGraph Checkpointer（SeekDBCheckpointSaver）自动管理 |
| 记忆系统         | PowerMem 自动创建的表                                                                                                                     | PowerMem SDK 通过 `provider: 'oceanbase'` 配置连接同一个库 |
| 定时任务         | scheduled_tasks, task_execution_logs                                                                                                | 定时任务定义与执行日志                                      |

### 7.1 REST API 总览

所有管理 API（非 WebSocket）统一前缀 `/api`，使用 `core/utils.py` 中的标准响应函数：

**成功响应（`success_response`）：**

```json

  "code": 200,
  "message": "成功",
  "data": {
    ...
  }
}
```

**错误响应（`error_response`）：**

```json

  "code": 500,
  "message": "Agent name already exists",
  "data": ""
}
```

**响应函数定义（`core/utils.py`）：**

```python
from fastapi.responses import JSONResponse
from fastapi.encoders import jsonable_encoder


def success_response(message="成功", data=""):
    """通用的成功响应"""
    response_content = {
        "code": 200,
        "message": message,
        "data": data
    }
    return JSONResponse(content=jsonable_encoder(response_content))


def error_response(message="失败", data=""):
    """通用的错误响应"""
    response_content = {
        "code": 500,
        "message": message,
        "data": data
    }
    return JSONResponse(content=jsonable_encoder(response_content))
```

**使用规范：**

- 所有 REST API 端点统一使用 `success_response()` / `error_response()` 返回
- WebSocket 接口不使用此格式，保持现有的事件协议
- `data` 字段：成功时为业务数据（对象/列表），失败时为空字符串或错误详情

```

### 7.9 WebSocket 协议

**连接端点：** `ws://host:port/ws`

> **三方对接说明**：此 WebSocket 接口即三方系统的唯一对接入口。外部系统（如业务前端、第三方应用）通过此 WebSocket 协议与 Agent 进行对话交互。MVP 不另设独立的 HTTP Chat API——所有对话请求统一走 WebSocket，以支持流式响应、实时取消和多轮会话管理。调试页面也复用同一协议。

> **连接模型**：以 session 为单位保持 WebSocket 长连接。一个 session 内的多轮对话在同一条 WS 连接内完成。会话结束时前端发送 `close_session` 关闭会话。

**会话生命周期：**

```
┌─ 新建对话 ────────────────────────────────────────────────────────┐
│                                                                    │
│  1. 前端建立 WebSocket 连接                                         │
│  2. 前端发送 create_session（user_id + scene_id + extra_params）    │
│  3. 后端创建 session 记录 → 返回 session_id                         │
│  4. 前端使用 session_id 发送 chat 消息（可多轮）                     │
│  5. 对话结束 → 前端发送 close_session                               │
│  6. 后端标记 session 关闭 → 异步触发记忆抽取                         │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘

┌─ 恢复对话 ────────────────────────────────────────────────────────┐
│                                                                    │
│  1. 前端建立 WebSocket 连接                                         │
│  2. 前端发送 chat（session_id + scene_id + user_id + message）     │
│  3. 后端通过 session_id 恢复历史对话（checkpointer 自动加载）        │
│  4. 继续多轮对话                                                    │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘

┌─ 异常兜底 ────────────────────────────────────────────────────────┐
│                                                                    │
│  WebSocket 断连（网络异常/刷新页面）                                  │
│      → 后端检测到连接断开                                            │
│      → 如果关联的 session 仍为 active 且有对话内容                    │
│      → 等待宽限期（30 秒）                                          │
│      → 宽限期内未重连 → 标记 session closed → 触发记忆抽取            │
│      → 宽限期内重连 → session 继续，不触发抽取                        │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

**客户端发送 Action：**

| Action           | 说明       | data 字段                                                |
|------------------|----------|--------------------------------------------------------|
| `create_session` | 创建新会话    | `user_id`(必传), `scene_id`(必传), `extra_params`(可选,JSON对象), `token`(可选) |
| `chat`           | 发送消息     | `user_id`(必传), `session_id`(必传), `scene_id`(必传), `message`(必传), `extra_params`(可选), `token`(可选) |
| `close_session`  | 关闭会话     | `session_id`(必传)                                       |
| `cancel`         | 取消当前请求   | `session_id`(必传)                                       |
| `ping`           | 心跳       | 无                                                      |
| `get_status`     | 查询会话状态   | `session_id`(必传)                                       |

**create_session 消息示例：**

```json
{
  "action": "create_session",
  "data": {
    "user_id": "zhangsan_001",
    "scene_id": "scene-uuid-001",
    "extra_params": {
      "topgroup_id": "group-001",
      "park_id": "park-123"
    },
    "token": "Bearer xxx..."
  }
}
```

**create_session 成功响应：**

```json
{
  "action": "session_created",
  "data": {
    "session_id": "session-uuid-generated",
    "scene_id": "scene-uuid-001",
    "user_id": "zhangsan_001"
  }
}
```

**chat 消息示例：**

```json
{
  "action": "chat",
  "data": {
    "user_id": "zhangsan_001",
    "session_id": "session-uuid-001",
    "scene_id": "scene-uuid-001",
    "message": "帮我看看华东区项目的利润情况",
    "extra_params": {
      "topgroup_id": "group-001",
      "park_id": "park-123"
    },
    "token": "Bearer xxx..."
  }
}
```

> **字段说明**：
> - `user_id`（必传）：三方用户唯一 ID，用于会话隔离、记忆隔离、MCP 权限透传
> - `session_id`（必传）：由 `create_session` 返回，标识一次对话。恢复对话时传入已有 session_id
> - `scene_id`（必传）：指定对话连接到哪个 Agent 编排场景。后端保存 session 时同步记录 scene_id，保证恢复对话时知道对应的 Agent 场景
> - `extra_params`（可选）：业务线自定义的透传参数 JSON 对象（如集团 ID、车场 ID 等）。框架全链路透传，MCP 层可利用这些参数定位用户并获取 token 鉴权
> - `token`（可选）：三方系统直传的认证令牌。如传入则存入上下文，工具调用时优先使用，方便 MCP 层直接鉴权。定时任务场景下不存在 token

## 八、非功能性需求

### 8.1 并发

| 指标                | 要求       |
|-------------------|----------|
| 同时在线 WebSocket 连接 | ≥ 50     |
| 同一 Agent 并行会话     | ≥ 10     |
| 管理 API 并发         | ≥ 20 QPS |

- 每个用户会话独立 Agent 实例，共享只读配置
- WebSocket 连接池管理，支持心跳检测和断线重连
- DB 连接池（async，使用 OceanBase 异步 dialect `mysql+aoceanbase`）

### 8.2 性能

| 指标            | 要求                                   |
|---------------|--------------------------------------|
| Agent 组装耗时    | ≤ 500ms（从 DB 读取配置到 Agent 实例就绪）       |
| 首 token 延迟    | ≤ 3s（从用户发送消息到第一个 token 返回）           |
| 工具调用开销        | 框架层 ≤ 50ms（不含 MCP Server / LLM 处理时间） |
| 管理 API 响应时间   | ≤ 200ms（常规 CRUD）                     |
| MCP Schema 刷新 | ≤ 10s（含网络请求）                         |

**配置缓存策略：**

- Agent / Scene 配置：每次请求从 DB 读取（保证热更新）
- MCP 工具 Schema：DB 持久化缓存，页面手动刷新更新
- 后续可增加内存缓存 + TTL 优化

### 8.3 可靠性

| 场景            | 处理方式                                                 |
|---------------|------------------------------------------------------|
| LLM 调用超时      | 重试 1 次，仍失败则返回 error 事件，不阻塞会话                         |
| MCP 工具调用失败    | 返回 tool_end 事件（含错误信息），Agent 可根据错误决定下一步               |
| WebSocket 断连  | 客户端重连后可通过 session_id 恢复会话状态                          |
| SubAgent 执行超时 | DeepAgents 框架层设置 max_iterations，超过后返回超时提示给 MainAgent |
| 定时任务执行失败      | 记录错误日志到 task_execution_logs，根据 Scene 通知配置发送失败通知      |
| 定时任务执行超时      | 设置单次执行最大时长（默认 300s），超时后终止并记录 timeout 状态              |
| DB 写入失败       | 事务回滚，返回 error 响应                                     |
| 记忆/画像更新失败    | 异步任务，不影响对话响应，记录错误日志                                  |

### 8.4 日志与追踪

| 维度      | 说明                                                         |
|---------|------------------------------------------------------------|
| traceId | 每个请求分配唯一 traceId，贯穿所有日志和事件                                 |
| 结构化日志   | JSON 格式，包含 timestamp、level、traceId、module、message          |
| 关键日志点   | 请求接收、Agent 组装完成、LLM 调用、工具调用、SubAgent 事件（task 工具）、记忆操作、请求完成 |
| 日志级别    | DEBUG（LLM 完整消息）、INFO（关键节点）、WARN（超时/重试）、ERROR（异常）           |

### 8.5 安全

| 维度        | 措施                                                 |
|-----------|----------------------------------------------------|
| 认证        | 平台不自建用户体系；三方系统通过 WebSocket 消息传递 `user_id` + `extra_params`，框架透传给 MCP Server，MCP 层自行获取 token 鉴权。chat 场景如直传 token 也会透传 |
| 管理页面      | MVP 阶段无独立鉴权（内网部署），后续按需增加                            |
| 输入校验      | API 层使用 Pydantic 模型校验所有输入参数                        |
| SQL 注入    | 使用 ORM（SQLAlchemy），禁止拼接 SQL                        |
| Prompt 注入 | MVP 阶段做基础防护（输入长度限制、敏感词过滤），后续增加多层防御                 |
| 脚本沙箱      | Skill 脚本模式在受限沙箱中执行，禁止文件系统访问和非白名单网络请求               |
| 数据隔离      | 记忆和会话按三方 user_id 隔离，不同用户数据互不可见                     |
