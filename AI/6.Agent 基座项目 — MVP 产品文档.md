## 一、产品概述

### 1.1 产品定位

Agent 基座是一个**各业务线单独部署**的 AI Agent 开发与管理平台。每个业务线拿到同一份基座代码，通过管理页面完成业务配置，独立部署、独立运行。

### 1.2 核心能力

| 能力            | 说明                                       |
|---------------|------------------------------------------|
| Agent 全生命周期管理 | 页面创建、编辑、停用 Agent，配置模型与 Prompt            |
| Skill 管理      | 在线上传 skill.md 自动解析，或页面直接创建编辑             |
| Scene 可视化编排   | 在页面上配置 MainAgent + SubAgent 主从关系，绑定工具和技能 |
| MCP 工具接入      | 页面配置 MCP Server，刷新获取工具列表，按 Server 分组导入   |
| 记忆系统          | 框架提供记忆抽取与召回能力，业务线自定义抽取层级与风格              |
| 调试能力          | 选择任意 Agent 或 Scene 直接对话调试，查看完整调用链路       |
| 三方接入          | WebSocket / HTTP API 供外部系统调用             |

### 1.3 用户角色

| 角色    | 说明                              | 使用页面                 |
|-------|---------------------------------|----------------------|
| 配置管理员 | 负责 Agent、Skill、Scene、MCP 的配置与管理 | 全部管理页面               |
| 开发者   | 开发 MCP Server 工具、编写 Skill 指令    | MCP 配置、Skill 管理、调试页面 |
| 调试人员  | 测试 Agent 对话效果、排查问题              | 调试页面                 |

## 二、核心概念与术语

### 2.1 概念定义

| 概念             | 定义             | 说明                                       |
|----------------|----------------|------------------------------------------|
| **Agent**      | 一个 AI 智能体单元    | 拥有名称、模型配置、系统提示词，本身不具有"主/从"属性             |
| **Skill**      | Agent 的结构化复合能力 | 由执行指令（Prompt）+ 依赖工具组成，粒度高于单个 Tool        |
| **Scene**      | 一组 Agent 的编排配置 | 定义 MainAgent + SubAgent 的主从关系，以及工具和技能的绑定 |
| **MCP Server** | 业务线部署的工具服务     | 遵循 MCP 协议，提供业务工具给 Agent 调用               |
| **Tool**       | 一个原子函数调用       | 分为内置工具（框架提供）和 MCP 工具（业务线提供）              |
| **Memory**     | 记忆系统           | 包括 Session 短期记忆、PowerMem 长期记忆、用户画像       |
| **MainAgent**  | 协调型 Agent      | 在某个 Scene 中担任主角色，负责意图理解和任务分派             |
| **SubAgent**   | 执行型 Agent      | 在某个 Scene 中担任从角色，专注某个业务领域的任务执行           |

### 2.2 概念关系

```
Agent（能力单元，无固有主从属性）
  │
  │  通过 Scene 编排确定角色
  │
  ▼
Scene（编排配置）
  │
  ├── main_agent ──→ 某个 Agent 充当 MainAgent
  ├── sub_agents ──→ 若干 Agent 充当 SubAgent
  │
  ├── tool_bindings ──→ 为每个 Agent 绑定工具
  │   ├── 内置工具（框架提供）
  │   └── MCP 工具（从 MCP Server 导入）
  │
  ├── skill_bindings ──→ 为每个 Agent 绑定技能
  │
  └── knowledge_bindings ──→ 为每个 Agent 绑定知识
```

**核心原则：定义与编排分离**

- 同一个 Agent 可以在 Scene A 中作为 MainAgent，在 Scene B 中作为 SubAgent
- 工具和技能的绑定由 Scene 决定，不写死在 Agent 定义中

### 2.3 关键区分

| 对比   | Tool               | Skill                      |
|------|--------------------|----------------------------|
| 本质   | 一个函数调用             | 结构化的执行指令                   |
| 粒度   | 原子操作（查一个数据）        | 复合任务流程（诊断分析）               |
| 来源   | 内置工具库 / MCP Server | 上传 skill.md / 页面创建         |
| 执行方式 | Agent 直接调用         | 工具模式（Prompt 注入）或脚本模式（沙箱运行） |
| 触发方式 | Agent 推理时自动调用      | 手动触发 / LLM 自动触发 / 定时触发     |

| 变量               | 说明       | 来源       |
|------------------|----------|----------|
| `{name}`         | Agent 名称 | Agent 定义 |

**交互逻辑：**

1. **创建流程**：点击"创建 Agent" → 填写表单 → 点击"保存" → 校验通过 → 调用 `POST /api/agents` → 成功后跳转到列表页

4. **名称唯一性校验**：输入名称后失焦时调用 `GET /api/agents/check-name?name=xxx` 校验
5. **Markdown 预览**：系统提示词编辑器支持实时预览模式切换

#### 3.3.2 Skill 上传解析流程

上传 skill.md 是 **辅助填写** 手段，**不会直接创建** Skill。上传的文件经后端解析后，自动填入创建页面的表单字段，用户确认或修改后通过"保存"按钮完成创建。

```
用户点击 [上传 skill.md]
    │
    ▼
选择本地 .md 文件
    │
    ▼
前端调用 POST /api/skills/parse（multipart/form-data）
    │
    ▼
后端解析文件（仅解析，不入库）
    │
    ├── 解析 YAML frontmatter（--- 之间的头部）
    │   ├── 提取 name, description, mode, user_invocable,
    │   │   model_invocable, schedule, argument_hint, requires
    │   │
    │   └── 校验格式合法性（字段类型、枚举值范围）
    │
    ├── 解析 Markdown body（--- 之后的内容）
    │   └── 提取为 instruction 字段
    │
    ├── 解析成功 → 返回结构化 JSON（不入库）
    │   └── 前端携带解析结果跳转到 **创建页**
    │       → 所有表单字段自动预填
    │       → 用户可检查、修改
    │       → 点击"保存"才真正调用 POST /api/skills 创建
    │
    └── 解析失败 → 返回错误信息
        └── 前端弹出提示（如："frontmatter 格式错误"或"缺少必填字段 name"）
```

> **设计原则**：创建页是新建 Skill 的唯一入口。上传解析只是"文件 → 表单"的快捷通道，确保所有 Skill 都经过人工确认后才写入系统。

#### 3.3.2.1 skill.md 标准格式

skill.md 文件由两部分组成：**YAML Frontmatter**（元数据头部）和 **Markdown Body**（执行指令）。

**完整格式模板：**

```markdown
---
# ====== 基础信息（必填） ======
name: "项目健康诊断"                    # Skill 名称，1-50 字符
description: "对指定项目进行全方位健康诊断分析"  # Skill 描述，1-200 字符

# ====== 调用控制（必填 mode，其余可选） ======
mode: tool                             # 执行模式：tool（工具模式）| script（脚本模式）
user_invocable: true                   # 用户可否手动 /命令 触发，默认 true
model_invocable: true                  # LLM 可否根据上下文自动触发，默认 true
schedule: null                         # cron 定时表达式，null 表示不启用定时触发
argument_hint: "<项目ID>"               # 参数提示，告知用户需提供什么参数，可选

# ====== 依赖声明（可选） ======
requires:
  tools:                               # 该 Skill 执行时依赖的工具列表
    - get_project_profit
    - get_operation_data

# ====== 脚本模式专属（仅 mode: script 时需要） ======
# script_path: "./scripts/export_report.py"
---

# 执行指令（Markdown 格式，作为 instruction 字段存储）

你正在执行【项目健康诊断】任务。

## 执行步骤
1. 调用 get_project_profit 获取项目财务数据
2. 调用 get_operation_data 获取运营数据
3. 综合分析，输出健康评分和关键发现

## 输出格式
- 健康评分（0-100）
- 关键发现（最多 5 条）
- 改进建议

## 约束
- $ARGUMENTS 为项目 ID，如果用户未提供则主动询问
- 数据精确，不估算不猜测
```

**字段说明：**

| 字段 | 位置 | 必填 | 类型 | 说明 |
|------|------|------|------|------|
| `name` | frontmatter | 是 | string | Skill 名称，对应表单"名称"字段 |
| `description` | frontmatter | 是 | string | Skill 描述，对应表单"描述"字段 |
| `mode` | frontmatter | 是 | enum | `tool` 或 `script`，对应表单"执行模式" |
| `user_invocable` | frontmatter | 否 | bool | 默认 `true`，对应表单"用户手动触发" |
| `model_invocable` | frontmatter | 否 | bool | 默认 `true`，对应表单"LLM 自动触发" |
| `schedule` | frontmatter | 否 | string/null | cron 表达式，非空时启用定时触发 |
| `argument_hint` | frontmatter | 否 | string | 参数提示文本，对应表单"参数提示" |
| `requires.tools` | frontmatter | 否 | list | 依赖工具名称列表，对应表单"依赖工具" |
| `script_path` | frontmatter | 条件 | string | 仅 `mode: script` 时需要，对应表单"脚本路径" |
| *(body)* | markdown | 是 | markdown | `---` 之后的全部内容，对应表单"执行指令" |

**frontmatter → 表单字段映射关系：**

```
skill.md                         创建页表单
─────────────                    ─────────────
name                      →     名称
description                →     描述
mode                       →     执行模式
user_invocable             →     用户手动触发（开关）
model_invocable            →     LLM 自动触发（开关）
schedule (非null)           →     定时触发（开关=开）+ 定时表达式
argument_hint              →     参数提示
requires.tools             →     依赖工具（标签列表）
script_path                →     脚本路径（仅脚本模式）
Markdown body              →     执行指令（Markdown 编辑器）
```

**解析校验规则：**

| 校验项 | 规则 | 失败提示 |
|--------|------|---------|
| frontmatter 存在性 | 文件必须包含 `---` 包裹的 YAML 头部 | "文件缺少 YAML frontmatter 头部（--- 包裹的元数据区域）" |
| name | 必填，1-50 字符 | "缺少必填字段 name" 或 "name 长度须在 1-50 字符之间" |
| description | 必填，1-200 字符 | "缺少必填字段 description" 或 "description 长度须在 1-200 字符之间" |
| mode | 必填，值为 `tool` 或 `script` | "缺少必填字段 mode" 或 "mode 取值须为 tool 或 script" |
| schedule | 若非 null，须为合法 cron 表达式 | "schedule 不是合法的 cron 表达式" |
| requires.tools | 若存在，须为字符串列表 | "requires.tools 格式错误，须为字符串列表" |
| Markdown body | 不可为空 | "文件缺少执行指令内容（Markdown body）" |

|----------|--------------|------|---------------|--------------------------|
| 名称       | 文本输入         | 是    | 1-50 字符，不可重复  | Skill 显示名称               |
| 描述       | 多行文本         | 是    | 1-200 字符      | 功能描述，同时作为 LLM 的 Skill 索引 |
| 执行模式     | 单选           | 是    | tool / script | 工具模式或脚本模式                |
| 用户手动触发   | 开关           | 否    | 布尔值，默认 true   | 用户可否通过 /命令 手动触发          |
| LLM 自动触发 | 开关           | 否    | 布尔值，默认 true   | LLM 可否根据上下文自动触发          |
| 定时触发     | 开关           | 否    | 布尔值，默认 false  | 是否启用 cron 定时触发           |

| 参数提示     | 文本输入         | 否    | 最大 100 字符     | 提示用户需要提供什么参数             |
| 依赖工具     | 标签输入         | 否    | 工具名需在系统中存在    | 该 Skill 执行时依赖的工具列表       |

| 脚本路径     | 文本输入         | 条件必填 | 脚本模式时必填       | Python 脚本文件路径            |

**条件显示逻辑：**

- 定时表达式字段：仅当"定时触发"开关开启时显示
- 脚本路径字段：仅当执行模式为 `script` 时显示

**交互逻辑：**

1. **创建流程**：填写表单 → 校验 → `POST /api/skills` → 成功跳转列表
2. **编辑流程**：加载数据 → 修改 → 校验 → `PUT /api/skills/:id` → 成功跳转列表
3. **上传预填**：从上传解析跳转来时，所有字段已预填，用户检查/修改后点击"保存"。此时与手动创建走**同一条 `POST /api/skills` 链路**，上传本身不创建数据
4. **依赖工具校验**：添加依赖工具时，调用 `GET /api/tools/search?keyword=xxx` 自动补全，只能选择系统中已导入的工具

    - **MCP 工具**：按 MCP Server 分组展示，每个 Server 下列出其已导入的工具，复选框勾选

**技能绑定区逻辑：**

- Tab 页同上，按 Agent 切换
- 点击"添加技能"弹出 Skill 选择弹窗，列出系统中所有已有 Skill，支持搜索
- 已绑定的 Skill 可拖动排序、可删除

1. **主 Agent 切换**：切换主 Agent 后，工具绑定和技能绑定 Tab 自动更新，原有绑定配置保留（如果 Agent 仍在列表中）
2. **添加子 Agent**：下拉选择 Agent，选项排除已在当前 Scene 中出现的 Agent
3. **删除子 Agent**：删除后该 Agent 的工具绑定和技能绑定配置一并清除
4. **保存**：调用 `POST /api/scenes`（创建）或 `PUT /api/scenes/:id`（编辑），一次性提交全部数据（基础信息 + 编排 + 绑定）
5. **依赖校验**：保存时校验 Skill 依赖的工具是否在该 Agent 的工具绑定中，缺少则弹出警告（非阻断，允许保存）

| `agent_router`  | Agent 定义的增删改查                                | `GET/POST/PUT/DELETE /api/agents`        |
| `skill_router`  | Skill 管理 + skill.md 上传解析                     | `POST /api/skills/parse`（仅解析不入库） |
| `scene_router`  | Scene 编排配置的增删改查                              | `GET/POST/PUT/DELETE /api/scenes`        |
| `mcp_router`    | MCP Server 管理 + 工具 Schema 刷新                 | `POST /api/mcp-servers/{id}/refresh`     |
| `memory_router` | 记忆配置管理                                       | `GET/PUT /api/memory-configs/{agent_id}` |
| `debug_router`  | 调试页面静态资源 + 调试专用 API                          | `GET /yc-ai/debug`                       |

#### 4.2.2 服务层

| 模块                   | 职责                                                                  |
|----------------------|---------------------------------------------------------------------|
| `agent_service`      | Agent CRUD、字段校验（name 唯一、model 合法）、关联 Scene 检查（删除前验证无引用）             |
| `skill_service`      | Skill CRUD、skill.md 文件上传解析（YAML front-matter → DB 记录）、依赖声明校验        |
| `scene_service`      | Scene CRUD、编排校验（main_agent 存在、sub_agents 存在、工具/技能引用合法）、Scene 启用/停用  |
| `mcp_service`        | MCP Server CRUD、调用 MCP Server `tools/list` 获取 Schema、更新 mcp_tools 表 |
| `memory_service`     | 记忆配置 CRUD、默认配置生成                                                    |
| `chat_service`       | 对话处理入口：接收消息 → 调用 assembler 组装 Agent → 执行对话 → 流式输出                   |
| `connection_manager` | WebSocket 连接池管理、消息广播、连接健康检查                                         |
| `session_manager`    | 会话创建/恢复/销毁、会话与 Agent 实例绑定                                           |

#### 4.2.3 运行时层（核心引擎）

| 模块                 | 职责                                                             |
|--------------------|----------------------------------------------------------------|
| `assembler`        | 从 DB 加载 Scene 配置 → 解析 Agent 定义 → 绑定工具/技能 → 组装为 LangChain Agent |
| `orchestrator`     | 主从调度：挂起 MainAgent → 创建 SubAgent 沙箱 → 等待完成 → 回注结果               |
| `skill_engine`     | Skill 索引构建、触发判断（手动/自动/定时）、工具模式注入、脚本模式沙箱执行                      |
| `context_assembly` | Prompt 模板渲染（`{variable}` 替换）、记忆召回注入、知识内容注入、Skill 索引注入          |
| `mcp_hub`          | MCP Client 连接池、Schema 缓存（从 DB 读取）、认证透传（注入 user token）          |

#### 4.2.4 记忆模块

| 模块                 | 职责                                            |
|--------------------|-----------------------------------------------|
| `session_memory`   | 会话级短期记忆：对话历史加载/持久化、上下文压缩（LLM 摘要）              |
| `long_term_memory` | PowerMem 封装：四路混合检索、记忆存储、按 user_id 隔离          |
| `user_profile`     | 用户画像加载（会话启动时）、画像更新（对话结束后）                     |
| `extraction`       | 异步记忆抽取：对话结束后提取关键信息 → PowerMem 存储，支持业务线自定义抽取规则 |

### 4.3 Agent 组装流程（assembler 核心逻辑）

```
输入: scene_id, user_id, session_id
    │
    ▼
┌─ Step 1: 加载配置 ─────────────────────────────────────────────┐
│                                                                 │
│  scene = db.query(Scene).get(scene_id)                         │
│  main_agent_def = db.query(Agent).get(scene.main_agent_id)     │
│  sub_agent_defs = [db.query(Agent).get(id)                     │
│                    for id in scene.sub_agent_ids]               │
│  tool_bindings = db.query(AgentToolBinding)                    │
│                    .filter(scene_id=scene_id)                  │
│  skill_bindings = db.query(AgentSkillBinding)                  │
│                    .filter(scene_id=scene_id)                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Step 2: 组装子 Agent ──────────────────────────────────────────┐
│                                                                  │
│  for sub_def in sub_agent_defs:                                  │
│      tools = resolve_tools(tool_bindings[sub_def.id])            │
│      skills = resolve_skills(skill_bindings[sub_def.id])         │
│      prompt = render_prompt(sub_def.system_prompt, context)      │
│      llm = create_llm(sub_def.model_config)                     │
│                                                                  │
│      sub_agent = create_langchain_agent(                         │
│          prompt=prompt, tools=tools, skills=skills, llm=llm     │
│      )                                                           │
│      # 包装为 Tool，供 MainAgent 调用                              │
│      sub_agent_tool = wrap_as_tool(sub_agent, sub_def)           │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Step 3: 组装主 Agent ──────────────────────────────────────────┐
│                                                                  │
│  main_tools = resolve_tools(tool_bindings[main_agent_def.id])    │
│  main_skills = resolve_skills(skill_bindings[main_agent_def.id]) │
│  main_tools += sub_agent_tools  # 子Agent作为工具                  │
│                                                                  │
│  main_prompt = render_prompt(main_agent_def.system_prompt, ctx)  │
│  main_llm = create_llm(main_agent_def.model_config)             │
│                                                                  │
│  main_agent = create_langchain_agent(                            │
│      prompt=main_prompt, tools=main_tools,                       │
│      skills=main_skills, llm=main_llm                           │
│  )                                                               │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Step 4: 上下文装配 ────────────────────────────────────────────┐
│                                                                  │
│  session_history = session_memory.load(session_id)               │
│  long_term_recall = long_term_memory.recall(user_id, query)      │
│  user_profile = user_profile.load(user_id)                       │
│  knowledge = load_knowledge(knowledge_bindings)                  │
│                                                                  │
│  context_assembly.inject(main_agent, {                           │
│      history: session_history,                                   │
│      memory: long_term_recall,                                   │
│      profile: user_profile,                                      │
│      knowledge: knowledge                                        │
│  })                                                              │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
    │
    ▼
输出: 完整组装的 LangChain Agent 实例（可执行）
```

## 五、数据流转

### 5.1 配置数据流（管理页面 → DB → Runtime）

描述管理员在页面操作配置后，数据如何流向运行时。

```
┌────────────┐     HTTP API      ┌──────────┐     SQL Query      ┌─────────────┐
│            │ ─── POST/PUT ───→ │          │ ←── SELECT ─────── │             │
│  管理页面    │                   │  数据库    │                    │  Agent      │
│  (前端 SPA) │ ←── 200 OK ───── │  (SQLite/ │ ──→ Config 对象 ──→ │  Runtime    │
│            │                   │   PgSQL)  │                    │  (assembler)│
└────────────┘                   └──────────┘                    └─────────────┘
      │                                │                               │
      │  1. 管理员创建/编辑 Agent        │  2. 数据持久化到 DB            │  3. 下次请求时
      │     填写表单 → 调用 API          │     agents/skills/scenes 表  │     assembler 从 DB
      │                                │                               │     读取最新配置
      │                                │                               │     动态组装 Agent
      └────────────────────────────────┴───────────────────────────────┘
                              热更新：无需重启
```

**关键流转节点：**

| 步骤 | 操作                    | 数据变化                                         |
|----|-----------------------|----------------------------------------------|
| 1  | 管理员在页面创建 Agent        | 前端表单 → JSON 请求体                              |
| 2  | API 层校验 + Service 层处理 | JSON → ORM 对象 → INSERT INTO agents           |
| 3  | 管理员编排 Scene           | 前端编排配置 → JSON（含 agent_ids, tool_bindings）    |
| 4  | Scene Service 处理      | JSON → scenes + scene_sub_agents + binding 表 |
| 5  | 用户发起对话请求              | chat_service 接收 scene_id                     |
| 6  | Assembler 从 DB 加载配置   | SELECT 各表 → 内存 Config 对象                     |
| 7  | 组装 LangChain Agent    | Config 对象 → LangChain Agent 实例               |

```
用户消息 (WebSocket)
    │
    │  {"action":"chat", "data":{"message":"...", "user_id":"...", "session_id":"..."}}
    │
    ▼

│                                                                        │
│  ws_router.py                                                          │
│  ├── 解析 action → 路由到 chat_service                                  │
│  ├── 提取 headers → set_request_context(token, user_id)                │

                                    │

┌─ 服务层 ──────────────────────────────────────────────────────────────┐
│                                                                        │
│  chat_service.py                                                       │
│  ├── session_manager.get_or_create(user_id, scene_id)                  │
│  ├── assembler.assemble(scene_id, user_id, session_id)                 │
│  │       ├── 从 DB 加载 Scene → Agent 定义 → 绑定关系                    │
│  │       ├── 组装 SubAgent → 包装为 Tool                                │
│  │       ├── 组装 MainAgent（tools = 内置 + MCP + SubAgent）            │

│                                                                        │
└───────────────────────────────────┬────────────────────────────────────┘
                                    │
                                    ▼
┌─ 运行时层 ─────────────────────────────────────────────────────────────┐
│                                                                        │
│  ReAct Loop (LangChain)                                                │
│  ├── LLM 推理 → 生成 token → stream 事件推送                            │
│  ├── 需要调用工具？                                                     │
│  │   ├── 内置工具 → 直接调用 → tool_start/tool_end 事件                  │
│  │   ├── MCP 工具 → mcp_hub 转发 → 认证透传 → tool_start/tool_end       │

│  └── user_profile.update_if_needed(user_id)      # 画像更新（如需）      │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘
```

### 5.3 记忆数据流

```
┌─ 会话启动 ────────────────────────────────────────────────────────────┐
│                                                                        │
│  user_id + session_id                                                  │
│      │                                                                 │
│      ├── ① session_memory.load(session_id)                             │
│      │       → DB messages 表 → 对话历史列表                             │
│      │       → 检查 token 总量 → 超阈值则触发 LLM 压缩                   │
│      │                                                                 │
│      ├── ② user_profile.load(user_id)                                  │
│      │       → PowerMem 查询 → 用户偏好/关注项目/习惯                    │
│      │                                                                 │
│      └── ③ long_term_memory.recall(user_id, context_query)             │
│              → PowerMem 四路混合检索 → 相关记忆条目                       │
│                                                                        │
│  → 注入 Agent 上下文 System Prompt                                      │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘

┌─ 对话进行中 ──────────────────────────────────────────────────────────┐
│                                                                        │
│  每轮对话：                                                             │
│      ├── 用户新消息 → 自动触发记忆检索（被动注入）                         │
│      └── Agent 可主动调用记忆搜索工具（主动检索）                          │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘

┌─ 对话结束后 ──────────────────────────────────────────────────────────┐
│                                                                        │
│  ④ session_memory.save(session_id)                                     │
│       → 完整对话历史 → INSERT INTO messages                              │
│                                                                        │
│  ⑤ extraction.extract_async(user_id, messages)                         │
│       → LLM 提取关键信息（按业务线配置的抽取规则）                         │
│       → PowerMem 存储长期记忆                                           │
│                                                                        │
│  ⑥ user_profile.update_if_needed(user_id)                              │
│       → 分析对话内容 → 更新用户偏好/关注点                                │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘
```

### 6.2 表结构详细定义

#### 6.2.1 agents 表

Agent 定义表，存储所有 Agent 的基础信息和配置。

| 字段                         | 类型           | 约束               | 说明                        |
|----------------------------|--------------|------------------|---------------------------|
| id                         | VARCHAR(36)  | PK               | UUID                      |
| name                       | VARCHAR(100) | NOT NULL, UNIQUE | Agent 名称                  |
| description                | TEXT         |                  | 描述                        |
| model                      | VARCHAR(50)  | NOT NULL         | 模型名称，如 `deepseek-chat`    |
| temperature                | FLOAT        | DEFAULT 0.7      | 温度参数                      |
| max_tokens                 | INT          | DEFAULT 4096     | 最大输出 token 数              |
| timeout                    | INT          | DEFAULT 120      | 超时时间（秒）                   |
| system_prompt              | TEXT         | NOT NULL         | 系统提示词（支持 `{variable}` 模板） |
| memory_session_compression | BOOLEAN      | DEFAULT TRUE     | 启用上下文压缩                   |
| memory_long_term           | BOOLEAN      | DEFAULT TRUE     | 启用长期记忆                    |
| memory_user_profile        | BOOLEAN      | DEFAULT TRUE     | 启用用户画像                    |
| status                     | VARCHAR(20)  | DEFAULT 'active' | 状态：active / inactive      |
| created_at                 | TIMESTAMP    | NOT NULL         | 创建时间                      |
| updated_at                 | TIMESTAMP    | NOT NULL         | 更新时间                      |

**索引：** `idx_agents_name` (name), `idx_agents_status` (status)

#### 6.2.2 skills 表

Skill 定义表，存储 Skill 的元数据和执行指令。

| 字段              | 类型           | 约束               | 说明                            |
|-----------------|--------------|------------------|-------------------------------|
| id              | VARCHAR(36)  | PK               | UUID                          |
| name            | VARCHAR(100) | NOT NULL, UNIQUE | Skill 名称                      |
| description     | TEXT         | NOT NULL         | 描述（注入 Agent 索引）               |
| mode            | VARCHAR(20)  | NOT NULL         | `tool` 或 `script`             |
| user_invocable  | BOOLEAN      | DEFAULT TRUE     | 用户可手动触发                       |

| schedule        | VARCHAR(50)  |                  | cron 表达式（定时触发）                |
| argument_hint   | VARCHAR(200) |                  | 参数提示                          |
| instruction     | TEXT         | NOT NULL         | 执行指令（MD body）                 |
| script_path     | VARCHAR(500) |                  | 脚本路径（mode=script 时）           |
| requires_tools  | JSON         |                  | 依赖的工具列表 `["tool_a","tool_b"]` |
| status          | VARCHAR(20)  | DEFAULT 'active' | 状态：active / inactive          |
| created_at      | TIMESTAMP    | NOT NULL         | 创建时间                          |

## 八、非功能性需求

### 8.1 并发

| 指标                | 要求       |
|-------------------|----------|
| 同时在线 WebSocket 连接 | ≥ 50     |
| 同一 Agent 并行会话     | ≥ 10     |
| 管理 API 并发         | ≥ 20 QPS |

- 每个用户会话独立 Agent 实例，共享只读配置
- WebSocket 连接池管理，支持心跳检测和断线重连
- DB 连接池（async，推荐 `asyncpg` 或 `aiosqlite`）

### 8.2 性能

| 指标            | 要求                                   |
|---------------|--------------------------------------|
| Agent 组装耗时    | ≤ 500ms（从 DB 读取配置到 Agent 实例就绪）       |
| 首 token 延迟    | ≤ 3s（从用户发送消息到第一个 token 返回）           |
| 工具调用开销        | 框架层 ≤ 50ms（不含 MCP Server / LLM 处理时间） |
| 管理 API 响应时间   | ≤ 200ms（常规 CRUD）                     |
| MCP Schema 刷新 | ≤ 10s（含网络请求）                         |

**配置缓存策略：**

- Agent / Scene 配置：每次请求从 DB 读取（保证热更新）
- MCP 工具 Schema：DB 持久化缓存，页面手动刷新更新
- 后续可增加内存缓存 + TTL 优化

### 8.3 可靠性

| 场景            | 处理方式                                     |
|---------------|------------------------------------------|
| LLM 调用超时      | 重试 1 次，仍失败则返回 error 事件，不阻塞会话             |
| MCP 工具调用失败    | 返回 tool_end 事件（含错误信息），Agent 可根据错误决定下一步   |
| WebSocket 断连  | 客户端重连后可通过 session_id 恢复会话状态              |
| SubAgent 执行超时 | Orchestrator 设置超时阈值，超时后返回超时错误给 MainAgent |
| DB 写入失败       | 事务回滚，返回 error 响应                         |
| 记忆抽取失败        | 异步任务，不影响对话响应，记录错误日志                      |

### 8.4 日志与追踪

| 维度      | 说明                                                |
|---------|---------------------------------------------------|
| traceId | 每个请求分配唯一 traceId，贯穿所有日志和事件                        |
| 结构化日志   | JSON 格式，包含 timestamp、level、traceId、module、message |
| 关键日志点   | 请求接收、Agent 组装完成、LLM 调用、工具调用、SubAgent 调度、记忆操作、请求完成 |
| 日志级别    | DEBUG（LLM 完整消息）、INFO（关键节点）、WARN（超时/重试）、ERROR（异常）  |

**日志格式示例：**

```json
{
  "timestamp": "2025-01-15T10:30:00.123Z",
  "level": "INFO",
  "traceId": "trace-abc-123",
  "module": "runtime.assembler",
  "message": "Agent assembled",
  "data": {
    "scene_id": "scene-001",
    "main_agent": "bi_master",
    "sub_agents": [
      "financial_manager",
      "project_manager"
    ],
    "tool_count": 8,
    "skill_count": 3,
    "duration_ms": 320
  }

```

### 8.5 安全

| 维度        | 措施                                                 |
|-----------|----------------------------------------------------|
| 认证        | 三方系统通过 WebSocket headers 传递 token，框架透传给 MCP Server |
| 授权        | MVP 阶段管理页面无独立鉴权（内网部署），后续迭代增加管理员认证                  |
| 输入校验      | API 层使用 Pydantic 模型校验所有输入参数                        |
| SQL 注入    | 使用 ORM（SQLAlchemy），禁止拼接 SQL                        |
| Prompt 注入 | MVP 阶段做基础防护（输入长度限制、敏感词过滤），后续增加多层防御                 |
| 脚本沙箱      | Skill 脚本模式在受限沙箱中执行，禁止文件系统访问和非白名单网络请求               |
| 数据隔离      | 记忆和会话按 user_id 隔离，不同用户数据互不可见                       |
