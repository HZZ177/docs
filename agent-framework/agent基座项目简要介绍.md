# Agent 基座项目介绍

## 一、项目定位

Agent 基座项目是面向各业务线提供统一 AI 对话能力的 **Agent 运行时平台**，目标是让业务线无需关心大模型调用细节，通过页面配置即可快速搭建和上线自己的 AI 助手场景。

---

## 二、整体架构 — 两个独立服务

平台拆分为两个独立部署的后端服务，职责清晰：

| 服务 | 职责 | 对接方 |
|------|------|--------|
| **管理服务** | Agent / Skill / Scene / MCP 工具等的页面配置管理 | 内部运营人员、业务线配置人员 |
| **基座服务** | 运行时对话引擎，驱动 Agent 执行，对外提供对话接入能力 | 各业务线系统 |

两个服务共用同一套数据库，基座服务**只读**管理侧配置，保证职责解耦。

---

## 三、业务线从配置到上线的完整流程

业务线从零开始接入，分为以下步骤：

### 第一步：配置 MCP 工具（对接业务系统能力）

业务线将自己的业务系统能力封装为 MCP Server 后，在管理后台登记 Server 地址。平台会自动拉取该 Server 下的所有可用工具（接口列表），供后续 Agent 绑定使用。

> 示例：ChatBI 业务线将「查询报表数据」「获取指标定义」等接口封装为 MCP Server，登记后平台即可感知这些工具。

---

### 第二步：配置 Skill（沉淀业务知识）

Skill 是 Agent 的「知识库」，在管理后台按需创建：

- **描述型 Skill**：写明某个业务场景的处理规则、注意事项
- **文档型 Skill**：上传文本类参考资料，Agent 对话时自动参考
- **手动触发型 Skill**：配置特定关键词触发，直接返回固定话术或执行指定工具链

Skill 创建后可绑定到任意 Agent，实现知识复用。

---

### 第三步：配置 Agent（定义 AI 角色）

在管理后台创建 Agent，填写：

- **名称与描述**：Agent 的角色定位
- **大模型选择**：选用哪个模型
- **系统提示词**：Agent 的行为指令，支持 {变量} 占位符（变量值由业务线在调用时动态传入，如公司名、园区 ID 等）
- **绑定工具**：从已登记的 MCP 工具中勾选该 Agent 可以调用哪些
- **绑定 Skill**：从已创建的 Skill 中勾选该 Agent 需要参考哪些知识

---

### 第四步：编排 Scene（组合多 Agent 场景）

一个 Scene 是实际对外提供服务的「场景单元」，在管理后台配置：

- 指定一个 **主 Agent**（负责与用户对话、任务分发）
- 可选挂载若干 **子 Agent**（各自负责垂直领域，由主 Agent 按需调度）
- 配置上下文窗口长度、记忆开关等参数

Scene 创建后会生成唯一的 scene_id，业务线系统凭此 ID 接入对话服务。

---

### 第五步：业务线系统接入，上线使用

业务线拿到 scene_id 后，将其嵌入自己的产品（App、Web、内部系统等），用户发起的每一次对话都会路由到该 Scene，由平台驱动 Agent 执行并返回结果。

平台全程负责：大模型调用、工具执行、多轮上下文管理、用户画像记忆注入。**业务线无需感知任何 AI 基础设施细节。**

---

│  │         ▼                                   │   │

## 五、核心功能 — MVP 范围与后续规划

### ✅ MVP 已包含

| 功能模块 | 说明 |
|----------|------|
| **完整对话链路** | 流式对话、多轮上下文、实时工具调用透出、取消与断连恢复 |
| **Agent 动态组装** | 从页面配置动态组装 Agent，热更新（改完配置下次对话即生效）|
| **MCP 工具集成** | 接入业务线 MCP Server，Agent 运行时自动调用，业务参数全链路透传 |
| **Skill 系统（基础版）** | Skill 描述和文本类参考文档自动注入 Agent 提示词；支持手动触发模式 |
| **用户画像记忆** | 会话结束后自动抽取用户偏好，下次对话注入，实现「记住用户」|
| **多轮会话持久化** | 会话历史持久化，断连后可恢复；滑动窗口控制上下文长度 |
| **调试页面** | 内置调试界面，支持 Agent 测试、会话回放、记忆查看 |
| **定时任务触发** | 支持定时自动唤起 Agent 执行任务，复用同一套对话链路 |

---

### ⏳ 暂未包含的功能及原因（后续规划）

#### 1. Skill 脚本执行（暂缓）

**现状**：MVP 的 Skill 仅支持描述文本和文档类内容注入。

**原因**：Skill 体系从 C 端产品（类 Claude Code）的本地文件系统环境迁移到了数据库存储 + 服务器环境。脚本执行涉及沙箱隔离、中间产物管理等工程复杂度较高，MVP 阶段优先保证核心对话链路稳定，该能力暂缓，后续作为独立模块规划。

---

#### 2. 记忆系统（仅用户画像，暂不支持场景记忆）

**现状**：MVP 只支持用户画像（长期用户偏好提取与注入），不支持场景级短期/长期记忆的语义召回与存储。

**原因**：当前接入的第一条业务线（ChatBI 一期）明确只提出了用户画像需求，MVP 按最小实现规划，避免过度建设。场景记忆的接口已预留，后续有需求时可快速接入。

---

#### 3. 会话压缩（暂缓，用滑动窗口替代）

**现状**：MVP 使用简单的滑动窗口机制控制对话上下文长度（截取最近 N 轮）。

**原因**：有效的会话压缩（语义摘要 + 关键信息保留）是相对独立的大功能，需要单独的模型调用和质量评估体系，MVP 时间内难以完整实现。滑动窗口在大多数短中期对话场景下已足够，后续待时间充裕再完善。
