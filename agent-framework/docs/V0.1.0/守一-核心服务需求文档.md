## 一、文档概述

### 1.1 定位

本文档是守一在 MVP 阶段的开发需求文档，覆盖 **后端服务框架搭建** 和 **核心服务（对话链路）** 两大部分。

核心服务是整个 Agent 基座的运行时核心——从 WebSocket 消息接入，到 Session 管理、Agent 动态组装、上下文构建、LLM
调用、流式事件解析、Skill 拦截注入，再到 Checkpointer 持久化，构成一条完整的对话处理管线。

### 1.2 负责范围总览

| 分类       | 模块                              | 说明                                                        |
|----------|---------------------------------|-----------------------------------------------------------|
| **服务搭建** | FastAPI 应用框架                    | 启动入口、路由注册、生命周期管理、数据库连接                                    |
| **服务搭建** | 通用组件                            | Logger（traceId）、Config、HttpClient、exceptions、utils        |
| **核心服务** | ws_router                       | WebSocket 长连接路由，消息解析与分发                                   |
| **核心服务** | connection_manager              | WebSocket 连接池管理                                           |
| **核心服务** | session_manager                 | 会话生命周期管理（创建/恢复/关闭/清理）                                     |
| **核心服务** | request_context                 | ContextVar 请求上下文（user_id / extra_params / token 透传）       |
| **核心服务** | assembler                       | Scene 配置 → 可执行 Agent 实例的组装器                               |
| **核心服务** | context_builder                 | Prompt 渲染 + 记忆注入 + Skill 索引注入                             |
| **核心服务** | agent_factory                   | LLM 实例创建与缓存、Agent 实例创建                                    |
| **核心服务** | checkpoint_saver                | SeekDBCheckpointSaver（LangGraph Checkpointer 的 SeekDB 适配） |
| **核心服务** | event_handler                   | astream_events() 事件流处理，转 WebSocket 推送消息                   |
| **核心服务** | skill_executor                  | Skill 手动触发拦截 + 工具模式 Prompt 注入 + 脚本模式沙箱执行                  |
| **核心服务** | chat_service                    | 对话处理入口，编排上述所有模块的调用顺序                                      |
| **后台功能** | debug_router                    | 调试页面静态资源 + 调试专用 API                                       |
| **数据库**  | sessions 表                      | 会话映射（user_id ↔ session_id ↔ scene_id）                     |
| **数据库**  | checkpoints / checkpoint_writes | Checkpointer 自动管理的对话历史表                                   |

### 1.3 不在范围内（由其他成员负责）

|-----------|----------------------------------------------------------------------------|
| 管理模块 CRUD | Agent / Skill / Scene / MCP Server 的增删改查 API 和 Service                     |
| 前端管理页面    | Vue SPA 的所有页面开发                                                            |
| 记忆模块      | PowerMem 封装（scene_memory / user_profile / powermem_client）                 |
| MCP Hub   | MCP Client 连接池、Schema 缓存、认证透传                                              |
| 定时任务      | scheduler_service / task_dispatcher / task_notifier                        |
| 内置工具      | time_provider / get_skill_content / execute_script / create_scheduled_task |

> **说明**：assembler 和 chat_service 中会 **调用** 记忆模块、MCP Hub、Skill 执行器等，但这些模块本身的内部实现不在守一的范围内。守一负责的是调用这些模块的
**编排逻辑**和**数据流转**。

## 二、服务搭建 — 后端框架与通用组件

### 2.1 整体架构说明

本项目包含**两个独立打包部署的后端服务** + **一个共享层**：

| 目录                | 说明                               | 部署方式                      |
|-------------------|----------------------------------|---------------------------|
| `admin_backend/`  | 后台管理服务（Agent/Skill/Scene 等 CRUD） | 独立打包，配合 admin_frontend 部署 |
| `agent_backend/`  | Agent 基座服务（对话链路、运行时引擎）           | 独立打包，对外提供 WebSocket 接入    |
| `common/`         | 共享基础设施（core + models）            | 被两个后端服务引用，不独立部署           |
| `admin_frontend/` | 后台管理前端（Vue SPA）                  | 独立打包，与 admin_backend 配合部署 |

**守一负责**：`agent_backend/` 全部 + `common/` 全部

## 三、核心服务 — WebSocket 协议定义

### 3.1 连接端点

> **📌 一期参考**：`backend/app/api/v1/chat.py` — 可复用 WebSocket 路由的整体结构（连接建立、消息循环、异常处理），需改写消息协议格式（新增 `action` 信封）和路由分发逻辑（新增 `create_session` / `close_session` 等 action）。

**文件**：`agent_backend/api/ws_router.py`

**端点**：`ws://host:port/agent-base/ws`

这是三方系统和调试页面的**统一对接入口**。MVP 不另设 HTTP Chat API，所有对话请求统一走 WebSocket，以支持流式响应、实时取消和多轮会话管理。

### 3.2 消息信封格式

所有 WebSocket 消息（客户端→服务端、服务端→客户端）统一使用 JSON 格式，信封结构如下：

**客户端 → 服务端：**

```json
{
  "action": "<action_name>",
  "data": {
    ...
  }
}
```

**服务端 → 客户端：**

```json
{
  "action": "<event_name>",
  "data": {
    "trace_id": "trace-abc-123",
    ...
  }
}
```

| 字段       | 类型     | 说明                                                                                       |
|----------|--------|------------------------------------------------------------------------------------------|
| `action` | string | 消息类型标识，区分请求类型或事件类型                                                                       |
| `data`   | object | 消息体，不同 action 的字段定义见下方表格                                                                 |
| `trace_id` | string | **仅服务端返回时携带**。由后端 logger 在请求进入系统时生成，与日志系统及 LLM 网关请求 header 中的 `X-Trace-Id` 保持一致，供业务方端到端追踪 |

### 3.3 协议总览

#### 客户端 → 服务端（请求消息）

| Action           | 说明                   | 必传字段                                           | 可选字段                    |
|------------------|----------------------|------------------------------------------------|-------------------------|
| `create_session` | 创建新会话                | `user_id`, `scene_id`                          | `extra_params`, `token` |
| `chat`           | 发送对话消息               | `user_id`, `session_id`, `scene_id`, `message` | `extra_params`, `token` |
| `scheduled_chat` | 定时任务触发对话             | `user_id`, `scene_id`, `message`, `task_id`    | `extra_params`          |
| `close_session`  | 主动关闭会话               | `session_id`                                   |                         |
| `cancel`         | 取消当前正在执行的请求          | `session_id`                                   |                         |
| `ping`           | 心跳保活                 | （无）                                            |                         |
| `get_status`     | 查询会话状态               | `session_id`                                   |                         |

#### 服务端 → 客户端（响应/推送事件）

> **约定**：所有服务端返回的事件 `data` 中均包含 `trace_id` 字段，与日志系统和 LLM 网关请求 header 中的 traceId 一致，便于业务方端到端追踪。

| Event (action)    | 说明         | data 字段（均含 `trace_id`）                          |
|-------------------|------------|------------------------------------------------|
| `session_created` | 会话创建成功     | `session_id`, `scene_id`, `user_id`            |
| `stream`          | 文字流式输出     | `content`, `session_id`                        |
| `tool_start`      | 工具调用开始     | `tool`, `params`                               |
| `tool_end`        | 工具调用结束     | `tool`, `result`, `duration_ms`                |
| `subagent_start`  | 子 Agent 启动 | `agent`, `task`                                |
| `subagent_end`    | 子 Agent 结束 | `agent`, `result_summary`                      |
| `completed`       | 单轮对话完成     | `session_id`                                   |
| `session_closed`  | 会话已关闭      | `session_id`                                   |
| `error`           | 错误         | `code`, `message`                              |
| `pong`            | 心跳回应       | （仅 `trace_id`）                                 |
| `task_result`     | 定时任务执行结果   | `task_id`, `session_id`, `result_summary`      |

### 3.4 请求消息字段规范

#### 3.4.1 公共字段定义

| 字段             | 类型     | 必传       | 说明                                                        |
|----------------|--------|----------|-----------------------------------------------------------|
| `user_id`      | string | 是        | 三方用户唯一 ID，用于会话隔离、记忆隔离、MCP 权限透传                            |
| `session_id`   | string | 视 action | 由 `create_session` 返回，`chat`/`cancel`/`close_session` 时必传 |
| `scene_id`     | string | 是        | 指定对话连接到哪个 Scene，后端 session 同步记录                           |
| `message`      | string | chat 时   | 用户消息文本                                                    |
| `token`        | string | 否        | 三方系统直传的认证令牌，存入上下文后工具调用时优先使用                               |
| `extra_params` | object | 否        | 业务线自定义透传参数，框架全链路透传，结构见 3.4.2                              |

#### 3.4.2 extra_params 结构定义

`extra_params` 是业务线传入的 JSON 对象，框架全链路透传给 MCP Server 和内部工具。除业务自定义字段外，框架约定以下子字段：

```json
{
  "extra_params": {
    "topgroup_id": "group-001",
    "park_id": "park-123",
    "prompt_render_params": {
      "company_name": "永策科技",
      "business_scope": "华东区"
    }
  }
}
```

| 字段                     | 类型     | 必传 | 说明                                                                                                                                                                                                                          |
|------------------------|--------|----|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `prompt_render_params` | object | 否  | **Prompt 动态渲染参数**。该对象中的 key-value 将用于替换 Agent 提示词模板中对应的 `{variable}` 占位符。如果提示词中存在 `{company_name}` 占位符但 `prompt_render_params` 中未传入该字段，`render_prompt_template()` 将抛 `KeyError`。因此三方系统需确保传入的参数与 Agent Prompt 模板中定义的动态变量一一对应 |
| 其他字段                   | any    | 否  | 业务线自定义参数（如 `topgroup_id`、`park_id` 等），框架透传不解析                                                                                                                                                                               |

**与普通 `chat` 的差异：**

| 对比项         | `chat`                   | `scheduled_chat`                 |
|-------------|--------------------------|----------------------------------|
| 触发来源        | 用户实时输入                   | APScheduler 定时任务模块              |
| `token`     | 可选传入                     | 无（定时任务场景无实时 token）              |
| `session_id` | 必传（已有会话）                 | 不传（后端自动创建 session，标记为定时任务会话）    |
| `task_id`   | 无                        | 必传，关联 `scheduled_tasks` 表的任务 ID |
| 结果出口        | WebSocket 流式推送给客户端       | `task_result` 事件推送（日志由定时任务模块记录） |
| session 标记  | `is_debug=false`         | `is_scheduled=true`             |
| 完成事件        | `completed`              | `task_result`（携带 task_id）       |

后端收到 `scheduled_chat` 后：

1. 自动创建 session（`is_scheduled=true`，`scheduled_task_id=task_id`）
2. `set_request_context` 时 token 为 None
3. 复用标准 `assembler → chat_service` 链路执行
4. 执行结果通过 `task_result` 事件推送（执行日志由定时任务模块自行写入 `task_execution_logs` 表）

### 3.5 会话生命周期

#### 3.5.1 新建对话

```
1. 客户端建立 WebSocket 连接
2. 客户端发送 create_session（user_id + scene_id + extra_params）
3. 服务端创建 session 记录 → 返回 session_created（含 session_id）
4. 客户端使用 session_id 发送 chat 消息（可多轮）
5. 对话结束 → 客户端发送 close_session
6. 服务端标记 session 关闭 → 异步触发记忆抽取
```

#### 3.5.2 恢复对话

```
1. 客户端建立 WebSocket 连接
2. 客户端发送 chat（携带已有 session_id + scene_id）
3. 服务端通过 session_id 恢复历史对话（Checkpointer 自动加载）
4. 继续多轮对话
```

#### 3.5.3 会话结束条件

| 条件   | 触发方式                     | 说明                                |
|------|--------------------------|-----------------------------------|
| 主动关闭 | 客户端发送 `close_session`    | 正常结束流程                            |
| 保活超时 | **session 池统一保活时间 5 分钟** | session 最后活跃时间超过 5min 无新消息，后端自动关闭 |
| 断连超时 | WebSocket 断连后宽限期 30 秒    | 30 秒内未重连则自动关闭                     |

### 3.6 路由分发逻辑

```
WebSocket 消息到达
    │
    ├── action == "create_session"
    │   → session_manager.create_session(user_id, scene_id)
    │   → 返回 session_created 事件
    │
    ├── action == "chat"
    │   → 后端生成 traceId，写入 request_context
    │   → set_request_context(user_id, extra_params, token, scene_id, trace_id)
    │   → chat_service.handle_chat(session_id, scene_id, message)
    │
    ├── action == "scheduled_chat"
    │   → 后端生成 traceId，写入 request_context
    │   → set_request_context(user_id, extra_params, token=None, scene_id, trace_id)
    │   → session_manager.create_session(user_id, scene_id, is_scheduled=true, task_id)
    │   → chat_service.handle_chat(session_id, scene_id, message)
    │   → 返回 task_result 事件（执行日志由定时任务模块自行记录）
    │
    ├── action == "close_session"
    │   → session_manager.close_session(session_id)
    │   → 触发记忆抽取（调用记忆模块）
    │   → 返回 session_closed 事件
    │
    ├── action == "cancel"
    │   → session_manager.cancel(session_id)
    │
    ├── action == "ping"
    │   → 返回 pong
    │
    └── action == "get_status"
```


#### 4.2.4 保活超时机制

session_manager 维护一个 session 池，统一管理所有 active session 的保活时间。

```
session 池保活检查（定时轮询，如每 60s 一次）
    │
    ├── 遍历所有 status='active' 的 session
    ├── 检查 last_active_at 距当前时间
    │
    ├── 超过 5 分钟无新消息
    │   → 标记 session status='closed'
    │   → 异步触发记忆抽取（传递 session_id，记忆模块自行获取 messages）
    │   → 向关联的 WebSocket 连接发送 session_closed 事件
    │
    └── 未超时 → 跳过
```

| 配置项  | 值     | 说明                        |
|------|-------|---------------------------|
| 保活超时 | 5 min | session 最后活跃时间超过 5 分钟自动关闭 |
| 检查周期 | 60s   | 保活检查定时轮询间隔                |

**last_active_at 更新时机**：每次收到该 session 的 `chat` 消息时，更新 sessions 表的 `updated_at` 字段。

**保活超时检查时序图：**

```mermaid
sequenceDiagram
    participant TIMER as 定时轮询（每60s）
    participant SM as session_manager
    participant DB as SeekDB（sessions 表）
    participant MEM_S as 记忆模块（scene_memory）
    participant MEM_U as 记忆模块（user_profile）
    participant CM as connection_manager
    participant C as 客户端

    loop 每 60 秒检查一次
        TIMER->>SM: 触发保活检查
        SM->>DB: SELECT * FROM sessions WHERE status='active'
        DB-->>SM: 所有 active session 列表

        loop 遍历每个 active session
            SM->>SM: 计算 now() - updated_at
            alt 超过 5 分钟无新消息
                SM->>DB: UPDATE sessions SET status='closed'
                par 异步记忆抽取（记忆模块自行从 Checkpointer 获取 messages）
                    SM--)MEM_S: scene_memory.store(user_id, scene_id, session_id)
                and
                    SM--)MEM_U: user_profile.update(user_id, session_id)
                end
                SM->>CM: send(ws, session_closed)
                CM-->>C: {"action":"session_closed","data":{session_id, trace_id}}
            else 未超时
                Note over SM: 跳过，继续检查下一个
            end
        end
    end
```

#### 4.2.5 断连兜底

```
WebSocket 断连（网络异常/刷新页面）
    │
    ├── 后端检测到连接断开
    ├── 如果关联的 session 仍为 active 且有对话内容
    ├── 等待宽限期（30 秒）
    │   ├── 宽限期内重连 → session 继续，不触发抽取
    │   └── 宽限期内未重连 → 标记 session closed → 触发记忆抽取（传递 session_id）
    └── cancel_event 管理（取消正在执行的 Agent）
```

**断连兜底时序图：**

```mermaid

        alt 30秒内客户端重连
            C->>WS: 新 WebSocket 连接建立
            WS->>CM: register(new_ws)
            CM->>SM: 绑定到原 session
            SM->>SM: 取消宽限期计时器
            Note over SM: session 继续，不触发记忆抽取
        else 30秒内未重连
            SM->>DB: UPDATE sessions SET status='closed'
            par 异步记忆抽取（记忆模块自行从 Checkpointer 获取 messages）
                SM--)MEM_S: scene_memory.store(user_id, scene_id, session_id)
            and
                SM--)MEM_U: user_profile.update(user_id, session_id)
            end
            SM->>SM: 设置 cancel_event（取消正在执行的 Agent）
        end
    else session 无对话内容
        Note over SM: 直接清理，无需记忆抽取
    end
```

#### 4.2.5 cancel 操作

```
cancel(session_id)
    │
    ├── 设置 cancel_event → Agent 执行循环检测到后停止
    └── 返回 error 事件（action=cancelled）
```

## 五、核心服务 — 请求上下文与参数透传

### 5.1 request_context

> **📌 一期参考**：`backend/app/core/request_context.py` — 可直接复用 ContextVar 的定义和 `get_token()` / `get_user_id()` / `get_extra_params()` 等读取函数的实现，需新增 `scene_id`、`session_id`、`trace_id` 等上下文变量。

**文件**：`common/core/request_context.py`

基于 Python `contextvars.ContextVar` 实现，在当前异步任务链中透传请求级别的参数。

| 上下文变量          | 类型       | 来源           | 说明                                                  |
|----------------|----------|--------------|-----------------------------------------------------|
| `user_id`      | str      | 三方系统（必传）     | 三方用户唯一标识，全链路使用                                      |
| `extra_params` | dict     | 三方系统（可选）     | 业务线自定义参数（如 topgroup_id、park_id）                     |
| `token`        | str/None | 三方系统（可选）     | chat 场景下如传入则透传；定时任务场景下为 None                        |
| `scene_id`     | str      | ws_router 设置 | 当前对话的 Scene ID                                      |
| `session_id`   | str      | ws_router 设置 | 当前对话的 Session ID                                    |
| `trace_id`     | str      | **后端生成**     | 后端在收到 chat 请求时自动生成 UUID，不由前端传入。同时注入 LLM 网关请求 header |

**设置时机**：ws_router 在收到 chat 消息后，调用 `set_request_context()` 写入上下文变量。

**读取方式**：

```python
from common.core.request_context import get_token, get_user_id, get_extra_params

token = get_token()  # 可能为 None（定时任务场景）
user_id = get_user_id()  # 始终有值
extra_params = get_extra_params()  # 可能为空 dict
```

### 5.2 参数透传设计

框架的认证模型是 **"user_id 必传 + extra_params 透传 + token 可选兼容"**。

核心原因：定时任务被唤起时没有实时 chat 场景，不存在有效 token，因此 token 获取下沉到 MCP 层。

**约束**：`user_id` + `extra_params` 的组合必须能唯一定位业务场景下的确定用户。

#### 场景一：实时 Chat（有 token）

```
三方系统 → WebSocket chat 消息
    │  data: { user_id, extra_params, token, message }
    ▼
ws_router → set_request_context(user_id, extra_params, token)
    ▼
Agent 执行 → 工具调用
    ├── get_user_id()       → "zhangsan_001"
    ├── get_extra_params()  → {"topgroup_id": "group-001"}
    └── get_token()         → "Bearer xxx..."  ← 有值
    ▼
MCP Hub 透传给 MCP Server
    ├── user_id + extra_params（始终透传）
    └── token（有值则透传，MCP 可直接鉴权）
```

### 5.3 注意事项

- **ContextVar 不能跨线程**：`get_token()` / `get_user_id()` 只在当前异步任务链中有效。如果 `threading.Thread` 或
  `run_in_executor` 启动新线程，上下文会丢失返回 `None`。需要在新线程入口处重新 `set_request_context()`。
- **token 不持久化**：token 仅存在于 `request_context` 内存中，不写入 DB。定时任务的 `creator_extra_params` 会持久化到
  `scheduled_tasks` 表。

## 六、核心服务 — Agent 组装器（assembler）

### 6.1 定位

> **📌 一期参考**：
> - `backend/app/agent/registry.py` — 可参考 Agent 自动发现和注册机制的实现思路，MVP 改为从 DB 动态加载，但注册表的设计模式可复用。
> - `backend/app/agent/base.py` — 可参考 `BaseSubAgent` / `BaseMainAgent` 的基类设计（`get_tools()`、`system_prompt`、`context_provider` 结构），MVP 改为 DB 配置驱动但组装流程相似。
> - `backend/app/agent/context/base.py` — 可参考 `BaseContextProvider` 的 `safe_fetch()` 模式，MVP 演进为 `context_builder` 通用构建器。

**文件**：`agent_backend/engine/assembler.py`

assembler 是 MVP 的核心模块，替代一期的 `AgentRegistry.get_agent()` + `AgentFactory` 组合。一期通过 Python 代码硬编码
Agent 定义，MVP 改为从 DB 加载 Scene 配置后动态组装。

**设计原则**：assembler 每次请求都从 DB 读取最新配置，不做内存缓存。这保证了"DB 修改后下次请求自动生效，无需重启"的热更新能力。

### 6.2 输入与输出

| 输入             | 说明                                     |
|----------------|----------------------------------------|
| `scene_id`     | 目标 Scene 的 ID                          |
| `user_id`      | 三方用户 ID                                |
| `session_id`   | 当前会话 ID（同时作为 Checkpointer 的 thread_id） |
| `extra_params` | 业务线透传参数                                |
| `token`        | 可选的认证令牌                                |
| `message`      | 用户消息                                   |

| 输出              | 说明                                            |
|-----------------|-----------------------------------------------|
| `main_agent` 实例 | 可执行的 DeepAgent 实例，已注入 SubAgent、工具、Prompt、用户画像 |
| `session_id`    | 作为 Checkpointer 的 thread_id                   |

### 6.3 组装流程

```
输入: scene_id, user_id, session_id, extra_params, token, message
    │
    ▼
┌─ Step 1: 加载 Scene 配置 ──────────────────────────────────┐
│                                                              │
│  scene       = db.query(Scene).get(scene_id)                │
│  api_key     = scene.llm_api_key  # Scene 级别的 LLM 密钥   │
│  main_def    = db.query(Agent).get(scene.main_agent_id)     │
│  sub_defs    = db.query(Agent).filter(id in scene.sub_agents)│
│  bindings    = db.query(AgentToolBinding + AgentSkillBinding)│
│                  .filter(scene_id)                           │
│  memory_cfg  = db.query(MemoryConfig).get(scene_id)         │
│                                                              │
└──────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Step 2: 创建 SubAgent 实例 ─────────────────────────────────┐
│                                                               │
│  for sub_def in sub_defs:                                    │
│                                                               │
│    # 2a. 解析工具绑定                                          │
│    tools = []                                                 │
│    tools += builtin_tools(bindings)       # 内置工具直接引用   │
│    tools += mcp_hub.get_tools(bindings)   # 从 MCP Hub 获取   │
│                                                               │
│    # 2b. 构建上下文，渲染 Prompt                               │
│    prompt_render_params = extra_params.get("prompt_render_params", {})│
│    context = context_builder.build(sub_def, prompt_render_params)│
│    prompt = render_prompt(sub_def.system_prompt, context)     │
│                                                               │
│    # 2c. 注入 Skill 内容（通过 skill_assembler 组装）          │
│    skill_text = skill_assembler.build(bindings)               │
│    prompt += skill_text                                       │
│                                                               │
│    # 2d. 创建 LangChain Agent                                │
│    #     model 从 Agent 配置获取，api_key 从 Scene 获取        │
│    llm = agent_factory.get_or_create_llm(                    │
│        model=sub_def.model, api_key=api_key                  │
│    )                                                          │
│    sub_agent = agent_factory.create_agent(prompt, tools, llm) │
│                                                               │
│    # 2e. 包装为 CompiledSubAgent（供 DeepAgents 框架使用）     │
│    compiled = CompiledSubAgent(                               │
│        name=sub_def.agent_type,                               │
│        description=sub_def.description,                       │
│        runnable=sub_agent                                     │
│    )                                                          │
│                                                               │
└───────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Step 3: 创建 MainAgent 实例 ────────────────────────────────┐
│                                                               │
│  # 3a. MainAgent 自己的工具 + Skill 内容                      │
│  main_tools = resolve_tools(bindings[main_def.id])            │
│  main_prompt = render_prompt(main_def.system_prompt, context) │
│  main_prompt += skill_assembler.build(bindings)               │
│                                                               │
│  # 3b. 调用 DeepAgents 创建，传入 SubAgent 列表               │
│  main_llm = agent_factory.get_or_create_llm(                 │
│      model=main_def.model, api_key=api_key                   │
│  )                                                            │
│  main_agent = create_deep_agent(                              │
│      model=main_llm,                                          │
│      tools=main_tools,                                        │
│      subagents=compiled_subagents,                            │
│      system_prompt=main_prompt,                               │
│      checkpointer=checkpointer                                │
│  )                                                            │
│                                                               │
└───────────────────────────────────────────────────────────────┘
    │
    ▼
┌─ Step 4: 条件注入用户画像 ───────────────────────────────────┐
│                                                               │
│  # 从 memory_configs 表读取该 Scene 的记忆配置                 │
│  if memory_cfg and memory_cfg.user_profile_enabled:           │
│      # 从记忆模块加载该用户的画像                               │
│      profile = user_profile.load(user_id)                     │
│      if profile:                                               │
│          prompt = context_builder.inject_profile(prompt, profile)│
│                                                               │
│  # 注意：当前阶段不启用场景记忆，无语义召回和时间近邻召回       │
│  # 对话上下文由 Checkpointer 自动管理                          │
│  # → 传入 session_id 作为 thread_id                           │
│                                                               │
└───────────────────────────────────────────────────────────────┘
    │
    ▼
输出: (main_agent 实例, session_id)
    → 交给 chat_service 执行
```

### 6.4 关键设计决策

| 决策                  | 说明                                                  |
|---------------------|-----------------------------------------------------|
| **请求级实例**           | Agent 实例是请求级别的（每次对话请求创建新实例），不是应用级别的单例               |
| **LLM 缓存**          | 只有 LLM 实例做缓存复用（创建开销大且配置变化少），Agent 实例不缓存             |
| **不做 orchestrator** | SubAgent 调度完全由 DeepAgents 框架通过 `task` 工具自动管理，不另建调度层 |
| **每次读 DB**          | 不做配置内存缓存，保证热更新。后续可增加 TTL 缓存优化                       |

### 6.5 热更新行为

| 资源类型                  | 修改方式            | 生效时机       | 实现机制                             |
|-----------------------|-----------------|------------|----------------------------------|
| Agent 定义（Prompt、描述等）  | 管理页面编辑          | 下次对话请求     | assembler 从 DB 读取                |
| Agent 的 LLM 配置        | 管理页面编辑          | 下次对话请求     | agent_factory 检测缓存 key 变化，创建新实例  |
| Scene 编排（增减 SubAgent） | 管理页面编辑          | 下次对话请求     | assembler 从 DB 读取                |
| 工具绑定（增减工具）            | 管理页面编辑          | 下次对话请求     | assembler 从 DB 读取                |
| Skill 内容              | 管理页面编辑          | 下次对话请求     | assembler 重建 Skill 索引            |
| MCP 工具列表              | 页面点击"刷新 Schema" | 刷新后的下次对话请求 | mcp_service 更新 DB → assembler 读取 |

### 6.6 与一期的对比

| 步骤             | 一期实现                                        | MVP 实现                                    |
|----------------|---------------------------------------------|-------------------------------------------|

## 七、核心服务 — 上下文组装（context_builder）

### 7.1 定位

> **📌 一期参考**：`backend/app/agent/base.py` 中的 `render_prompt_template()` 方法 — 可复用 `str.format()` 模板渲染和花括号转义的处理逻辑，MVP 将其抽离为独立的 `context_builder` 模块，并新增 L1~L4 分层结构。

**文件**：`agent_backend/engine/context_builder.py`

context_builder 是一期 `render_prompt_template` 的演进版本，负责将多个来源的数据按固定分层结构组装为最终发送给 LLM 的完整
system_prompt。

### 7.2 职责

1. **L1 内置基础提示词**：追加框架固定的基础内容（防注入、时间戳等）
2. **L2 Agent 提示词渲染**：将 Agent 的 system_prompt 中的 `{variable}` 占位符替换为实际值
3. **L3 用户画像注入**：条件注入（按 memory_config 配置决定是否启用）
4. **L4 Skill 内容注入**：将绑定的 Skill 组装后的文本追加到 Prompt

### 7.3 模板变量

模板变量分两类来源：

**框架内置变量（自动填充）：**

| 变量               | 说明       | 来源           |
|------------------|----------|--------------|
| `{name}`         | Agent 名称 | Agent 定义（DB） |
| `{current_time}` | 当前时间戳    | 运行时生成        |

**业务动态变量（由三方传入）：**

| 变量         | 说明     | 来源                                               |
|------------|--------|--------------------------------------------------|
| `{任意业务变量}` | 业务线自定义 | `extra_params.prompt_render_params` 中的 key-value |

**注意**：

- Prompt 模板使用 `str.format()` 语法
- 如果 Prompt 文本中需要出现花括号（如 JSON 示例），必须用 `{{` `}}` 转义
- 如果 Prompt 中存在 `{xxx}` 占位符但 `prompt_render_params` 中未传入对应字段，`render_prompt_template()` 将抛 `KeyError`

## 八、核心服务 — LLM 对接与 Agent 创建

### 8.1 LLM 配置获取策略

基座项目不走配置文件和环境变量方式管理 LLM 参数，全部从 DB 获取：

| 参数              | 存储位置                 | 说明                                    |
|-----------------|----------------------|---------------------------------------|
| **网关地址**        | 全局配置（唯一固定值）          | `LLM_GATEWAY_ENDPOINT`，所有请求共用         |
| **api_key**     | `scenes.llm_api_key` | **以 Scene 为单位**，同一 Scene 下所有 Agent 共用 |
| **model**       | `agents.model`       | **以 Agent 为单位**，每个 Agent 可配置不同模型      |
| **temperature** | `agents.temperature` | Agent 级别配置，默认 0.7                     |
| **max_tokens**  | `agents.max_tokens`  | Agent 级别配置，默认 4096                    |
| **timeout**     | `agents.timeout`     | Agent 级别配置，默认 120s                    |

**LLM 创建流程**：

```
assembler 组装 Agent 时
    │
    ├── 从 Scene 获取 api_key
    │   api_key = scene.llm_api_key
    │
    ├── 从 Agent 定义获取 model 和其他参数
    │   model = agent_def.model
    │   temperature = agent_def.temperature
    │   max_tokens = agent_def.max_tokens
    │
    └── 调用 agent_factory.get_or_create_llm(
            api_key=api_key,
            model=model,
            temperature=temperature,
            max_tokens=max_tokens
        )
```

### 8.3 SubAgent 调度机制

SubAgent 调度由 DeepAgents 框架自动管理，不需要自定义 orchestrator：

```
MainAgent 的 SubAgent 列表
    ↓
create_deep_agent() 传入 SubAgent 列表
    ↓
DeepAgents 框架自动将 SubAgent 注册为名为 task 的内置工具
    ↓
运行时：
    LLM 根据用户问题 + 各 SubAgent 的 description
    自主决定调用哪个 task
    框架自动管理：SubAgent 执行、checkpoint_ns 上下文隔离、结果回注
```

**要点**：MainAgent 调度 SubAgent 的本质是 LLM 调用 `task` 工具，和调用普通 Tool 没有本质区别。我们不需要在框架之上再叠加自定义调度器。

| `checkpoint_writes` | 状态写入记录 |

> 框架不再维护独立的 messages 表。对话消息的存储/加载完全由 Checkpointer 自动管理。

### 9.6 上下文长度限制（滑动窗口）

> **📌 一期方案**：放弃 LLM 摘要压缩和 session 窗口迁移，采用简单的滑动窗口策略限制上下文长度。

**实现方式**：通过 Scene 配置的 `history_window_length` 字段，在加载历史消息时只取最近 N 轮对话。

#### 9.6.1 配置字段

| 字段                     | 存储位置     | 说明                                     |
|------------------------|----------|----------------------------------------|
| `history_window_length` | scenes 表 | 历史消息窗口长度（轮数），如 10 表示保留最近 10 轮对话 |

**说明**：
- 一轮对话 = 1 条 user 消息 + 1 条 assistant 消息
- `history_window_length = 0` 表示不限制（加载全部历史）
- 推荐值：10-20 轮

#### 9.6.2 实现逻辑

```
每轮对话开始时
    │
    ├── 从 Checkpointer 读取完整 messages 列表
    ├── 如果 history_window_length > 0
    │   └── 只保留最近 N 轮消息（2N 条消息）
    └── 将截取后的 messages 传给 Agent
```

**注意**：
- Checkpointer 仍然保存完整历史，滑动窗口只影响传给 LLM 的消息
- 这种方式简单可靠，不涉及 session 迁移和状态变更

### 10.3 事件处理流程

```
Agent 执行（astream_events() 返回事件流）
    │
    ├── stream 事件 → 文字 token
    │   → 构造 {"action": "stream", "data": {"content": "...", "session_id": "...", "trace_id": "..."}}
    │   → connection_manager.send(ws, message)
    │
    ├── tool_call 事件 → 工具调用开始
    │   → 构造 {"action": "tool_start", "data": {"tool": "...", "params": {...}, "trace_id": "..."}}
    │
    ├── tool_result 事件 → 工具调用结束
    │   → 构造 {"action": "tool_end", "data": {"tool": "...", "result": "...", "duration_ms": N, "trace_id": "..."}}
    │
    ├── task 工具事件 → SubAgent 调度（由 DeepAgents 框架触发）
    │   ├── 通过 event_name == "task" 识别
    │   ├── 通过 checkpoint_ns 区分不同 SubAgent
    │   ├── SubAgent 启动 → {"action": "subagent_start", "data": {"agent": "...", "task": "...", "trace_id": "..."}}
    │   ├── SubAgent 内部的 tool_start / tool_end / stream 事件正常处理
    │   └── SubAgent 结束 → {"action": "subagent_end", "data": {"agent": "...", "result_summary": "...", "trace_id": "..."}}
    │
    └── 执行完成
        → {"action": "completed", "data": {"session_id": "...", "trace_id": "..."}}
```

```mermaid
    CS->>AGENT: astream_events(message)

    loop 遍历事件流
        AGENT-->>EH: event（stream/tool_call/tool_result/task/完成）

        alt event = stream（文字 token）
            EH->>CM: send(ws, {action:"stream", data:{content, session_id, trace_id}})
            CM-->>C: 流式文字片段

        else event = tool_call（工具调用开始）
            EH->>CM: send(ws, {action:"tool_start", data:{tool, params, trace_id}})
            CM-->>C: 工具调用开始通知

        else event = tool_result（工具调用结束）
            EH->>EH: 计算 duration_ms
            EH->>CM: send(ws, {action:"tool_end", data:{tool, result, duration_ms, trace_id}})
            CM-->>C: 工具调用结果

        else event = task（SubAgent 事件）
            EH->>EH: 通过 checkpoint_ns 识别 SubAgent
            alt SubAgent 启动
                EH->>CM: send(ws, {action:"subagent_start", data:{agent, task, trace_id}})
                CM-->>C: SubAgent 启动通知
            else SubAgent 内部事件
                Note over EH: 正常处理内部 stream/tool 事件
                EH->>CM: send(ws, 对应事件（均含 trace_id）)
                CM-->>C: 实时推送
            else SubAgent 结束
                EH->>CM: send(ws, {action:"subagent_end", data:{agent, result_summary, trace_id}})
                CM-->>C: SubAgent 结束通知
            end

        else 检测到 cancel_event
            EH->>SM: 确认取消
            EH->>CM: send(ws, {action:"error", data:{code:"cancelled", trace_id}})
            CM-->>C: 取消通知
            Note over EH: 中断事件循环
        end
    end

    EH->>CM: send(ws, {action:"completed", data:{session_id, trace_id}})
    CM-->>C: 对话完成通知
```

### 10.4 SubAgent 事件识别

SubAgent 事件的识别方式：

- event_name == "task" 表示 SubAgent 相关事件
- 通过 checkpoint_ns 区分不同 SubAgent 的事件流
- 框架自动为每个 SubAgent 创建独立的 checkpoint_ns

### 10.5 错误事件

```
Agent 执行中异常
    │
    ├── LLM 调用超时 → 重试 1 次，仍失败 → {"action": "error", ...}
    ├── MCP 工具调用失败 → {"action": "tool_end", "data": {"result": "错误信息"}}
    ├── SubAgent 执行超时 → DeepAgents max_iterations 限制 → 超时提示
    └── 其他异常 → {"action": "error", "data": {"code": N, "message": "..."}}
```

## 十一、核心服务 — Skill 组装与拦截

### 11.1 定位

> **📌 一期无对应实现**，需全新开发。Skill 体系（索引注入、内容获取、手动触发拦截）是 MVP 新增功能。

守一负责两部分：

1. **skill_assembler**（`agent_backend/engine/skill_assembler.py`）：Skill 文本内容组装，将 Skill 的不同字段拼接为注入
   Prompt 的文本
2. **chat_service 中的拦截逻辑**：用户手动触发（"/" 前缀）的拦截处理

> **重要**：当前设计中不区分"工具模式"和"脚本模式"。脚本的执行通过一个**内置工具 `execute_script`** 来实现，skill_assembler
> 只管文本内容组装。

### 11.2 Skill 内容字段（由 Skill 模块定义）

Skill 表中存储的不同字段，skill_assembler 根据需要分别获取并组装：

| 字段               | 说明       | 组装用途                     |
|------------------|----------|--------------------------|
| `name`           | Skill 名称 | 索引标题                     |
| `description`    | Skill 描述 | 索引摘要，供 LLM 判断是否调用        |
| `instruction`    | 执行指令正文   | LLM 触发时获取的完整指令内容         |
| `script_content` | 脚本代码（如有） | 通过 `execute_script` 工具执行 |
| `template`       | 输出模板（如有） | 结果格式化模板                  |

### 11.3 skill_assembler — 文本组装

**文件**：`agent_backend/engine/skill_assembler.py`

skill_assembler 负责将绑定的 Skill 内容组装为文本，注入到 Prompt 的 L4 层。

#### 11.3.1 build_index() — 构建索引摘要

```python
def build_index(skill_bindings) -> str:
    """
    构建 Skill 索引，仅包含 name + description，
    供 LLM 了解有哪些可用技能。
    """
    skills = load_skills_from_bindings(skill_bindings)
    lines = ["## 可用技能"]
    for skill in skills:
        lines.append(f"- [{skill.name}] {skill.description}")
    return "\n".join(lines)
```

输出示例：

```
## 可用技能
- [项目健康诊断] 对指定项目进行全方位健康诊断分析
- [异常项目扫描] 扫描全部项目识别异常指标
- [月度报告生成] 生成月度项目运营分析报告
```

#### 11.3.2 get_full_content() — 获取完整内容

当 LLM 决定执行某个 Skill 时，调用内置工具 `get_skill_content(skill_name)` 获取完整内容：

```python
def get_full_content(skill_name: str) -> str:
    """
    返回 Skill 的完整执行内容，包含 instruction、template 等。
    """
    skill = db.query(Skill).filter(name=skill_name).first()
    content = f"## 技能：{skill.name}\n\n"
    content += f"### 执行指令\n{skill.instruction}\n\n"
    if skill.template:
        content += f"### 输出模板\n{skill.template}\n"
    if skill.script_content:
        content += f"### 脚本代码\n可通过 execute_script 工具执行\n"
    return content
```

### 11.5 脚本执行机制

脚本执行通过**内置工具 `execute_script`** 实现（由其他模块开发），skill_assembler 不直接执行脚本。

当 Skill 包含 `script_content` 时：

1. skill_assembler 在 `get_full_content()` 中提示"可通过 execute_script 工具执行"
2. LLM 根据指令决定是否调用 `execute_script(skill_name, arguments)`
3. `execute_script` 工具从 DB 读取 `script_content`，在沙箱中执行，返回结果

```mermaid
    par Layer 1：场景记忆抽取（异步，记忆模块自行获取 messages）
        SM--)MEM_S: scene_memory.store(user_id, scene_id, session_id)
        MEM_S->>CP: 读取 messages（通过 session_id = thread_id）
        CP-->>MEM_S: 完整对话消息
        Note over MEM_S: 异步执行，不阻塞<br/>失败仅记录错误日志
    and Layer 2：用户画像更新（异步，记忆模块自行获取 messages）
        SM--)MEM_U: user_profile.update(user_id, session_id)
        MEM_U->>CP: 读取 messages（通过 session_id = thread_id）
        CP-->>MEM_U: 完整对话消息
        Note over MEM_U: 异步执行，不阻塞<br/>失败仅记录错误日志
    end

    SM->>CM: send(ws, session_closed)
    CM-->>C: {"action":"session_closed","data":{session_id, trace_id}}
```

### 12.4 取消操作

```
cancel(session_id)
    │
    ├── session_manager 设置 cancel_event
    ├── Agent 执行循环中检测到 cancel_event → 停止推理
    └── 发送 error 事件（action=cancelled）
```

### 12.5 错误处理

| 场景            | 处理方式                                                  |
|---------------|-------------------------------------------------------|
| LLM 调用超时      | 重试 1 次，仍失败则返回 error 事件，不阻塞会话                          |
| MCP 工具调用失败    | 返回 tool_end 事件（含错误信息），Agent 可根据错误决定下一步                |
| SubAgent 执行超时 | DeepAgents 框架层 max_iterations 限制，超过后返回超时提示给 MainAgent |
| DB 写入失败       | 事务回滚，返回 error 响应                                      |
| 记忆/画像更新失败     | 异步任务，不影响对话响应，记录错误日志                                   |

### 12.6 性能要求

| 指标         | 要求                                   |
|------------|--------------------------------------|
| Agent 组装耗时 | ≤ 500ms（从 DB 读取配置到 Agent 实例就绪）       |
| 首 token 延迟 | ≤ 3s（从用户发送消息到第一个 token 返回）           |
| 工具调用开销     | 框架层 ≤ 50ms（不含 MCP Server / LLM 处理时间） |

### 13.3 调试信息实时入库

#### 13.3.1 设计背景

调试的 Agent 配置（Prompt、model、tool 绑定等）可能随时变化。如果仅实时展示，后续恢复历史调试会话时，看到的内容与当时的调试配置对不上。

因此**调试面板的信息需要实时入库**，保存当时的 Agent 配置快照。

#### 13.4.2 入库时机

调试会话**首轮对话完成时**（event_handler 发送 completed 事件后），将 Agent 配置快照写入 `debug_snapshots` 表。一个 session
只写入一条记录。

> 同一 session 内 Agent 配置不会变化（assembler 在首次 chat 时组装一次），每轮不同的 token_usage 和 tool_calls
> 已通过左侧对话流的事件实时展示，无需重复入库。

#### 13.4.3 入库内容

| 字段                      | 说明                        |
|-------------------------|---------------------------|
| session_id              | 关联的会话 ID（一个 session 一条快照） |
| agent_name              | 当时的 Agent 名称              |
| agent_model             | 当时的模型配置                   |
| system_prompt           | 当时渲染后的完整 Prompt（L1~L4）    |
| tool_bindings_snapshot  | 当时绑定的工具列表（JSON）           |
| skill_bindings_snapshot | 当时绑定的 Skill 列表（JSON）      |
| user_profile_snapshot   | 当时加载的用户画像内容               |
| created_at              | 快照时间                      |

## 十四、数据库设计（守一相关表）

### 14.1 sessions 表

会话映射表，管理 `user_id ↔ session_id ↔ scene_id` 的对应关系。session_id 同时作为 LangGraph Checkpointer 的 `thread_id`。

| 字段                | 类型           | 约束               | 说明                                        |
|-------------------|--------------|------------------|-------------------------------------------|
| id                | VARCHAR(36)  | PK               | UUID（session_id = Checkpointer thread_id） |
| user_id           | VARCHAR(100) | NOT NULL         | 三方用户唯一 ID                                 |
| scene_id          | VARCHAR(36)  |                  | 场景 ID（引用 scenes.id）                        |
| status            | VARCHAR(20)  | DEFAULT 'active' | `active` / `closed`                       |
| is_debug          | BOOLEAN      | DEFAULT FALSE    | 是否为调试会话                                   |
| debug_type        | VARCHAR(20)  |                  | 调试类型：`scene`（非调试时为 NULL）                   |
| is_scheduled      | BOOLEAN      | DEFAULT FALSE    | 是否为定时任务会话                                 |
| scheduled_task_id | VARCHAR(36)  |                  | 定时任务 ID（引用 scheduled_tasks.id）             |
| title             | VARCHAR(200) |                  | 会话标题（可由 LLM 自动生成）                         |
| created_at        | TIMESTAMP    | NOT NULL         | 创建时间                                      |
| updated_at        | TIMESTAMP    | NOT NULL         | 最后活跃时间（用于保活超时判断）                          |

**索引：**

- `idx_sessions_user` (user_id)
- `idx_sessions_user_scene` (user_id, scene_id)
- `idx_sessions_status` (status)

**status 取值：**

| status       | 说明                         |
|--------------|----------------------------|
| `active`     | 正常活跃                       |
| `closed`     | 已关闭（主动 / 保活超时 / 断连超时）      |

**设计说明：**

- session_id 同时作为 Checkpointer 的 thread_id，对话消息由 Checkpointer 自动管理
- 框架不维护独立的 messages 表
- scene_id 存储在 session 记录中，保证恢复对话时知道对应的 Scene

### 14.2 Checkpointer 自动管理表

由 `SeekDBCheckpointSaver` 自动创建和维护，不需要手动建表。

| 表                 | 说明                     |
|-------------------|------------------------|
| checkpoints       | 对话状态快照（按 thread_id 索引） |
| checkpoint_writes | 状态写入记录                 |

> 表结构参考 LangGraph `PostgresSaver` 的建表 SQL，适配为 MySQL 兼容语法。

### 14.3 debug_snapshots 表

调试快照表，保存调试会话的 Agent 配置快照（一个 session 一条），确保后续恢复历史调试会话时能看到当时的配置。

| 字段                      | 类型           | 约束               | 说明                                       |
|-------------------------|--------------|------------------|------------------------------------------|
| id                      | VARCHAR(36)  | PK               | UUID                                     |
| session_id              | VARCHAR(36)  | NOT NULL, UNIQUE | 关联的会话 ID（引用 sessions.id），一个 session 一条快照 |
| agent_name              | VARCHAR(100) |                  | 当时的 Agent 名称                             |
| agent_model             | VARCHAR(100) |                  | 当时的模型配置（如 deepseek-chat）                 |
| system_prompt           | TEXT         |                  | 当时渲染后的完整 Prompt（L1~L4）                   |
| tool_bindings_snapshot  | JSON         |                  | 当时绑定的工具列表快照                              |
| skill_bindings_snapshot | JSON         |                  | 当时绑定的 Skill 列表快照                         |
| user_profile_snapshot   | TEXT         |                  | 当时加载的用户画像内容                              |
| created_at              | TIMESTAMP    | NOT NULL         | 快照时间                                     |

**索引：**

- UNIQUE `idx_debug_snapshots_session` (session_id)

### 14.4 其他守一使用到的表（由管理模块负责建表）

以下表由管理模块的 CRUD 负责创建和维护，守一在 assembler 中**只读访问**：

| 表                    | 守一的使用方式                                              |
|----------------------|------------------------------------------------------|
| agents               | assembler 读取 Agent 定义（name, system_prompt, model 等）  |
| skills               | assembler 读取 Skill 索引；chat_service 读取 Skill 用于手动触发校验 |
| scenes               | assembler 读取 Scene 编排（main_agent_id, sub_agent_ids）  |
| agent_tool_bindings  | assembler 读取工具绑定关系                                   |
| agent_skill_bindings | assembler 读取 Skill 绑定关系                              |
| mcp_servers          | MCP Hub 读取 Server 连接信息                               |
| mcp_tools            | MCP Hub 读取工具 Schema                                  |
| memory_configs       | assembler 读取记忆配置参数                                   |
| scheduled_tasks      | 定时任务模块读写（守一不直接操作，但 request_context 需设置 creator 信息）   |

## 十五、与其他模块的数据边界

### 15.1 边界总览

守一的核心服务作为运行时管线，需要与多个模块交互。以下明确每个交互点的**数据边界**。

### 15.2 与管理模块的边界

| 方向        | 数据       | 说明                                                                |
|-----------|----------|-------------------------------------------------------------------|
| 守一 ← 管理模块 | Agent 定义 | 从 agents 表读取 name, description, model, system_prompt 等            |
| 守一 ← 管理模块 | Scene 编排 | 从 scenes 表读取 main_agent_id, sub_agent_ids                         |
| 守一 ← 管理模块 | 工具绑定     | 从 agent_tool_bindings 表读取 tool_type, tool_name, mcp_server_id     |
| 守一 ← 管理模块 | Skill 绑定 | 从 agent_skill_bindings 表读取 skill_id                               |
| 守一 ← 管理模块 | Skill 内容 | 从 skills 表读取 name, description, instruction, mode, user_invocable |
| 守一 ← 管理模块 | 记忆配置     | 从 memory_configs 表读取召回参数等                                        |
| 守一 ← 管理模块 | 滑动窗口配置  | 从 scenes 表读取 history_window_length                                  |

> 守一对管理模块的数据是**只读**的，不写入这些表。

### 15.3 与记忆模块的边界

| 方向        | 数据       | 调用时机                                                                              |
|-----------|----------|-----------------------------------------------------------------------------------|
| 守一 → 记忆模块 | 请求加载用户画像 | assembler Step 4（`user_profile.load(user_id)`），条件：memory_cfg.user_profile_enabled |
| 守一 ← 记忆模块 | 用户画像文本   | 注入 system_prompt L3 层                                                             |
| 守一 → 记忆模块 | 触发用户画像更新 | session 关闭后（`user_profile.update(user_id, session_id)`），记忆模块自行从 Checkpointer 获取 messages |

> 当前阶段不启用场景记忆。后续启用时增加 scene_memory.recall / scene_memory.store 的边界（同样传递 session_id，由记忆模块自行获取 messages）。

### 15.4 与 MCP Hub 的边界

| 方向           | 数据                             | 调用时机                                            |
|--------------|--------------------------------|-------------------------------------------------|
| 守一 → MCP Hub | 请求 MCP 工具实例                    | assembler Step 2（`mcp_hub.get_tools(bindings)`） |
| 守一 ← MCP Hub | MCP 工具实例列表                     | 绑定到 Agent                                       |
| 守一 → MCP Hub | user_id + extra_params + token | 运行时工具调用透传（通过 request_context）                   |

### 15.5 与定时任务模块的边界

| 方向        | 数据                                    | 说明                                            |
|-----------|---------------------------------------|-----------------------------------------------|
| 定时任务 → 守一 | scene_id + task_prompt + creator_info | TaskDispatcher 调用 assembler 组装 Agent          |
| 定时任务 → 守一 | set_request_context                   | TaskDispatcher 设置执行身份（user_id + extra_params） |
| 守一 → 定时任务 | Agent 执行结果（task_result 事件）            | 定时任务模块自行收集结果写入 execution_logs                 |

> 定时任务的执行完全复用 chat_service → assembler → Agent 的链路。差异仅在触发入口（APScheduler vs
> WebSocket）和结果出口（定时任务模块自行记录 vs WebSocket 推送）。

### 15.6 与前端的边界

| 方向          | 数据                                                                                    | 协议        |
|-------------|---------------------------------------------------------------------------------------|-----------|
| 前端 → 守一     | chat / cancel / create_session / close_session 消息                                     | WebSocket |
| 守一 → 前端     | stream / tool_start / tool_end / subagent_start / subagent_end / completed / error 事件 | WebSocket |
| 守一 → 前端（调试） | debug_* 扩展事件                                                                          | WebSocket |
| 前端 → 守一     | 调试页面 API 请求                                                                           | REST API  |
