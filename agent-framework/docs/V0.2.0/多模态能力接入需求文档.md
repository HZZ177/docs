## 一、背景与问题

### 1.1 现状

当前 Agent 基座服务仅支持**纯文本对话**。WebSocket `chat` 消息格式中只有 `message` 字段（字符串），不支持携带任何附件：

```json
{
    "action": "chat",
    "user_id": "xxx",
    "session_id": "xxx",
    "message": "纯文本消息"
}
```

Agent 与 LLM 之间的消息也仅使用 `{"role": "user", "content": "文本"}` 格式，无法传递图片或文件信息。

### 1.2 需求

| 需求 | 说明 |
|------|------|
| **文档上传与理解** | 用户上传 PDF、Word、Excel、PPT 等文档，Agent 能读取并理解其内容 |
| **图片上传与识别** | 用户上传图片（JPG、PNG、WebP），Agent 能"看到"图片内容（依赖模型视觉能力） |
| **多文件支持** | 单次消息支持同时上传多个文件 |
| **对话内引用** | Agent 在后续对话中能持续引用已上传的文件 |

### 1.3 设计目标

| 目标 | 说明 |
|------|------|
| **最小侵入** | 复用现有中间件架构，不改变 Agent 执行主流程 |
| **文档服务端处理** | 文档在服务端提取文本后以文本形式传递给 LLM，不依赖 LLM 供应商的文件处理能力 |
| **图片 Base64 传输** | 图片转为 Base64 后通过 `image_url` 内容块传递给 LLM，利用模型原生视觉能力 |
| **模型无关** | 文档能力不依赖模型类型；图片能力自动检测模型是否支持视觉 |
| **安全可控** | 文件大小限制、类型白名单、路径安全校验 |
| **OSS 持久化** | 文件持久化到对象存储（阿里云 OSS），不依赖本地文件系统，支持多实例部署 |

### 1.4 不在范围内

| 模块 | 说明 |
|------|------|
| 文件在线预览 | 前端不实现 PDF/Office 在线预览，仅显示文件名和状态 |
| 视频/音频处理 | 仅支持图片和文档，不支持视频和音频 |
| OCR 识别 | 图片走模型原生视觉能力，不做单独 OCR |

## 二、方案参考：DeerFlow 的实现

### 2.1 DeerFlow 的文件处理架构

DeerFlow 是一个基于 LangGraph 的开源多 Agent 框架，其文件上传方案有以下特点：

| 维度 | DeerFlow 做法 |
|------|--------------|
| 上传方式 | HTTP REST API（`POST /api/threads/{thread_id}/uploads`） |
| 文档转换 | 使用 `markitdown` 库将 PDF/Word/Excel/PPT 转为 Markdown |
| 图片处理 | 专用 `view_image_tool` 工具读取图片 → 转 Base64 → `ViewImageMiddleware` 注入 `image_url` 内容块 |
| 文件感知 | `UploadsMiddleware` 在消息中注入 `<uploaded_files>` XML 标签，告知 Agent 可用文件 |
| 文件访问 | Agent 通过 `read_file` 工具读取文件内容 |
| 存储隔离 | 按 thread 隔离，文件存储在 `threads/{id}/user-data/uploads/` |

### 2.2 本项目的适配策略

DeerFlow 的方案依赖 `read_file` / `view_image` 等文件系统工具，Agent 需要主动调用工具才能读取文件。这意味着：

1. 每次文件读取消耗一次工具调用轮次
2. Agent 可能"忘记"调用工具，导致文件内容未被读取
3. 对于简单场景（用户上传一个文档问问题），增加了不必要的交互轮次

**本项目采用更直接的方式**：

| 维度 | DeerFlow | 本项目方案 |
|------|----------|----------|
| 文档内容传递 | Agent 调用 `read_file` 工具读取 | **中间件直接注入文本到消息中**，Agent 无需额外操作 |
| 图片传递 | Agent 调用 `view_image` → 中间件注入 | **中间件直接将 Base64 注入 `image_url` 内容块** |
| 工具依赖 | 需要 `read_file`、`view_image` 工具 | **不需要额外工具**，通过中间件透明处理 |
| 交互轮次 | 文件读取消耗额外轮次 | **零额外轮次**，首次 LLM 调用即可获取文件内容 |

**选择理由**：本项目的场景（客服/业务助手）更注重响应效率，用户上传文件后期望 Agent 立即理解并回答，不适合增加额外的工具调用轮次。

## 三、总体架构

### 3.1 架构定位

多模态能力以**上传 API + 中间件**的形式集成，不改变 Agent 执行主流程：

```
                    ┌─────────────────────────────┐
                    │       Admin Frontend          │
                    │   (Vue 3 + Ant Design Vue)    │
                    │                               │
                    │  ┌─────────┐  ┌────────────┐  │
                    │  │ 文件选择  │  │  消息输入    │  │
                    │  └────┬────┘  └─────┬──────┘  │
                    │       │             │          │
                    └───────┼─────────────┼──────────┘
                            │             │
               HTTP Upload  │             │  WebSocket chat
               (REST API)   │             │  (附带 file_ids)
                            ▼             ▼
                    ┌─────────────────────────────┐
                    │      Agent Backend            │
                    │                               │
                    │  ┌──────────────────────┐     │
                    │  │  Upload Router        │     │
                    │  │  (接收/转换/上传OSS)   │     │
                    │  └──────────┬───────────┘     │
                    │             │ 文件元数据         │
                    │             ▼                  │
                    │  ┌──────────────────────┐     │
                    │  │  Chat Service         │     │
                    │  │  (对话主流程)          │     │
                    │  └──────────┬───────────┘     │
                    │             │                  │
                    │             ▼                  │
                    │  ┌──────────────────────┐     │
                    │  │  MultimodalMiddleware  │     │
                    │  │  (文件内容注入)        │     │
                    │  └──────────┬───────────┘     │
                    │             │ 多模态消息         │
                    │             ▼                  │
                    │  ┌──────────────────────┐     │
                    │  │  LLM (ChatOpenAI)     │     │
                    │  └──────────────────────┘     │
                    └─────────────────────────────┘
```

### 3.2 数据流全景

```
用户选择文件
    │
    ├─ Step 1: 前端调用 HTTP Upload API 上传文件
    │           → 后端接收、校验
    │           → 文档类型自动转换为 Markdown
    │           → 原始文件 + Markdown 上传至 OSS
    │           → 返回文件元数据（file_id, filename, type, oss_key）
    │
    ├─ Step 2: 前端发送 WebSocket chat 消息
    │           → message: "帮我分析这个文档"
    │           → file_ids: ["file_id_1", "file_id_2"]
    │
    ├─ Step 3: Chat Service 接收消息
    │           → 从 file_ids 查询文件元数据
    │           → 将文件信息写入 Agent 运行时上下文
    │
    ├─ Step 4: MultimodalMiddleware (abefore_agent)
    │           → 读取上下文中的文件信息
    │           → 文档类文件：从 OSS 下载 Markdown 文本，注入 system message
    │           → 图片类文件：从 OSS 下载二进制，转 Base64，构造 image_url 内容块
    │           → 修改 messages，追加文件内容
    │
    ├─ Step 5: LLM 执行
    │           → 接收到包含文本 + 图片的多模态消息
    │           → 生成回复
    │
    └─ Step 6: 流式返回结果给用户
```

### 3.3 文件类型处理策略

| 文件类型 | 扩展名 | 处理方式 | 传递给 LLM 的形式 |
|---------|--------|---------|----------------|
| PDF | `.pdf` | markitdown 转 Markdown | 文本注入 system message |
| Word | `.doc`, `.docx` | markitdown 转 Markdown | 文本注入 system message |
| Excel | `.xls`, `.xlsx` | markitdown 转 Markdown | 文本注入 system message |
| PPT | `.ppt`, `.pptx` | markitdown 转 Markdown | 文本注入 system message |
| 图片 | `.jpg`, `.jpeg`, `.png`, `.webp` | 读取二进制 → Base64 | `image_url` 内容块（多模态） |
| 其他 | — | 拒绝上传 | — |

## 四、文件上传模块

### 4.1 为什么新增 HTTP API 而非走 WebSocket

WebSocket 设计用于实时文本消息流，不适合大文件传输：

| 维度 | WebSocket | HTTP Upload |
|------|-----------|-------------|
| 传输效率 | 无分块上传、无断点续传 | 支持 multipart、可扩展断点续传 |
| 进度反馈 | 需要自行实现 | 浏览器原生支持 `XMLHttpRequest.upload.onprogress` |
| 超时处理 | 共享连接，大文件上传可能阻塞消息收发 | 独立连接，不影响 WebSocket 通道 |
| 代理/网关 | 部分反向代理对 WebSocket 帧大小有限制 | 成熟的 `client_max_body_size` 配置 |

因此，文件上传通过独立的 **HTTP REST API** 处理，上传完成后返回 `file_id`，再通过 WebSocket `chat` 消息引用。

### 4.2 上传 API 设计

#### 上传文件

```

响应:
{
    "success": true,
    "files": [
        {
            "file_id": "uuid-xxx",

            "content_type": "application/pdf",

```

### 4.3 上传处理流程

```python
async def upload_files(session_id: str, user_id: str, files: List[UploadFile]):
    """文件上传处理流程"""
    results = []

    for file in files:
        # 1. 文件名安全检查
        safe_filename = sanitize_filename(file.filename)

        # 2. 文件类型白名单校验
        ext = get_extension(safe_filename)
        if ext not in ALLOWED_EXTENSIONS:
            results.append({"filename": file.filename, "error": "不支持的文件类型"})
            continue

        # 3. 文件大小校验（读取前先检查 Content-Length）
        if file.size and file.size > MAX_FILE_SIZE:
            results.append({"filename": file.filename, "error": "文件大小超过限制"})
            continue

        # 4. 生成 file_id，读取文件内容到内存
        file_id = str(uuid4())
        file_bytes = await file.read()

        # 5. 判断文件类型
        file_type = classify_file(ext)  # "document" | "image"

        # 6. 图片类型：预处理（缩放、格式标准化）
        if file_type == "image":
            file_bytes = await preprocess_image(file_bytes)

        # 7. 上传原始文件到 OSS
        oss_key = f"uploads/{session_id}/{file_id}/{safe_filename}"
        await oss_client.put_object(oss_key, file_bytes)

        # 8. 文档类型：转换为 Markdown 并上传
        markdown_available = False
        markdown_oss_key = None
        if file_type == "document" and ext in CONVERTIBLE_EXTENSIONS:
            try:
                # 写入临时文件 → markitdown 转换 → 读取结果 → 上传 OSS → 清理临时文件
                md_text = await convert_to_markdown_from_bytes(file_bytes, safe_filename)
                markdown_oss_key = f"uploads/{session_id}/{file_id}/{safe_filename}.md"
                await oss_client.put_object(markdown_oss_key, md_text.encode("utf-8"))
                markdown_available = True
            except Exception as e:
                logger.warning(f"文档转换失败: {safe_filename}, {e}")

        # 9. 写入数据库
        await create_upload_record(
            file_id=file_id,
            session_id=session_id,
            user_id=user_id,
            filename=safe_filename,
            oss_key=oss_key,
            file_type=file_type,
            content_type=file.content_type,
            size=len(file_bytes),
            markdown_available=markdown_available,
            markdown_oss_key=markdown_oss_key,
        )

        results.append({
            "file_id": file_id,
            "filename": safe_filename,
            "file_type": file_type,
            ...
        })

    return results
```

### 4.4 OSS 存储路径规划

```
{oss_bucket}/
└── uploads/
    └── {session_id}/
        └── {file_id}/
            ├── report.pdf              # 原始文件
            ├── report.pdf.md           # 转换后的 Markdown（仅文档类型）
            ├── photo.jpg               # 图片（预处理后）
            └── ...
```

**路径设计说明**：

| 层级 | 作用 |
|------|------|
| `uploads/` | 统一前缀，便于 OSS 生命周期规则和权限策略管理 |
| `{session_id}/` | 按会话隔离，会话关闭时可按前缀批量删除 |
| `{file_id}/` | 按文件隔离，同一文件的原始版本和转换版本放在一起 |

### 4.5 OSS 客户端封装

封装统一的 OSS 操作接口，隔离具体 SDK 依赖：

```python
class OSSClient:
    """对象存储客户端（基于阿里云 OSS SDK）"""

    def __init__(self, endpoint: str, access_key_id: str, access_key_secret: str, bucket_name: str):
        import oss2
        auth = oss2.Auth(access_key_id, access_key_secret)
        self.bucket = oss2.Bucket(auth, endpoint, bucket_name)

    async def put_object(self, key: str, data: bytes | str) -> None:
        """上传文件到 OSS"""
        await asyncio.to_thread(self.bucket.put_object, key, data)

    async def get_object(self, key: str) -> bytes:
        """从 OSS 下载文件"""
        result = await asyncio.to_thread(self.bucket.get_object, key)
        return await asyncio.to_thread(result.read)

    async def get_object_text(self, key: str, encoding: str = "utf-8") -> str:
        """从 OSS 下载文本文件"""
        data = await self.get_object(key)
        return data.decode(encoding)

    async def delete_object(self, key: str) -> None:
        """删除 OSS 上的文件"""
        await asyncio.to_thread(self.bucket.delete_object, key)

    async def delete_prefix(self, prefix: str) -> None:
        """批量删除指定前缀下的所有文件（用于会话清理）"""
        # 列举并批量删除
        ...

    async def object_exists(self, key: str) -> bool:
        """检查文件是否存在"""
        return await asyncio.to_thread(self.bucket.object_exists, key)
```

**为什么用 `asyncio.to_thread`**：阿里云 OSS SDK（oss2）是同步 SDK，包装为异步避免阻塞事件循环。

### 4.6 文档转换与 OSS 的配合

由于 `markitdown` 需要读取本地文件路径，文档转换流程需要使用临时文件：

```
文件字节流 → 写入临时文件 → markitdown 转换 → 读取 Markdown 文本 → 上传到 OSS → 清理临时文件
```

```python
async def convert_to_markdown_from_bytes(file_bytes: bytes, filename: str) -> str:
    """从文件字节流转换为 Markdown 文本"""
    import tempfile
    from markitdown import MarkItDown

    with tempfile.NamedTemporaryFile(suffix=Path(filename).suffix, delete=True) as tmp:
        tmp.write(file_bytes)
        tmp.flush()

        md = MarkItDown()
        result = md.convert(tmp.name)
        return result.text_content
```

**注意**：临时文件在转换完成后立即清理，不占用持久磁盘空间。

**按 session 隔离**，会话关闭时可通过 `delete_prefix(f"uploads/{session_id}/")` 批量清理。

## 五、文档转换模块

### 5.1 转换引擎

使用 `markitdown` 库（与 DeerFlow 相同），将文档转为 Markdown 格式。

```
pip install markitdown[all,xlsx]
```

### 5.2 支持的格式

```python
CONVERTIBLE_EXTENSIONS = {
    ".pdf",
    ".ppt", ".pptx",
    ".xls", ".xlsx",
    ".doc", ".docx",
}
```

### 5.3 转换逻辑

转换流程在上传阶段完成，使用临时文件配合 `markitdown`，转换结果上传至 OSS：

```python
async def convert_to_markdown_from_bytes(file_bytes: bytes, filename: str) -> str:
    """从文件字节流转换为 Markdown 文本"""
    import tempfile
    from markitdown import MarkItDown

    with tempfile.NamedTemporaryFile(suffix=Path(filename).suffix, delete=True) as tmp:
        tmp.write(file_bytes)
        tmp.flush()

        md = MarkItDown()
        result = md.convert(tmp.name)
        return result.text_content
```

转换后的 Markdown 文本上传到 OSS（key = `{原始文件 oss_key}.md`），并在数据库记录 `markdown_oss_key`。

### 5.4 转换失败处理

- 转换失败不中断上传流程
- `markdown_available` 标记为 `false`
- 中间件注入时，如果无 Markdown 版本，则注入原始文件名提示（告知 Agent 文件存在但无法解析）
- 记录 warning 日志

### 5.5 大文件文本截断

文档转换后的 Markdown 文本可能非常长（例如一个 200 页的 PDF），直接注入会占用大量 token。需要截断：

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `max_document_tokens` | `8000` | 单个文档注入的最大 token 数（近似估算） |
| `max_total_document_tokens` | `20000` | 单次消息中所有文档的最大总 token 数 |

超过限制时，截断并附加提示：`[文档内容已截断，以上为前 {n} 字符]`

## 六、图片处理模块

### 6.1 设计思路

图片不做文本提取（不做 OCR），直接以 Base64 编码通过 `image_url` 内容块传递给 LLM，利用模型的原生视觉能力。

### 6.2 为什么使用 Base64 而非 URL

| 方式 | Token 消耗 | 可行性 |
|------|-----------|--------|
| Base64 | 取决于图片像素尺寸 | 始终可行，无需公网暴露 |
| URL | 取决于图片像素尺寸（相同） | 需要 LLM 供应商能访问到的公网 URL |

`image_url` 内容块中的 Base64 数据**不按文本 tokenize**，token 消耗由图片像素尺寸决定，与传 URL 完全一致。而本项目通过 LLM 网关对接多种模型供应商，无法保证所有供应商都能访问 OSS 的内网/签名 URL，因此 Base64 是最通用的方案。

### 6.3 支持的图片格式

```python
IMAGE_EXTENSIONS = {".jpg", ".jpeg", ".png", ".webp"}
```

### 6.4 图片预处理

为控制请求体积和 token 消耗，上传时对图片做预处理：

| 处理项 | 规则 | 说明 |
|--------|------|------|
| 尺寸缩放 | 长边不超过 2048px | 超过时等比缩放，减少 token 消耗 |
| 格式标准化 | 统一转为 JPEG（quality=85） | 减小 Base64 体积，WebP/PNG 透明通道填充白底 |
| 大小限制 | 处理后不超过 5MB | 防止单张图片占用过多请求体积 |

### 6.5 图片数量限制

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `max_images_per_message` | `5` | 单次消息最多附带的图片数 |
| `max_images_per_session` | `20` | 单个会话累计上传的图片数上限 |

## 七、中间件设计

### 7.1 MultimodalMiddleware

新增 `MultimodalMiddleware`，在 Agent 执行前将文件内容注入到 LLM 消息中。

#### 核心职责

1. 从 Agent 运行时上下文中读取当前消息关联的文件信息
2. 文档类文件：从 OSS 下载 Markdown 文本，构造 `SystemMessage` 注入
3. 图片类文件：从 OSS 下载二进制并转 Base64，改造 `HumanMessage` 为多模态格式
4. 历史文件：扫描会话已上传文件，在 system message 中提供文件列表供 Agent 参考

#### 钩子选择

| 钩子 | 用途 |
|------|------|
| `abefore_agent` | 注入文档内容和图片到 messages 中 |

**选择 `abefore_agent` 而非 `wrap_model_call`** 的原因：

- `abefore_agent` 在整个 Agent 执行前只调用一次，而 `wrap_model_call` 在每次 LLM 调用时都会触发（包括工具调用循环中的每次调用）
- 文件内容只需注入一次，不需要在每次 LLM 调用时重复注入
- 避免图片 Base64 数据在多次工具调用循环中重复发送，节省请求体积

#### 消息注入策略

**文档注入** — 以 SystemMessage 形式：

```python
SystemMessage(content="""<uploaded_documents>
以下是用户上传的文档内容：

### 文档：report.pdf
{markdown_content}

### 文档：data.xlsx
{markdown_content}

请基于以上文档内容回答用户的问题。
</uploaded_documents>""")
```

**图片注入** — 改造最后一条 HumanMessage 为多模态格式：

```python
# 原始消息
HumanMessage(content="帮我分析这张图片")

# 改造后
HumanMessage(content=[
    {"type": "text", "text": "帮我分析这张图片"},
    {
        "type": "image_url",
        "image_url": {"url": f"data:image/jpeg;base64,{base64_data}"}
    },
    {
        "type": "image_url",
        "image_url": {"url": f"data:image/jpeg;base64,{base64_data_2}"}
    }
])
```

**历史文件提示** — 对于非当前消息上传但同会话内的历史文件，仅注入文件列表（不重复注入内容）：

```python
SystemMessage(content="""<session_files>
当前会话中用户还上传过以下文件（非本次消息附带，内容已在之前的对话中处理）：
- report.pdf (1.2 MB, 文档)
- chart.png (456 KB, 图片)
</session_files>""")
```

### 7.2 在中间件链中的位置

```
自定义中间件栈：
┌──────────────────────────────────────────────────┐
│ 1. MultimodalMiddleware       ← 新增，排在最前      │
│ 2. ToolCallLimitMiddleware    ← 条件启用            │
│ 3. UserProfileMiddleware      ← 条件启用            │
│ 4. MemoryUpdateMiddleware     ← 条件启用            │
└──────────────────────────────────────────────────┘
```

**排在最前的原因**：文件内容需要在其他中间件处理之前注入，确保 UserProfile 等中间件能感知到完整的消息上下文。

### 7.3 模型视觉能力检测

图片注入前需要检测当前模型是否支持视觉。如果模型不支持视觉，图片将以文本描述形式注入：

```python

用户上传了以下图片，但当前模型不支持图片识别：
- photo.jpg (1.2 MB)
- screenshot.png (800 KB)
请告知用户当前模型不支持图片分析，建议切换到支持视觉的模型。
</uploaded_images>""")
```

视觉能力的判断方式：

| 优先级 | 方式 | 说明 |

| 兜底 | 模型名称匹配规则 | 包含 `vision`、`gpt-4o`、`claude-3` 等关键词的模型默认支持 |

### 10.2 新增 Upload API 封装

```typescript
// admin_frontend/src/api/uploads.ts

export interface UploadFileResult {
    file_id: string;
    filename: string;
    size: number;
    file_type: 'document' | 'image';
    content_type: string;
    markdown_available: boolean;
}

export interface UploadResponse {
    success: boolean;
    files: UploadFileResult[];
    errors: Array<{ filename: string; error: string }>;
}

/** 上传文件 */
export function uploadFiles(
    sessionId: string,
    userId: string,
    files: File[]
): Promise<UploadResponse> {
    const formData = new FormData();
    formData.append('user_id', userId);
    files.forEach(f => formData.append('files', f));
    return axios.post(
        `/agent-base/api/sessions/${sessionId}/uploads`,
        formData,
        { headers: { 'Content-Type': 'multipart/form-data' } }
    );
}

/** 查询会话文件列表 */
export function listUploads(
    sessionId: string,
    userId: string
): Promise<{ files: UploadFileResult[] }> {
    return axios.get(
        `/agent-base/api/sessions/${sessionId}/uploads`,
        { params: { user_id: userId } }
    );
}

/** 删除文件 */
export function deleteUpload(
    sessionId: string,
    fileId: string,
    userId: string
): Promise<{ success: boolean }> {
    return axios.delete(
        `/agent-base/api/sessions/${sessionId}/uploads/${fileId}`,
        { params: { user_id: userId } }
    );
}
```

## 十二、安全设计

### 12.1 文件名安全

```python
def sanitize_filename(filename: str) -> str:
    """安全处理文件名，防止路径遍历"""
    # 1. 提取基础文件名（去除路径）
    name = Path(filename).name

    # 2. 拒绝危险文件名
    if not name or name in {".", ".."} or "/" in name or "\\" in name:
        raise ValueError(f"不安全的文件名: {filename}")

    # 3. 移除控制字符和特殊字符
    name = re.sub(r'[<>:"|?*\x00-\x1f]', '_', name)

    return name
```

### 12.2 OSS 安全

| 策略 | 说明 |
|------|------|
| Bucket 权限 | 设置为**私有（private）**，不开放公网匿名访问 |
| OSS Key 构造 | 使用 `uploads/{session_id}/{file_id}/{filename}` 结构，file_id 为 UUID 不可猜测 |
| 访问控制 | 仅服务端通过 AK/SK 访问，前端不直接访问 OSS |
| 生命周期规则 | 可选：对 `uploads/` 前缀设置自动过期（如 30 天），作为会话清理的兜底 |

### 12.3 文件类型校验

双重校验：
1. **扩展名白名单**：仅允许 `ALLOWED_EXTENSIONS` 中的扩展名
2. **MIME 类型校验**：通过 `python-magic` 或文件头字节检测真实文件类型，防止伪造扩展名

### 12.4 会话隔离

- 文件按 session_id 隔离存储在 OSS 不同前缀下
- 上传/查询/删除 API 需要同时校验 session_id 和 user_id
- 数据库查询文件时必须带 session_id 条件，禁止跨会话访问

## 十三、边界情况与容错

| 场景 | 处理方式 |
|------|----------|
| 文档转换失败 | 不中断上传，`markdown_available=false`，中间件注入时提示 Agent 文件存在但无法解析 |
| 图片过大（超限） | 上传时预处理缩放，超过阈值仍过大则拒绝 |
| 模型不支持视觉但用户上传了图片 | 降级为文本提示，告知用户当前模型不支持图片分析 |
| 单次消息文档总 token 超限 | 按上传顺序依次注入，超过总限额后截断，提示 Agent 部分文档内容已截断 |
| 上传中 WebSocket 断开 | HTTP 上传不受 WebSocket 状态影响，上传结果由 HTTP 响应返回 |
| file_ids 引用了不存在的文件 | 忽略无效 file_id，日志记录 warning |
| 会话关闭后的文件清理 | 会话关闭时通过 `delete_prefix` 异步清理 OSS 对应前缀（可通过定时任务或 OSS 生命周期规则兜底） |
| 并发上传同一文件名 | file_id 保证唯一，OSS key 中包含 file_id 目录层级避免冲突 |
| OSS 上传失败 | 返回上传失败错误，不写入数据库记录，前端提示重试 |
| OSS 下载失败（中间件读取时） | 记录 error 日志，跳过该文件内容注入，Agent 提示文件暂时不可用 |
