## 一、文档概述

### 1.1 背景与问题

当前系统的对话消息通过 LangGraph Checkpoint 持久化到数据库，每次 Agent 执行前自动从 `checkpointer` 加载完整历史 messages，执行后自动追加新消息并保存。**没有任何上下文窗口限制或压缩机制**，`chat_service.py` 中仅有一条 TODO 注释。

这带来三个核心问题：

| 问题 | 说明 |
|------|------|
| **Context Window 溢出** | 长对话的 messages 列表无限增长，超出 LLM 的 context window 时直接报错，对话中断 |
| **响应质量下降** | 即使未溢出，过长的上下文会导致 LLM 注意力分散，回复质量下降（"Lost in the Middle" 问题） |
| **存储与性能压力** | Checkpoint 中序列化的完整消息体积持续膨胀，数据库读写延迟增大，每次 Agent 启动加载时间变长 |

### 1.2 设计目标

| 目标 | 说明 |
|------|------|
| **上下文可控** | 保证传入 LLM 的 messages 总量始终在可控范围内，不会溢出 |
| **信息保留** | 压缩后 Agent 仍能回溯早期对话的关键决策、操作记录和因果链 |
| **用户无感** | 压缩过程不阻塞用户当前对话，不增加可感知的延迟 |
| **配置灵活** | 触发阈值、保留策略、压缩模型等均可按 Scene 独立配置 |

### 1.3 重构范围

| 层级 | 涉及模块 | 变更内容 |
|------|----------|----------|
| `agent_backend` | middleware、engine | 新增 ContextCompressionMiddleware，集成到中间件链 |
| `common` | models | scenes 表增加压缩配置字段 |
| `admin_backend` | Scene CRUD | Scene 表单新增压缩配置项（常规 CRUD 变更，不展开） |
| `admin_frontend` | Scene 配置页 | 新增上下文压缩配置区域：触发阈值、压缩模型、保留轮数（见 3.9） |

### 1.4 不在范围内

| 模块 | 说明 |
|------|------|
| 前端对话界面 | 压缩过程对前端对话界面完全透明，无 UI 变更（管理后台 Scene 配置页新增压缩配置项，见 3.9） |
| Checkpoint 清理 | 旧 checkpoint 的定期归档/清理属于运维范畴，本期不实现 |
| 工具返回值截断 | DeepAgents 内置的工具输出卸载（超 20k tokens 写入文件系统）属于框架层能力，本期不涉及 |

## 二、业界方案参考

在设计本方案前，调研了业界主流项目的上下文管理策略。以下是各方案的核心思路和对本项目的参考价值。

### 2.1 LangChain SummarizationMiddleware

**来源**：LangChain 框架内置中间件（`langchain.agents.middleware.summarization`）

**核心机制**：

| 维度 | 做法 |
|------|------|
| 触发时机 | `before_model`（每次 LLM 调用前检查） |
| 触发条件 | 支持三种：消息条数、绝对 token 数、上下文窗口占比（fraction） |
| Token 计算 | 近似估算（英文 4.0 chars/token，Claude 3.3 chars/token），结合 AIMessage 的 `usage_metadata` 校准 |
| Context Window 获取 | 依赖 `BaseChatModel.profile.max_input_tokens`，无 profile 则 fraction 模式不可用 |
| 保留策略 | 保留最近 N 条消息（默认 20 条） |
| 压缩方式 | 调用轻量 LLM 对旧消息生成自由文本摘要 |
| 切分保护 | 二分搜索找切分点，保护 AIMessage(tool_calls) 与 ToolMessage 不被拆散 |
| 状态修改 | 通过 `RemoveMessage(id=REMOVE_ALL_MESSAGES)` 清空旧消息，写入摘要 + 保留消息 |

**参考价值**：触发条件的多模式设计、AI/Tool 消息对保护逻辑、token 近似计算结合 usage_metadata 校准的思路。

**不足**：摘要为自由文本，容易丢失操作骨干和因果链；`before_model` 同步执行会阻塞当前对话；保留策略只按消息条数，不区分 turn 粒度。

### 2.2 Factory.ai — 双阈值 + 增量合并

**来源**：Factory.ai 生产环境实践

**核心机制**：

| 维度 | 做法 |
|------|------|
| 双阈值 | T_max（触发线）：上下文达到此 token 数时触发压缩；T_retained（保留线）：压缩后保留的 token 上限，始终低于 T_max |
| 增量合并 | 不重新摘要全部旧消息，而是将新淘汰的消息与旧摘要合并为更新的摘要 |
| 摘要持久化 | 摘要锚定在对话中持久保存，每次压缩只处理增量部分 |
| 评估方法 | Probe-based：压缩后向 Agent 提问，验证能否回忆起被压缩内容中的关键细节 |

**参考价值**：增量合并避免重复摘要全部历史的 token 浪费；双阈值的"触发线/保留线"分离设计；Probe-based 的评估思路。

**不足**：摘要仍为自由文本摘要，随着对话增长摘要体积持续膨胀，没有过期淘汰机制。

### 2.3 JetBrains Research — NeurIPS 2025 对比实验

**来源**：JetBrains 研究团队在 NeurIPS 2025 Deep Learning 4 Code Workshop 发表的实证研究

**核心结论**：

| 维度 | 结论 |
|------|------|
| 最佳保留窗口 | 保留最近 10 轮完整内容 + 之前做摘要，是性能/效率最优的平衡点 |
| Observation Masking | 只保留最近 N 轮、丢弃更早的内容（不做摘要），在简单任务上效果与摘要接近 |
| 摘要模型 | 用独立模型做摘要（不占用主模型 token 预算），测试范围 32B~480B 参数模型 |
| 摘要粒度 | 每次摘要 21 轮，始终保留最近 10 轮原文 |

**参考价值**："保留最近 N 轮 + 之前摘要"的策略被实验验证为最优；用独立轻量模型做摘要的做法。

### 2.4 Claude Code — 结构化活动日志

**来源**：Anthropic Claude Code 的上下文压缩实现

**核心机制**：

| 维度 | 做法 |
|------|------|
| 压缩形式 | 不是语义摘要，而是**结构化活动日志**：按时间顺序逐条记录每次用户交互 |
| 用户消息 | 原文或近原文保留，编号引用 |
| 操作记录 | 具体到"修改了约 20 处"，逐条列出改了哪里 |
| 代码/文件 | 保留具体代码片段、文件路径、表结构 |
| 纠错追踪 | 专门记录用户纠正了什么以及修正后的做法 |
| 因果链 | 保留"用户说 X → Agent 做了 Y → 用户纠正 Z → Agent 改为 W"的完整链路 |
| 结构化分区 | 分为 9 个标准化区块：意图、骨干、技术概念、文件代码、错误修复、问题解决、用户消息列表、待办、当前工作 |
| 历史引用 | 压缩末尾附带原始对话文件路径，Agent 可自行回查 |

**参考价值**：这是本方案最核心的参考。Claude Code 的实践证明，**结构化提取远优于自由文本摘要**——它保留了对话骨干、因果链、纠错记录等关键信息，Agent 读到压缩内容时能像回顾工作日志一样快速定位上下文。

### 2.5 DeerFlow — 配置化集成 LangChain 压缩

**来源**：DeerFlow 开源多 Agent 框架（基于 LangGraph）

DeerFlow 的上下文压缩方案特点是**最小化集成**——直接使用 LangChain 内置的 `SummarizationMiddleware`，通过 YAML 配置文件驱动参数，不做任何自定义压缩逻辑。

**整体架构**：

```
config.yaml（YAML 配置文件）
    ↓
SummarizationConfig（Pydantic 模型，参数校验与格式转换）
    ↓
_create_summarization_middleware()（工厂函数，按配置条件化实例化）
    ↓
LangChain SummarizationMiddleware（原封不动使用，无任何定制）
    ↓
注册到中间件链第 4 位（靠前执行，先压缩再做标题/记忆等处理）
```

**配置示例**（`config.yaml`）：

```yaml
summarization:
  enabled: true
  model_name: null               # null = 用默认模型（关闭思考链的轻量版本）
  trigger:
    - type: tokens
      value: 15564               # token 超过 15564 触发
  keep:
    type: messages
    value: 10                    # 保留最近 10 条消息
  trim_tokens_to_summarize: 4000 # 摘要时最多处理 4000 tokens 的待摘要内容
  summary_prompt: null           # null = 用 LangChain 默认 prompt
```

**关键设计决策**：
- 中间件链中位置靠前（第 4 个），注释明确写道 "should be early to reduce context before other processing"
- 模型选择：若未指定 `model_name`，使用 `create_chat_model(thinking_enabled=False)` 创建关闭思考链的轻量版本
- 配置模型 `SummarizationConfig` 负责将 YAML 中的 `{type, value}` 字典格式转为 LangChain 要求的 `("type", value)` tuple 格式

**优点**：实现极简（配置文件 + Pydantic 模型 + 工厂函数约 80 行代码），开箱即用，维护成本低。

**局限性**：
- 完全依赖 LangChain 内置逻辑，无法控制压缩内容的结构和质量
- 自由格式语义摘要，容易丢失结构化信息（操作顺序、文件变更、用户纠正等）
- 无分级压缩和过期淘汰，永续 session 中摘要会持续膨胀
- `before_model` 同步执行，压缩期间阻塞用户交互
- 压缩后原始消息直接丢弃，无归档可追溯
- 全局配置，所有场景共享同一套压缩策略

### 2.6 方案对比与本项目取舍

| 维度 | LangChain | Factory.ai | JetBrains | DeerFlow | Claude Code | **本方案** |
|------|-----------|------------|-----------|----------|-------------|----------|
| 压缩形式 | 自由文本摘要 | 自由文本摘要 | 自由文本摘要 | 自由文本摘要 | 结构化日志 | **结构化日志** |
| 自研程度 | 框架内置 | 自研 | 研究原型 | 配置化集成 LangChain | 自研 | **自研** |
| 触发时机 | before_model（同步） | 阈值触发 | 固定轮数 | before_model（同步） | 阈值触发 | **after_agent（异步）+ before_model（兜底）** |
| 保留粒度 | 消息条数 | token 数 | 轮数 | 消息条数 | 轮数 | **可配置保留最近 N 轮（默认 2）** |
| 增量合并 | 不支持 | 支持 | 不明确 | 不支持 | 支持 | **支持（两级压缩）** |
| 膨胀控制 | 无 | 无 | 无 | 无 | 新建会话 | **两级压缩 + 过期丢弃** |
| 配置粒度 | 代码级 | 不明确 | 不明确 | 全局 YAML | 内置 | **Scene 级别** |

## 三、总体架构

> 本章提供所有关键流程的逻辑图，建立全局认知。各环节的详细设计见第四章到第八章。

### 3.1 架构定位

上下文压缩中间件（`ContextCompressionMiddleware`）作为 Agent 中间件链中的一环，利用 `abefore_model` 和 `aafter_agent` 两个钩子实现异步压缩与无感替换。

### 3.2 数据流全景

```
用户传入 session_id ──→ Session 解析 ──→ original_session_id + active_session_id
                                              │                      │
                                         用户侧操作              AI 侧操作
                                              │                      │
                                     event_message 写入    Checkpoint 加载（thread_id=active）
                                     WS 连接绑定                     │
                                     会话生命周期            abefore_model 检查/替换
                                              │                      │
                                              │                LLM 执行 + Tool 循环
                                              │                      │
                                     事件持久化 ←─────── 返回结果给用户
                                              │                      │
                                              │                aafter_agent
                                              │                      │
                                              │           超阈值 → 异步压缩
                                              │                      │
                                              │            创建新 Session + 更新链路
                                              │            写入暂存区
                                              │                      │
                                              │            下一轮 abefore_model 应用
                                              │                      │
持久化层：                                                           │
  message_events 表  ← session_id = original（始终不变）              │
  checkpoints 表     ← thread_id = active（压缩后指向新 Session）     │
  compression_staging 表  ← 暂存压缩结果（session_id = 新生成的 active）
```

### 3.3 触发判断流程

> 详见 [四、触发策略](#四触发策略)

```
┌──────────────────────────────────────────────────────────────┐
│               abefore_model（每次 LLM 调用前）                 │
│                                                              │
│  ① 暂存区是否有待应用的压缩结果？                                │
│     ├─ 有 → 增量合并后替换 state → 继续 ②                     │
│     └─ 没有 → 继续 ②                                        │
│                                                              │
│  ② token 是否超过紧急阈值？                                    │
│     ├─ 超了 → 同步压缩（阻塞保命）→ 放行                       │
│     └─ 没超 → 直接放行                                       │
└──────────────────────────────────────────────────────────────┘

                    ↓ Agent 正常执行 ↓

┌──────────────────────────────────────────────────────────────┐
│               aafter_agent（Agent 执行完成后）                  │
│                                                              │
│  当前 token 占比是否超过常规阈值（trigger_fraction）？            │
│     ├─ 超了 → 启动异步后台压缩任务                              │
│     └─ 没超 → 不处理                                         │
└──────────────────────────────────────────────────────────────┘
```

### 3.4 保留区确定流程

> 详见 [五、保留策略](#五保留策略--固定保留最近两轮)

```
保留最近 N 个 Turn（N = `keep_recent_turns`，默认 2）的完整原始消息，其余全部进入压缩区。

  以默认 N=2 为例：
  messages: [...older...] [Turn N-1 全部消息] [Turn N 全部消息]
                 ↓                    ↓                ↓
              压缩区              保留区            保留区

⚠ 切分点不能落在 AIMessage(tool_calls) 与 ToolMessage 之间，
   需整体归入同一区域。
```

### 3.5 两级压缩生命周期

> 详见 [六、两级压缩与过期淘汰](#六两级压缩与过期淘汰)

```
═══ 初始 ═══════════════════════════════════════════════

  [原始消息]

═══ 压缩 #1 ════════════════════════════════════════════

  新淘汰消息 ──→ L1_v1
  State:  [L1_v1] + [保留区]

═══ 压缩 #2 ════════════════════════════════════════════

  旧 L1_v1 ──→ 降级为 L2_v1
  新淘汰消息 ──→ L1_v2
  State:  [L2_v1] + [L1_v2] + [保留区]

═══ 压缩 #3（进入稳态）════════════════════════════════════

  旧 L2_v1 ──→ 丢弃
  旧 L1_v2 ──→ 降级为 L2_v2
  新淘汰消息 ──→ L1_v3
  State:  [L2_v2] + [L1_v3] + [保留区]

  ↑ 以后每次压缩都重复此模式
  大小恒定：L2(≤1500) + L1(≤5000) = 压缩部分 ≤ 6500 tokens
```

### 3.6 异步压缩与无感替换时序

> 详见 [八、异步压缩与无感替换](#八异步压缩与无感替换)，其中 [8.7](#87-压缩完整流程含双-session-id-链路) 包含标注了双 Session ID 的完整流程图。

```
Turn N（用户发送消息，session_id = A）
  │
  ├─ Session 解析：original = A, active = A.last_session_id 或 A
  │
  ├─ abefore_model（使用 active）
  │    ├─ 暂存区有结果？→ 增量合并替换 state（快速，不阻塞）
  │    └─ 紧急阈值检查 → 未超 → 放行
  │
  ├─ Agent 正常执行（thread_id = active）
  │
  ├─ 返回结果给用户（事件 session_id = original）
  │    ← 用户已收到回复，以下用户无感 ──────
  │
  └─ aafter_agent → 超阈值 → 启动后台压缩
       │
       ├── 确定保留区 → 提取骨架 → 并行 LLM 压缩
       ├── 创建新 Session（new_session_id）
       ├── 更新链路：active.child = new, A.last = new
       └── 写入暂存区（session_id = new）

Turn N+1（用户发送消息，session_id = A）
  │
  ├─ Session 解析：original = A, active = new（从 A.last 解析）
  │
  └─ abefore_model（使用 active = new）→ 暂存区有结果 → 应用
       ├─ 通过 anchor_message_id 定位切分点
       ├─ anchor 之前 → 替换为 [L2] + [L1]
       └─ anchor 及之后（含 Turn N 新增）→ 原样保留
```

### 3.7 单次压缩事件完整流水线

> 综合 [五](#五保留策略--可配置保留最近-n-轮)~[八](#八异步压缩与无感替换) 各章节

```
触发（aafter_agent 判定超阈值）
  │  输入：original_session_id, active_session_id
  │
  ├─ Step 1  确定保留区（保留最近 N 轮，N 由配置决定，默认 2）
  │          → 划分出：压缩区 ← | → 保留区
  │
  ├─ Step 2  提取对话骨架（零 LLM 调用）
  │          → 遍历压缩区，提取每轮用户消息原文，生成 Turn 列表
  │
  ├─ Step 3  并行 LLM 压缩（asyncio.gather）
  │          ├─ Task A: 对话骨架 + 压缩区原始消息 → L1 结构化活动日志
  │          └─ Task B: 旧 L1 → L2 浓缩结论（首次压缩无此步骤）
  │
  ├─ Step 4  创建新 Session + 更新链路
  │          → 创建 new_session_id
  │          → 当前 active Session 设置 child_session_id = new
  │          → 原始 Session 更新 last_session_id = new
  │
  ├─ Step 5  暂存压缩结果
  │          → 写入 compression_staging 表（session_id = new_session_id）
  │
  └─ Step 6  等待应用
             → 下一轮 abefore_model 读取暂存区，增量合并替换 state
```

### 3.8 State 中的消息层次

任意时刻，Checkpoint 的 messages 列表由以下部分组成（从前到后）：

| 层次 | 内容 | 来源 | 存在条件 |
|------|------|------|----------|
| L2 压缩摘要 | 高度浓缩的早期结论性信息 | 由旧 L1 降级而来 | 至少经历过两次压缩 |
| L1 压缩日志 | 结构化详细活动日志 | 由新淘汰消息提取而来 | 至少经历过一次压缩 |
| 原始消息 | 完整的 HumanMessage / AIMessage / ToolMessage | 未被压缩的近期对话 | 始终存在 |

**大小恒定**：无论对话持续多久，压缩部分的 token 开销上限固定（L2 ≤ 1500 tokens + L1 ≤ 5000 tokens = 总计 ≤ 6500 tokens），不会随对话增长而膨胀。

### 3.9 其他模块涉及变更

应用（Scene）配置页面新增配置项，包含以下配置项：

| 配置项 | 控件形式 | 说明 |
|--------|----------|------|
| 触发压缩阈值 | 百分比输入（0%~100%） | 对应 `trigger_fraction`，token 占比达到此值时触发异步压缩，默认 75%，建议 60%~80% |
| 快速任务模型 | 下拉选择 | 对应 `fast_task_model`，用于压缩摘要等快速轻量任务的 LLM 模型，默认 `qwen-flash`。此为 Scene 级通用配置，后续所有快速轻量任务均复用 |
| 保留最新轮数 | 数字输入 | 压缩时默认保留最新 x 轮原始消息不压缩，默认 2 轮 |

> 注：紧急触发阈值（90%）、L1/L2 大小上限等为系统内置常量，不在前端暴露。

## 四、触发策略

### 4.1 双时机触发

| 时机 | 钩子 | 触发条件 | 执行方式 | 目的 |
|------|------|----------|----------|------|
| **常规触发** | `aafter_agent` | token 占比超过 `trigger_fraction` | 异步后台执行，不阻塞 | 主动压缩，为下一轮腾出空间 |
| **紧急触发** | `abefore_model` | token 占比超过 `emergency_fraction` | 同步执行，阻塞当前调用 | 防止 LLM 调用因超出 context window 而报错 |

### 4.2 阈值配置

统一使用**上下文窗口占比（fraction）**作为触发条件，配合 LiteLLM 自动获取的 `context_window` 使用：

| 阈值 | 配置项 | 默认值 | 说明 |
|------|--------|--------|------|
| 常规阈值 | `trigger_fraction` | `0.75` | 当前 messages 总 token 数 / context_window ≥ 此值时，触发异步压缩。**可配置**，建议 0.6~0.8 |
| 紧急阈值 | `EMERGENCY_FRACTION` | `0.9` | 当前 messages 总 token 数 / context_window ≥ 此值时，触发同步压缩（保命）。**固定不可配置** |

**约束**：`EMERGENCY_FRACTION`（0.9）必须大于 `trigger_fraction`，配置时需确保 `trigger_fraction < 0.9`。

### 4.3 Token 计算方式

使用 LangChain Core 提供的官方工具函数 `count_tokens_approximately()`，该函数内置了"优先真实数据、降级近似"的逻辑。

**实现方式**：

```python
from langchain_core.messages.utils import count_tokens_approximately

def calculate_token_count(messages: list) -> int:
    """计算消息列表的 token 数（自动校准）"""
    return count_tokens_approximately(
        messages=messages,
        chars_per_token=2.0,  # 中文约 2.0，英文约 4.0
        extra_tokens_per_message=3.0,  # 每条消息的固定开销（角色、分隔符等）
        tokens_per_image=85,  # 多模态消息中每张图像的 token 成本
        use_usage_metadata_scaling=True,  # 🔥 关键：自动用最近 AIMessage 的 usage_metadata 校准
    )
```

**工作原理**：

| 参数 | 说明 |
|------|------|
| `messages` | 需要计数的消息列表（从 `state["messages"]` 传入） |
| `chars_per_token` | 字符与 token 的比率，中文约 2.0，英文约 4.0 |
| `extra_tokens_per_message` | 每条消息的固定开销（角色标记、分隔符等特殊 token） |
| `tokens_per_image` | 多模态消息中每张图像的固定 token 成本 |
| `use_usage_metadata_scaling=True` | **自动查找最近的 AIMessage 的 `usage_metadata.input_tokens`，用真实数据校准近似计算结果** |

**优势**：

1. **官方标准方法**：LangChain Core 内置函数，无需自己解析事件流或遍历消息
2. **自动校准**：`use_usage_metadata_scaling=True` 会自动查找最后一条 AIMessage 的真实 token 数据进行校准
3. **优雅降级**：如果没有 `usage_metadata`，自动降级为近似估算
4. **多模态支持**：自动处理包含图像的消息
5. **代码简洁**：一行调用，无需手动实现优先级逻辑

**适用时机**：

- **`aafter_agent`**（常规触发）：直接传入 `state["messages"]` 计算
- **`abefore_model`**（紧急触发 + 应用暂存结果）：同样传入 `state["messages"]` 计算

### 4.4 Context Window 获取

使用 [LiteLLM](https://github.com/BerriAI/litellm) 公开服务自动获取模型的 context window 信息。LiteLLM 维护了 100+ 主流模型的完整档案（包括 OpenAI、Anthropic、Google、阿里通义千问、百度文心一言、智谱 GLM 等）。

**实现方式**：

```python
from litellm import get_max_tokens

def get_context_window(model_name: str) -> int:
    """获取模型的 context window

    优先级：
    1. LiteLLM 公开服务查询（第一优先）
    2. LangChain BaseChatModel.profile.max_input_tokens（兜底）
    3. 查询失败则抛出异常，中间件注册时跳过
    """
    # 第一优先：LiteLLM 查询
    try:
        context_window = get_max_tokens(model_name)
        if context_window and context_window > 0:
            return context_window
    except Exception as e:
        logger.warning(f"LiteLLM 查询失败: {e}")

    # 第二优先：LangChain profile（仅部分模型可用）
    if hasattr(llm, 'profile') and llm.profile:
        max_input = getattr(llm.profile, 'max_input_tokens', None)
        if max_input and max_input > 0:
            return max_input

    # 兜底：抛出异常，中间件注册时跳过
    raise ValueError(f"无法获取模型 {model_name} 的 context window")
```

**获取策略**：

| 优先级 | 来源 | 说明 |
|--------|------|------|
| **第一优先** | **LiteLLM `get_max_tokens(model_name)`** | 社区维护的公开模型档案服务，支持 100+ 模型，包括国内模型 |
| 第二优先 | LangChain `BaseChatModel.profile.max_input_tokens` | 框架内置值（仅部分官方集成模型可用，通过网关访问的模型通常拿不到） |
| 兜底处理 | 查询失败时跳过中间件注册 | 记录 warning 日志，该 Scene 不启用压缩功能 |

**LiteLLM 支持的模型示例**：

```python
get_max_tokens("gpt-4")                      # 8192
get_max_tokens("gpt-4-turbo")                # 128000
get_max_tokens("claude-3-opus-20240229")     # 200000
get_max_tokens("qwen-max")                   # 通义千问
get_max_tokens("ERNIE-Bot-4")                # 文心一言
get_max_tokens("glm-4")                      # 智谱 GLM
```

**依赖安装**：

```bash
pip install litellm
```

**优势**：

1. ✅ **自动化**：无需手动配置，支持 100+ 模型自动查询
2. ✅ **国内模型支持**：覆盖通义千问、文心一言、智谱 GLM 等
3. ✅ **社区维护**：数据持续更新，跟随模型发布
4. ✅ **零成本**：本地查询，无 API 调用开销

## 五、保留策略 — 可配置保留最近 N 轮

### 5.1 设计动机

保留策略的目标是让 Agent 在压缩后仍能看到最近的完整对话原文，保持上下文连贯。默认保留最近 2 轮原始消息（可通过 `keep_recent_turns` 配置调整），规则简单且覆盖绝大多数场景——Agent 能看到"用户上一轮说了什么、自己怎么回的"以及"用户当前这轮的消息"。

### 5.2 Turn 的定义

一个 Turn（轮）= 一条 HumanMessage 到下一条 HumanMessage 之间的所有消息。

```
Turn 1: [HumanMessage, AIMessage(tool_calls), ToolMessage, ToolMessage, AIMessage(final)]
Turn 2: [HumanMessage, AIMessage]
Turn 3: [HumanMessage, AIMessage(tool_calls), ToolMessage, AIMessage(tool_calls), ToolMessage, AIMessage]
```

### 5.3 保留规则

保留最近 **N 个 Turn**（`keep_recent_turns`，默认 2）的完整原始消息，之前的全部进入压缩区。

```
Turn 1 (最早):   500 tokens   — 简单问答         → 压缩区
Turn 2:          800 tokens   — 简单对话          → 压缩区
Turn 3:       200000 tokens   — Agent 大量工具调用  → 压缩区
Turn 4:         1200 tokens   — 用户追问          → 保留区
Turn 5 (最新):   600 tokens   — 当前对话          → 保留区
```

### 5.4 AI/Tool 消息对保护

切分时必须保护 `AIMessage(tool_calls)` 与对应的 `ToolMessage` 不被拆散。如果切分点落在 ToolMessage 上，向前搜索找到发起 tool_call 的 AIMessage，将其整体归入压缩区或保留区。

## 六、两级压缩与过期淘汰

### 6.1 设计动机

如果只做单级滚动压缩（每次将旧摘要和新淘汰消息合并），摘要会随对话持续增长，最终摘要本身膨胀到不可接受的大小。需要一个过期淘汰机制，确保压缩部分的 token 开销有固定上限。

### 6.2 两级压缩模型

| 级别 | 名称 | 内容特征 | Token 上限 | 生命周期 |
|------|------|----------|-----------|----------|
| **L1** | 一级压缩（结构化详细日志） | 逐 Turn 记录用户消息、操作、回复、纠错，保留因果链 | 5000 tokens | 下一次压缩时降级为 L2 |
| **L2** | 二级压缩（浓缩结论摘要） | 只保留关键决策、核心产物、必须遵守的约束 | 1500 tokens | 再下一次压缩时丢弃 |

### 6.3 压缩事件生命周期

#### 首次压缩（未有任何压缩历史）

```
操作：
  1. 确定保留区（固定保留最近 2 轮）
  2. 压缩区消息 → L1 压缩（结构化提取）
  3. 无旧 L1，不需要 L2 降级

压缩前 State:  [原始消息 ×N 条]
压缩后 State:  [L1_v1] + [保留区原始消息]
```

#### 第二次压缩

```
操作：
  1. 确定保留区
  2. 旧 L1_v1 → L2 降级压缩 → 生成 L2_v1
  3. 新淘汰消息 → L1 压缩 → 生成 L1_v2

压缩前 State:  [L1_v1] + [原始消息 ×N 条]
压缩后 State:  [L2_v1] + [L1_v2] + [保留区原始消息]
```

#### 第三次及以后（稳态）

```
操作：
  1. 确定保留区
  2. 旧 L2_v1 → 丢弃（已过期）
  3. 旧 L1_v2 → L2 降级压缩 → 生成 L2_v2
  4. 新淘汰消息 → L1 压缩 → 生成 L1_v3

压缩前 State:  [L2_v1] + [L1_v2] + [原始消息 ×N 条]
压缩后 State:  [L2_v2] + [L1_v3] + [保留区原始消息]
```

### 6.4 State 流转全景图

```
═══ 初始 ══════════════════════════════════════════════════

  [原始消息]

═══ 压缩 #1 ═══════════════════════════════════════════════

  [L1_v1] + [原始消息(保留区)]

═══ 压缩 #2 ═══════════════════════════════════════════════

  [L2_v1] + [L1_v2] + [原始消息(保留区)]

═══ 压缩 #3 ═══════════════════════════════════════════════

  丢弃L2_v1 → [L2_v2] + [L1_v3] + [原始消息(保留区)]

═══ 压缩 #4 ═══════════════════════════════════════════════

  丢弃L2_v2 → [L2_v3] + [L1_v4] + [原始消息(保留区)]

═══ 以此类推 ═══════════════════════════════════════════════

  每次压缩：丢弃旧L2 → 旧L1降级为新L2 → 新生成L1
  State 大小恒定，不会膨胀
```

### 6.5 L1 与 L2 的内容差异

| 维度 | L1（活动日志） | L2（浓缩结论） |
|------|---------------|---------------|
| 对话骨干 | 逐 Turn 编号记录，保留因果链 | 不保留逐 Turn 记录 |
| 用户消息 | 原文或近原文保留 | 不保留 |
| 执行与产出 | 用自然语言描述做了什么、发现了什么 | 只保留最终产物 |
| 纠错记录 | 完整记录用户纠正和修正 | 只保留最终结论 |
| 决策 | 保留决策过程和原因 | 只保留决策结论 |
| 产物 | 具体到文件路径、代码片段 | 只到文件级别 |
| 目的 | Agent 回溯"最近做了什么" | Agent 知道"早期确定了什么规则" |

## 七、压缩内容策略 — 结构化提取

### 7.1 设计原则

参考 Claude Code 的实践，采用**结构化提取**而非自由文本摘要。核心区别：

| 维度 | 自由文本摘要 | 结构化提取（本方案） |
|------|-------------|-------------------|
| 信息组织 | AI 按"重要性"自由浓缩 | 按时间顺序逐 Turn 编号记录 |
| 用户消息 | 可能被摘要吞没 | 原文或近原文保留 |
| 操作记录 | 模糊概括 | 用自然语言描述做了什么、发现了什么（outcome 导向） |
| 因果链 | 被打散重组 | 完整保留 |
| 纠错记录 | 大概率丢失 | 专门记录 |
| 可审计性 | 不可预期 | 结构化模板约束输出 |

### 7.2 L1 压缩 — 结构化活动日志

#### 对话骨架（轻量辅助）

在调用 LLM 之前，用代码从压缩区消息中提取**每轮的用户消息原文**，生成极轻量的对话骨架，随 prompt 一起传给压缩 LLM，作为逐轮处理的路线图，防止遗漏：

```
对话骨架（共 N 轮，请逐轮处理）：
  Turn 1: 用户："帮我写一个用户登录接口"
  Turn 2: 用户："加上 token 刷新逻辑"
  Turn 3: 用户："现在文档中，第三方用户体系信息的参数流转是怎么设计的？"
```

骨架只包含用户消息原文和 Turn 编号，不含工具名、参数、AI 回复等内容，token 开销极小。

#### L1 输出模板

L1 压缩的 LLM 输出遵循以下固定模板，采用 **outcome 导向的描述性风格**（参考 Claude Code），用自然语言概括每轮做了什么、发现了什么，而非罗列工具调用细节：

**会话目标**：用户在这段对话中的核心目标。

**对话骨干（按时间顺序）**：逐 Turn 编号记录，每个 Turn 包含：
- 用户消息（原文或近原文保留）
- 执行与产出（用自然语言描述做了什么、得到了什么结果，如"创建了认证路由和服务文件"、"搜索了文档中所有参数流转的部分，找到了完整的流转路径"）
- 用户纠正（如果有，记录纠正内容和修正后的做法）

**关键决策与结论**：所有重要决策及其原因。

**产物变更记录**：文件、代码、数据库等的具体修改。

**遇到的问题与修复**：问题描述和修复方式。

**当前状态**：最后一次操作的结果和当前进展。

#### L1 压缩的 LLM 指令要点

- 对照对话骨架逐轮处理，不要跳过任何一轮
- 每条用户消息必须按顺序编号记录，保留原意
- 因果链（用户说 X → Agent 做了 Y → 用户纠正 Z）必须完整保留
- 用自然语言描述行为和产出，不需要罗列工具名称和参数
- 可压缩的部分：AI 的冗长推理过程、重复的工具输出、格式性内容
- 控制在 5000 tokens 以内

### 7.3 L2 压缩 — 浓缩结论摘要

#### L2 输出模板

**早期会话背景**：1-2 句话概括这段对话的整体方向。

**已确定的关键决策**：仍然有效的决策结论，每条一行。

**已完成的核心产物**：文件/代码/数据的变更记录，只到文件级别。

**需注意的约束**：后续工作必须遵守的规则或限制。

#### L2 压缩的 LLM 指令要点

- 不保留逐轮对话骨干，只保留结论
- 不保留执行过程细节
- 不保留已修复的错误（只保留仍需注意的约束）
- 被推翻的决策只保留最终结论
- 控制在 1500 tokens 以内

### 7.4 增量合并时的精简策略

当 L1 即将降级为 L2 时，需要大幅精简。精简优先级（从最先精简到最后精简）：

| 优先级 | 精简对象 | 做法 |
|--------|----------|------|
| 1（最先） | Turn 的"执行与产出"描述 | 合并为一句话概述 |
| 2 | 已修复的错误 | 删除（除非产生了仍需遵守的约束） |
| 3 | 产物变更记录 | 只保留到文件级别 |
| 4（最后） | 关键决策和约束 | 不精简 |

## 八、异步压缩与无感替换

### 8.1 设计动机

压缩需要调用 LLM 生成摘要，如果同步执行（如 LangChain 的 `before_model` 方式），会阻塞用户当前消息的响应。本方案将压缩完全异步化，用户在整个压缩过程中无感知。

### 8.2 执行时序

```
Turn N（用户发送消息）
  │
  ├─ abefore_model 检查
  │    ├─ 暂存区有待应用的压缩结果？
  │    │    ├─ 有 → 增量合并后替换 state（快速操作，不阻塞）
  │    │    └─ 没有 → 跳过
  │    └─ 紧急阈值检查
  │         ├─ 超了 → 同步压缩（阻塞但保命）
  │         └─ 没超 → 放行
  │
  ├─ Agent 正常执行（LLM 调用 + Tool 循环）
  │
  ├─ 返回结果给用户（用户已收到回复）
  │
  └─ aafter_agent 检查
       ├─ messages 未超正常阈值 → 不处理
       └─ messages 超正常阈值 → 启动后台压缩任务
            │
            ├─ 确定保留区 → 提取对话骨架 → 并行压缩
            ├─ 并行执行：
            │    ├─ Task A: 对话骨架 + 压缩区原始消息 → L1 压缩
            │    └─ Task B: 旧 L1 → L2 降级（如果有）
            ├─ 压缩完成 → 暂存结果到暂存区
            └─ 等待下一轮 before_model 应用

Turn N+1（用户发送下一条消息）
  │
  └─ abefore_model 检查
       └─ 暂存区有压缩结果 → 应用
            ├─ 取出压缩后的 [L2, L1]
            ├─ 找到 anchor 之后的增量消息（Turn N 期间新增的）
            ├─ 组装新 state = [L2] + [L1] + [增量消息] + [Turn N+1 新消息]
            └─ 替换 state，清除暂存区
```

### 8.3 暂存区设计

压缩结果持久化到 `compression_staging` 表，以 `session_id` 为唯一键。

暂存记录包含：

| 字段 | 说明 |
|------|------|
| `session_id` | 唯一键，同一 session 只保留一条暂存记录 |
| `compressed_messages` | 压缩后的消息列表（L2 + L1 的 SystemMessage，JSON 序列化） |
| `anchor_message_id` | 保留区第一条消息的 `id`（详见下方说明） |

#### anchor_message_id 说明

LangGraph 中每条 message 都有一个框架自动生成的 `id` 字段（UUID），持久化在 checkpoint 中，不会改变。

压缩时确定保留区后，**保留区第一条消息的 `id`** 即为 anchor。它是压缩区与保留区的分界点——后续合并时，通过这个 `id` 在 messages 列表中定位切分位置。

```
压缩时的 messages:

  msg.id=aaa  HumanMessage    ┐
  msg.id=bbb  AIMessage       │ 压缩区（将被替换为 L2 + L1）
  msg.id=ccc  ToolMessage     ┘
  msg.id=ddd  HumanMessage    ← anchor_message_id = "ddd"（保留区起点）
  msg.id=eee  AIMessage       ┐
  msg.id=fff  HumanMessage    │ 保留区（原样保留）
  msg.id=ggg  AIMessage       ┘
```

#### 写入与覆盖策略

| 场景 | 行为 |
|------|------|
| 首次压缩完成 | INSERT 一条暂存记录 |
| 同一 session 再次压缩完成（前一次尚未被应用） | 按 `session_id` 覆盖更新（UPSERT） |
| `abefore_model` 应用暂存结果后 | DELETE 该记录 |

### 8.4 增量合并逻辑

`abefore_model` 应用压缩结果时，需要处理"压缩期间新产生的消息"。

合并步骤：
1. 遍历当前 messages，找到 `id == anchor_message_id` 的消息位置
2. 该位置之前的所有消息 → 丢弃，替换为暂存区的 `compressed_messages`（L2 + L1）
3. 该位置及之后的所有消息（含压缩期间新增的）→ 原样保留

```
压缩时的 state:

  msg.id=aaa  HumanMessage    ┐
  msg.id=bbb  AIMessage       │ 压缩区
  msg.id=ccc  ToolMessage     ┘
  msg.id=ddd  HumanMessage    ← anchor（id="ddd"）
  msg.id=eee  AIMessage       ┐ 保留区
  msg.id=fff  HumanMessage    │
  msg.id=ggg  AIMessage       ┘

压缩期间用户又发了消息，当前 state 多了 hhh、iii：

  msg.id=aaa  HumanMessage
  msg.id=bbb  AIMessage
  msg.id=ccc  ToolMessage
  msg.id=ddd  HumanMessage    ← 通过 id="ddd" 找到这里
  msg.id=eee  AIMessage
  msg.id=fff  HumanMessage
  msg.id=ggg  AIMessage
  msg.id=hhh  HumanMessage    ┐ 压缩期间新增
  msg.id=iii  AIMessage       ┘

合并后的 state：

  [L2 SystemMessage]          ┐ 替换 aaa~ccc
  [L1 SystemMessage]          ┘
  msg.id=ddd  HumanMessage    ← anchor 及之后全部保留
  msg.id=eee  AIMessage
  msg.id=fff  HumanMessage
  msg.id=ggg  AIMessage
  msg.id=hhh  HumanMessage
  msg.id=iii  AIMessage
```

因为 message `id` 是稳定的 UUID，无论压缩期间新增了多少消息，都能可靠定位分界点。

### 8.5 并行压缩

每次压缩事件需要两个 LLM 调用（L1 新生成 + L2 从旧 L1 降级），两者互不依赖，使用 `asyncio.gather` 并行执行，减少总耗时。

### 8.6 双 Session ID 机制与链式管理

#### 8.6.1 设计动机

压缩后 checkpoint 写入新 Session，但用户对此无感知。系统内部需要同时维护两个 Session ID：

- **`original_session_id`**：用户创建对话时生成的 ID，用户始终持有，**永不变化**
- **`active_session_id`**：当前实际承载 checkpoint 的 Session ID，**每次压缩后指向新 Session**

未发生过压缩时，`original_session_id == active_session_id`。

分离的原因：
- **用户侧**（WS 连接、事件推送、消息持久化、会话生命周期）需要始终绑定原始会话，保证用户无感
- **AI 侧**（checkpoint 读写、Agent 执行）需要指向最新的压缩后 Session，保证加载正确的上下文
- **回退能力**：每个 Session 的 checkpoint 都是独立快照，可作为回退点（面向编程 Agent 等需要撤销压缩的场景）

#### 8.6.2 Session 链式结构

sessions 表新增字段（见 [十一、数据库变更](#十一数据库变更)）：

| 字段 | 说明 |
|------|------|
| `child_session_id` | 该 Session 压缩后生成的下一个 Session ID（直接后继） |
| `last_session_id` | 整条链路中最新的 Session ID（快捷指针，仅在原始 Session 上维护） |

链路示例（经过 3 次压缩）：

```
Session A（原始，用户持有的 session_id）
  child_session_id = B
  last_session_id  = D    ← 快捷指针，始终指向最新
  → 所有 event_message 都存储在此 session_id 下
  → checkpoint：压缩前的完整原始消息（冻结，可作为回退点 #0）

Session B（第 1 次压缩产物）
  child_session_id = C
  → 无 event_message
  → checkpoint：[L1_v1] + [保留区]（回退点 #1）

Session C（第 2 次压缩产物）
  child_session_id = D
  → 无 event_message
  → checkpoint：[L2_v1] + [L1_v2] + [保留区]（回退点 #2）

Session D（第 3 次压缩产物，当前活跃）
  child_session_id = null
  → 无 event_message
  → checkpoint：[L2_v2] + [L1_v3] + [保留区]
```

**关键规则**：
- **event_message 始终写入 `original_session_id`**（Session A），不跟随链路迁移，无需跨 Session 聚合
- 每个 Session 的 checkpoint 都是**独立、干净的快照**，可独立作为回退点
- `last_session_id` 仅在原始 Session 上维护，提供 O(1) 直达最新 Session 的能力

#### 8.6.3 压缩时的链路更新

每次压缩创建新 Session 时执行以下操作：

1. 创建新 Session（`new_session_id`），写入压缩后的 checkpoint
2. 当前活跃 Session 设置 `child_session_id` → `new_session_id`
3. 原始 Session 更新 `last_session_id` → `new_session_id`

```
首次压缩：
  压缩前：A (last=null, child=null)
  压缩后：A (last=B, child=B) → B (child=null)

第二次压缩：
  压缩前：A (last=B, child=B) → B (child=null)
  压缩后：A (last=C, child=B) → B (child=C) → C (child=null)

第三次压缩：
  压缩前：A (last=C, child=B) → B (child=C) → C (child=null)
  压缩后：A (last=D, child=B) → B (child=C) → C (child=D) → D (child=null)
```

#### 8.6.4 Session 解析规则

`chat_service.handle_chat()` 入口处执行一次解析，分离出双 ID 并注入 `request_context`：

```
用户传入 session_id（始终是 original）
  │
  ├─ original_session_id = session_id（直接使用，不变）
  │
  └─ active_session_id = ?
       ├─ 查询该 session 的 last_session_id
       │    ├─ 不为空 → active_session_id = last_session_id（O(1) 直达最新）
       │    └─ 为空 → active_session_id = session_id（未经过压缩，自身即最新）
       │
       └─ 两个 ID 分别注入 request_context
```

#### 8.6.5 request_context 扩展

`common/core/request_context.py` 需要支持双 Session ID：

```python
# 原有 session_id_var 拆分为两个
original_session_id_var: ContextVar[Optional[str]]  # 用户侧，永不变
active_session_id_var: ContextVar[Optional[str]]    # AI 侧，压缩后变

# getter / setter
def get_original_session_id() -> Optional[str]: ...
def set_original_session_id(value: str): ...
def get_active_session_id() -> Optional[str]: ...
def set_active_session_id(value: str): ...

# 兼容：保留 get_session_id() 作为 get_original_session_id() 的别名
# 现有代码调用 get_session_id() 无需修改，默认获取 original
def get_session_id() -> Optional[str]:
    return get_original_session_id()
```

#### 8.8.2 核心原则

1. **用户侧一切操作绑定 `original_session_id`**：event_message、WS 连接、会话生命周期、调试快照、记忆更新——凡是面向用户或面向前端的操作，使用 original
2. **AI 侧一切操作绑定 `active_session_id`**：checkpoint 读写、Agent 执行配置——凡是面向 LangGraph 的操作，使用 active
3. **压缩中间件是唯一的桥梁**：负责创建新 Session、维护链路指针、在两个 ID 之间建立映射
4. **解析只在入口做一次**：`chat_service.handle_chat()` 入口解析后注入 `request_context`，下游各模块通过 getter 获取所需的 ID
5. **兼容性**：`get_session_id()` 保留为 `get_original_session_id()` 的别名，现有代码默认获取 original，无需批量修改

> **关于 event_message 的历史连续性**：所有 event_message 始终写入 `original_session_id`，不跟随压缩链路迁移。前端恢复历史消息时直接按原始 session_id 查询即可，**无需跨 Session 聚合**。这是与旧设计的核心区别——旧设计将 event_message 分散在链路各节点，需要聚合；新设计将其钉在原始 Session，查询路径零改动。

## 十、配置设计

### 10.1 配置位置

压缩配置作为 Scene 配置的一部分，存储在 `scenes` 表中（JSON 字段），与 Scene 一对一关系。不同 Scene 可以有不同的压缩策略。

### 10.2 配置项

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `compression_enabled` | bool | `false` | 是否启用压缩 |
| `trigger_fraction` | float | `0.75` | 常规触发阈值，token 占比达到此值时异步压缩（0~1），建议 0.6~0.8 |
| `keep_recent_turns` | int | `2` | 压缩时保留最新几轮原始消息不压缩，建议 2~5 |

**注意**：
- 压缩使用的 LLM 模型取自 Scene 级通用配置 `fast_task_model`（见 11.2），不在 `compression_config` 中单独配置
- `context_window` 不再需要手动配置，由 LiteLLM 根据模型名称自动查询
- 紧急触发阈值固定为 90%（代码内置，不可配置）
- 其他参数（L1/L2 大小、工具参数截断等）均为内置常量，无需配置

**内置常量**（代码中定义，不可配置）：

| 常量 | 值 | 说明 |
|------|------|------|
| `EMERGENCY_FRACTION` | `0.9` | 紧急触发阈值，token 占比达到 90% 时同步压缩（保命），固定不可配置 |
| `L1_MAX_TOKENS` | `5000` | L1 结构化活动日志的最大 token 数 |
| `L2_MAX_TOKENS` | `1500` | L2 浓缩结论摘要的最大 token 数 |
| `TRUNCATE_TOOL_ARGS` | `true` | 是否截断旧消息中的大型工具参数（性能优化） |
| `TRUNCATE_MAX_LENGTH` | `2000` | 工具参数截断的最大字符数 |

### 10.3 配置示例

**最简配置**（使用默认值）：

```json
{
  "compression_enabled": true
}
```

**自定义触发阈值**（小窗口模型，更激进的压缩策略）：

```json
{
  "compression_enabled": true,
  "trigger_fraction": 0.6
}
```

**完整配置**：

```json
{
  "compression_enabled": true,
  "trigger_fraction": 0.75,
  "keep_recent_turns": 3
}
```

## 十二、边界情况与容错

### 12.1 边界情况处理

| 场景 | 处理方式 |
|------|----------|
| LiteLLM 查询模型 context window 失败 | 尝试 LangChain profile 兜底，都失败则记录 warning 日志并跳过中间件注册，该 Scene 不启用压缩 |
| 压缩还没完成，用户就发了下条消息 | `abefore_model` 查询 staging 表无记录，正常执行，压缩完成后在后续轮次应用 |
| 同一 session 并发触发两次压缩 | 后完成的压缩结果按 session_id UPSERT 覆盖前一次 |
| 压缩 LLM 调用失败 | 记录错误日志，不影响主对话流程，下一轮 `aafter_agent` 重新判断是否需要压缩 |
| 紧急阈值触发同步压缩也失败 | 降级为简单截断：直接移除最早的消息直到 token 数降至安全范围，不做摘要 |
| 暂存结果长时间未被应用 | 不影响系统运行，记录持久化在 DB 中不会丢失，用户下次发消息时自动应用 |
| 压缩后 messages 中只剩摘要，没有原始消息 | 不应发生（固定保留最近 2 轮原始消息） |
| 首轮对话就超过阈值（单轮极长） | 保留区只有当前这 1~2 轮，压缩区为空或过少，跳过本次压缩 |

### 12.2 容错原则

| 原则 | 说明 |
|------|------|
| **主流程优先** | 压缩失败绝不影响用户正常对话，所有异常静默降级 |
| **幂等安全** | 同一批消息多次压缩结果一致（虽然不要求精确相同，但语义等价） |
| **日志可追踪** | 每次压缩事件记录详细日志：触发原因、压缩区范围、保留区范围、L1/L2 版本号、耗时 |

### 13.1 压缩中间件的直接依赖

| 本文档模块 | 依赖的外部模块 | 交互方式 |
|------------|---------------|----------|
| ContextCompressionMiddleware | `agent_factory.get_or_create_llm()` | 调用其创建摘要用的 LLM 实例 |
| ContextCompressionMiddleware | `request_context` | 读取 session_id |
| ContextCompressionMiddleware | `SeekDBCheckpointSaver` | 压缩后通过 LangGraph state 更新间接修改 checkpoint |
| ContextCompressionMiddleware | `compression_staging` 表 | 暂存 / 读取 / 删除压缩结果 |
| ContextCompressionMiddleware | Scene 配置 | 读取 `compression_config` JSON 字段 |
| chain_builder | ContextCompressionMiddleware | 条件注册中间件 |
| UserProfileMiddleware | ContextCompressionMiddleware | 无直接依赖，但执行顺序在其之后（画像注入不受压缩影响） |
| MemoryUpdateMiddleware | ContextCompressionMiddleware | 无直接依赖，记忆更新从 state 取最近消息，不受压缩影响 |
| admin_backend Scene CRUD | `compression_config` | 管理后台负责配置项的增删改查和前端表单 |

### 13.2 压缩与 Session 关联数据的关系

压缩**只修改 checkpoint 中的 messages**（AI 的工作上下文），**不修改**其他 session 关联表的数据。各表在压缩前后的状态如下：

| 表 | 压缩是否修改 | 说明 |
|----|------------|------|
| `checkpoints` | **是** | 压缩的目标——旧消息被替换为 L2 + L1 + 保留区原始消息 |
| `checkpoint_writes` | **是** | 随 checkpoint 联动更新 |
| `message_events` | **否** | 完整的事件流记录，不受压缩影响 |
| `debug_snapshots` | **否** | 只记录首轮 Agent 配置快照，与消息内容无关 |
| `memories` | **否** | 记忆独立于消息，按 user_id + agent_id 管理 |
| `sessions` | **是** | 压缩后更新 `child_session_id` 和 `last_session_id`，建立 Session 链路 |
| `compression_staging` | **新增/覆盖/删除** | 暂存压缩结果，应用后删除 |

### 13.3 两套数据各司其职

压缩后同一个 session 存在两套消息数据，服务于不同读取方：

```
┌─────────────────────────────────────────────────────────┐
│                     同一个 Session                       │
│                                                         │
│  checkpoint（AI 读取）                                   │
│    → 压缩后的工作上下文                                    │
│    → [L2] + [L1] + [最近 2 轮原始消息]                    │
│    → 目的：控制 context window，保证 Agent 正常执行        │
│                                                         │
│  message_events（前端读取）                               │
│    → 完整的历史事件流，逐条记录                             │
│    → 不受压缩影响，始终保持完整                             │
│    → 目的：用户查看历史对话记录                             │
└─────────────────────────────────────────────────────────┘
```

**关键约束**：前端恢复历史消息**必须从 `message_events` 读取**，不能从 checkpoint 读取。压缩后 checkpoint 中的消息列表不再是完整的对话历史。
