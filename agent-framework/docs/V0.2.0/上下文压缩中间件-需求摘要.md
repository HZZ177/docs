# 上下文压缩中间件 — 需求摘要

> 面向 Coding 平台的需求文档，仅保留需求与约束，不含实现细节。

---

## 一、背景与问题

当前对话消息通过 LangGraph Checkpoint 持久化，无上下文窗口限制或压缩机制，导致：

| 问题 | 说明 |
|------|------|
| **Context Window 溢出** | 长对话 messages 无限增长，超出 LLM context window 时对话中断 |
| **响应质量下降** | 过长上下文导致 LLM 注意力分散（"Lost in the Middle"） |
| **存储与性能压力** | Checkpoint 体积膨胀，数据库读写与 Agent 加载变慢 |

---

## 二、设计目标

| 目标 | 说明 |
|------|------|
| **上下文可控** | 传入 LLM 的 messages 总量始终在可控范围内，不溢出 |
| **信息保留** | 压缩后仍能回溯早期对话的关键决策、操作记录和因果链 |
| **用户无感** | 压缩不阻塞当前对话，不增加可感知延迟 |
| **配置灵活** | 触发阈值、保留策略、压缩模型等可按 Scene 独立配置 |

---

## 三、范围说明

**在范围内：**

- agent_backend：新增上下文压缩中间件并接入中间件链
- common：scenes 表增加压缩相关配置字段
- admin_backend：Scene CRUD 支持压缩配置项
- admin_frontend：Scene 配置页增加「触发阈值、压缩模型、保留轮数」等配置区域

**不在范围内：**

- 前端对话界面无 UI 变更（对用户透明）
- Checkpoint 定期归档/清理（运维范畴）
- 工具返回值截断等框架层能力

---

## 四、总体行为要求

### 4.1 触发策略

- **常规触发**：在 Agent 执行完成后（aafter_agent），若当前 messages 的 token 占比超过可配置的常规阈值，则**异步**启动压缩，不阻塞用户。
- **紧急触发**：在每次 LLM 调用前（abefore_model），若 token 占比超过固定紧急阈值（如 90%），则**同步**执行压缩，避免调用因超窗报错。
- 阈值统一采用「当前 token 数 / 模型 context_window」的占比；context_window 需能按模型名自动获取（如通过 LiteLLM 或 LangChain profile），无需在配置里手填。

### 4.2 保留策略

- 压缩时**保留最近 N 轮**完整原始消息不压缩，N 可配置（默认 2）。
- 一轮（Turn）定义为：一条用户消息到下一条用户消息之间的所有消息。
- 切分时**不得拆散**「AIMessage(tool_calls) 与对应 ToolMessage」对，需整体归入同一侧（压缩区或保留区）。

### 4.3 两级压缩与膨胀控制

- **L1（一级压缩）**：结构化详细活动日志，保留逐轮用户消息、操作与产出、纠错、因果链；有 token 上限（如 5000）；下次压缩时降级为 L2。
- **L2（二级压缩）**：浓缩结论摘要，只保留关键决策、核心产物、必须遵守的约束；有更小 token 上限（如 1500）；再下一次压缩时**丢弃**。
- 压缩部分总 token 有固定上限（L2 + L1），不随对话增长而无限膨胀。

### 4.4 压缩内容形式

- 采用**结构化提取**，而非自由文本摘要：按时间顺序、逐 Turn 编号记录，保留用户消息原意、执行与产出、纠错、因果链。
- L1 需包含：会话目标、对话骨干（每轮用户消息 + 执行与产出 + 用户纠正）、关键决策与结论、产物变更记录、问题与修复、当前状态。
- L2 需包含：早期会话背景、已确定的关键决策、已完成的核心产物（到文件级）、需注意的约束。

### 4.5 异步压缩与无感替换

- 常规触发时，压缩在**后台异步**执行；用户在本轮已收到回复，对压缩过程无感知。
- 压缩结果写入**暂存区**（按 session 唯一），在**下一轮** LLM 调用前（abefore_model）再应用到 state：用「保留区起点消息 id」定位切分点，之前替换为 [L2]+[L1]，之后（含压缩期间新增消息）原样保留。
- 若压缩尚未完成用户就发下一条消息：本轮不应用，压缩完成后在后续轮次应用即可。

### 4.6 Session 与历史展示

- 压缩后可选择「写回新 Session + Session 链」或「原地替换 checkpoint」等方案；若采用新 Session，需保证：用户始终用同一 session_id 交互，后端透明解析到「当前最新」Session；前端恢复历史对话时，**必须从完整事件流（如 message_events）读取**，不能从已压缩的 checkpoint 读取。

## 五、配置需求

### 5.1 Scene 级配置

- **是否启用压缩**：布尔，默认关闭。
- **常规触发阈值**：0~1 比例，默认 0.75，建议 0.6~0.8。
- **保留最近轮数**：整数，默认 2，建议 2~5。
- **压缩用模型**：Scene 级「快速任务模型」，用于压缩摘要等轻量任务，与主对话模型可不同。

### 5.2 不暴露的常量

- 紧急触发阈值（如 90%）、L1/L2 token 上限等为系统内置，不在前端或配置中暴露。

---

## 六、数据与存储需求

### 6.1 暂存压缩结果

- 需要按 **session 唯一** 的暂存区，存储：压缩后的消息列表（L2 + L1）、保留区起点消息 id（用于下一轮合并时定位）。
- 同一 session 再次压缩可覆盖前一次暂存；应用后删除该条暂存。

### 6.2 Scene 配置存储

- scenes 表需支持：压缩开关、常规触发阈值、保留轮数（及可选 fast_task_model）；可为 JSON 或独立字段，与现有 Scene 配置方式一致。

### 6.3 Session 链路（若采用「新 Session」方案）

- 若压缩后写入新 Session，sessions 表需能表达「当前 Session 压缩后的后继 Session」以及「整条链路中的最新 Session」，以便用原始 session_id 解析到最新 Session，且不影响前端从事件流拉取完整历史。

---

## 七、边界与容错

- **主流程优先**：压缩失败不得影响用户正常对话，异常静默降级。
- **context_window 获取失败**：可跳过该 Scene 的压缩中间件注册并打日志，不启用压缩。
- **压缩 LLM 调用失败**：记日志，下一轮再判断是否压缩；不阻塞主流程。
- **紧急同步压缩也失败**：可降级为「简单截断最早消息直至安全范围」，不做摘要。
- **同一 session 并发两次压缩**：后完成的可覆盖先完成的暂存结果（按 session 唯一）。
- **首轮或极少量消息就超阈值**：保留区不足或压缩区过少时，可跳过本次压缩。

---

## 八、与其他模块的边界

- 压缩**只修改** checkpoint 中的 messages（以及若采用新 Session 则 sessions 的链路字段），**不修改** message_events 等完整事件流；前端历史**必须**从事件流获取。
- 管理后台：负责 Scene 压缩配置的增删改查与表单展示。
- 中间件链中：压缩相关逻辑需在「应用/判断上下文」的合适位置执行（例如在应用压缩结果后再让其他中间件处理 messages），具体顺序由实现决定，需求上不规定具体文件或类名。

---

## 九、前端（管理后台）需求

- 在 Scene 配置页增加「上下文压缩」配置区域，包含：
  - 触发压缩阈值（百分比，对应常规触发占比）
  - 快速任务模型（下拉等，用于压缩）
  - 保留最新轮数（数字输入）
- 不展示紧急阈值、L1/L2 上限等内置常量。
