## 一、文档概述

### 1.1 背景与问题

当前调试页面的「调用链路」Tab 展示的内容是框架中间件的生命周期事件（ToolFilterMiddleware、ToolCallLimitMiddleware、UserProfileMiddleware、MemoryUpdateMiddleware 的 start/end），对开发者和运维人员**没有实际诊断价值**。

核心问题：

| 问题 | 说明 |
|------|------|
| **信息无价值** | 中间件执行耗时通常 < 5ms，知道"工具过滤中间件完成了"对排查问题毫无帮助 |
| **缺少关键信息** | 没有记录 LLM 的完整输入输出、工具调用的参数和返回值、token 消耗明细 |
| **缺少层级关系** | 当前是扁平时间线，无法看出"这个工具调用是哪次 LLM 决策触发的" |
| **仅调试可用** | 链路数据仅在 `is_debug=True` 时采集，生产环境完全无数据 |
| **数据耦合** | 链路数据以 JSON 追加方式存储在 `debug_snapshots` 表中，与配置快照耦合 |
| **无法支撑监控** | 没有独立的追踪表，无法按时间/场景/用户维度做聚合查询 |

### 1.2 设计目标

| 目标 | 说明 |
|------|------|
| **信息有价值** | 追踪每轮对话中的 LLM 调用、工具调用、SubAgent 执行的完整输入输出和耗时 |
| **层级清晰** | 通过 parent_run_id 建立树形调用关系，一眼看出因果链 |
| **全量采集** | 所有对话（调试 + 生产）都采集链路数据，为监控面板做铺垫 |
| **实时写入** | Run 级别实时 INSERT/UPDATE，对话进行中即可查看链路状态 |
| **独立存储** | 新建 `trace_records` + `trace_runs` 两张表，与 debug_snapshot 解耦 |
| **前端重构** | 调用树 + 详情面板的双栏布局，按轮次分组展示 |

### 1.3 重构范围

| 层级 | 涉及模块 | 变更内容 |
|------|----------|----------|
| `common` | models | 新增 `trace_records` 表、`trace_runs` 表 |
| `common` | models | `debug_snapshots` 表删除 `call_chain_data`、`llm_calls_data` 两个字段 |
| `common` | common_service | 新增 `trace_collector.py` 采集服务 |
| `agent_backend` | engine | `event_handler.py` 集成 TraceCollector 调用 |
| `agent_backend` | services | `chat_service.py` 管理 Trace 生命周期 |
| `admin_backend` | api | `debug_router.py` 新增两个 trace 查询接口 |
| `admin_frontend` | pages/debug | 重构 `CallChainTab.vue`，新增 `TraceTree.vue`、`RunDetail.vue` |
| `admin_frontend` | composables | 重构 `useDebugStore.ts` 中链路相关逻辑 |

### 1.4 不在范围内

| 模块 | 说明 |
|------|------|
| 监控面板页面 | 本期只做数据采集和调试页面展示，不做独立监控面板页面 |
| 监控面板 API | 仅在文档中预留接口设计说明，不实现 |
| 告警系统 | 错误率告警、延迟告警等不在本期范围 |
| 成本计算 | Token 成本估算需要模型定价表，本期不实现 |
| 数据清理 | 历史 trace 数据的定期清理/归档属于运维范畴，本期不涉及 |

## 二、设计参考

### 2.1 LangSmith

**核心概念**：Run（等价于 OpenTelemetry Span）

- 每个 Run 有 `run_type`（chain / llm / tool / retriever / embedding）
- 通过 `parent_run_id` 建立父子嵌套关系，形成 Trace 树
- 每个 Run 记录完整的 `inputs`、`outputs`、`error`、`start_time`、`end_time`、`token_usage`、`metadata`
- UI 提供**树形视图**（层级展开）和**瀑布图**（时间轴横条，看并行/串行）

**对本项目的参考价值**：
- Run 数据模型设计（扁平表 + parent_run_id 表达任意深度嵌套）
- `run_type` 分类体系
- `inputs` / `outputs` 全量记录的理念

### 2.2 AxonHub

**核心概念**：Trace → Segment → Span（三层结构）

- Segment = 一次 LLM 请求（含嵌套子请求）
- Span 分两类：`RequestSpans`（系统指令、用户消息）和 `ResponseSpans`（文本输出、工具调用、工具结果）
- 前端使用**平铺时间线** + **详情面板**的双栏布局
- 点击时间线节点，右侧展示该节点的完整内容

**对本项目的参考价值**：
- 前端双栏布局设计（左侧调用树 + 右侧详情面板）
- Span 类型体系（system_instruction / user_query / text / tool_use / tool_result）
- Token 统计聚合展示

### 2.3 综合取舍

| 维度 | 采纳方案 | 理由 |
|------|----------|------|
| 数据模型 | LangSmith 的 Run 模型 | 扁平表 + parent_run_id，灵活支撑任意深度嵌套（SubAgent 内嵌 SubAgent） |
| 层级粒度 | 不采用 AxonHub 的三层结构 | 我们有 SubAgent、中间件，层级不固定，三层结构不够灵活 |
| 前端布局 | 钻入式导航（调试面板空间有限） | 第一层调用树视图 → 点击节点钻入第二层详情页 → 返回键回到调用树。避免双栏布局在窄面板中过于拥挤 |
| 采集策略 | 参考 LangSmith 的全量采集 | 所有对话都记录，支撑后续监控面板 |

## 三、数据模型设计

### 3.1 trace_records 表（追踪记录表）

一轮对话对应一条 trace 记录。同一个 session 的多轮对话通过 `session_id` 关联。

```python
class TraceRecord(BaseModel):
    """trace_records 表 — 追踪记录，一轮对话一条"""

    __tablename__ = "trace_records"
    __table_args__ = (
        Index("udx_trace_id", "trace_id", unique=True),
        Index("idx_session_id", "session_id"),
        Index("idx_scene_id", "scene_id"),
        Index("idx_user_id", "user_id"),
        Index("idx_status", "status"),
        {"comment": "追踪记录表，每轮对话产生一条trace，记录整轮的聚合统计"}
    )

    id = Column(String(36), primary_key=True, default=uuid4, comment="主键ID（UUID）")
    trace_id = Column(String(36), nullable=False, unique=True, comment="追踪唯一标识（UUID），业务主键")
    session_id = Column(String(36), nullable=False, comment="关联的会话ID（关联 sessions.id），同一会话多轮对话共享")
    scene_id = Column(String(36), nullable=False, comment="场景ID（关联 scenes.id）")
    user_id = Column(String(64), nullable=False, comment="用户ID")

    # 状态与时间
    status = Column(String(20), nullable=False, default="running", comment="状态：running-执行中 / completed-完成 / error-错误 / cancelled-已取消")
    start_time = Column(DateTime, nullable=False, comment="本轮对话开始时间")
    end_time = Column(DateTime, nullable=True, comment="本轮对话结束时间")
    duration_ms = Column(Integer, nullable=True, comment="总耗时（毫秒）")

    # Token 聚合统计（从所有 llm 类型的 runs 汇总）
    total_input_tokens = Column(Integer, nullable=False, default=0, comment="聚合输入token数")
    total_output_tokens = Column(Integer, nullable=False, default=0, comment="聚合输出token数")
    total_tokens = Column(Integer, nullable=False, default=0, comment="聚合总token数")

    # 扩展信息
    metadata = Column(JSON, nullable=True, comment="扩展元数据（JSON对象，包含 agent_name/model/is_debug/is_scheduled 等）")
```

**metadata 字段约定**：

```json
{
  "agent_name": "天气助手",
  "model": "qwen-max",
  "is_debug": false,
  "is_scheduled": false,
  "scheduled_task_id": null,
  "user_message_preview": "帮我查一下明天的天气"
}
```

### 3.2 trace_runs 表（运行记录表）

一个 trace 下的每个执行步骤对应一条 run 记录。通过 `parent_run_id` 建立树形嵌套关系。

```python
class TraceRun(BaseModel):
    """trace_runs 表 — 运行记录，trace下每个执行步骤一条"""

    __tablename__ = "trace_runs"
    __table_args__ = (
        Index("udx_run_id", "run_id", unique=True),
        Index("idx_trace_id", "trace_id"),
        Index("idx_parent_run_id", "parent_run_id"),
        Index("idx_run_type", "run_type"),
        Index("idx_status", "status"),
        {"comment": "运行记录表，记录trace内每个执行步骤（LLM调用/工具调用/SubAgent等）的详细信息"}
    )

    id = Column(String(36), primary_key=True, default=uuid4, comment="主键ID（UUID）")
    run_id = Column(String(36), nullable=False, unique=True, comment="运行唯一标识（UUID），业务主键")
    trace_id = Column(String(36), nullable=False, comment="关联的追踪ID（关联 trace_records.trace_id）")
    parent_run_id = Column(String(36), nullable=True, comment="父运行ID（关联 trace_runs.run_id），用于树形嵌套，根节点为NULL")

    # 类型与名称
    run_type = Column(String(20), nullable=False, comment="运行类型：chain-编排 / llm-LLM调用 / tool-工具调用 / agent-SubAgent / middleware-中间件")
    name = Column(String(128), nullable=False, comment="可读名称（如工具名、模型名、中间件名）")

    # 状态与时间
    status = Column(String(20), nullable=False, default="running", comment="状态：running-执行中 / completed-完成 / error-错误")
    start_time = Column(DateTime, nullable=False, comment="开始时间")
    end_time = Column(DateTime, nullable=True, comment="结束时间")
    duration_ms = Column(Integer, nullable=True, comment="耗时（毫秒）")

    # 输入输出（全量记录）
    inputs = Column(JSON, nullable=True, comment="输入数据（JSON），内容因 run_type 不同而异")
    outputs = Column(JSON, nullable=True, comment="输出数据（JSON），内容因 run_type 不同而异")
    error = Column(Text, nullable=True, comment="错误信息（仅 status=error 时有值）")

    # Token 统计（仅 run_type=llm 时有值）
    token_usage = Column(JSON, nullable=True, comment="Token统计（JSON对象：input_tokens/output_tokens/total_tokens/cache_read_tokens），仅llm类型有值")

    # 扩展信息
    metadata = Column(JSON, nullable=True, comment="扩展元数据（JSON对象），内容因 run_type 不同而异")
```

### 3.3 各 run_type 的 inputs/outputs/metadata 约定

#### run_type = "chain"（编排节点）

每轮对话的根节点，parent_run_id 为 NULL。

```json
// inputs
{
  "user_message": "帮我查一下明天的天气"
}

// outputs
{
  "ai_message": "明天北京晴，最高温度25°C，最低温度12°C，适合外出。"
}

// metadata
{
  "agent_name": "天气助手",
  "model": "qwen-max"
}
```

#### run_type = "llm"（LLM 调用）

每次 LLM 推理调用，parent_run_id 指向所属的 chain 或 agent 节点。

```json
// inputs
{
  "messages": [
    { "type": "system", "content": "你是一个天气助手..." },
    { "type": "human", "content": "帮我查一下明天的天气" }
  ]
}

// outputs
{
  "content": "我来帮你查询明天的天气。",
  "tool_calls": [
    { "name": "get_weather", "args": { "city": "北京", "date": "2026-03-16" } }
  ]
}

// token_usage
{
  "input_tokens": 520,
  "output_tokens": 180,
  "total_tokens": 700,
  "cache_read_tokens": 100
}

// metadata
{
  "model": "qwen-max",
  "temperature": 0.7,
  "max_tokens": 4096
}
```

#### run_type = "tool"（工具调用）

MCP 工具或内置工具的调用，parent_run_id 指向触发它的 chain 或 agent 节点。

```json
// inputs
{
  "city": "北京",
  "date": "2026-03-16"
}

// outputs
{
  "weather": "晴",
  "temp_high": 25,
  "temp_low": 12,
  "wind": "东北风3级"
}

// metadata
{
  "tool_source": "mcp",
  "mcp_server": "weather-service"
}
```

> 工具调用出错时，`status` = "error"，`error` 字段存储错误信息，`outputs` 可为空。

#### run_type = "agent"（SubAgent 执行）

SubAgent 调用，parent_run_id 指向主 Agent 的 chain 节点。SubAgent 内部的 llm/tool 调用的 parent_run_id 指向此 agent 节点。

```json
// inputs
{
  "task": "查询北京明天天气并给出穿衣建议",
  "subagent_type": "weather_advisor"
}

// outputs
{
  "result": "明天北京晴，建议穿薄外套..."
}

// metadata
{
  "subagent_id": "uuid-xxx",
  "subagent_type": "weather_advisor"
}
```

#### run_type = "middleware"（中间件执行）

中间件执行记录，parent_run_id 指向所属的 chain 节点。

```json
// inputs
{
  "middleware_input": { /* 中间件收到的输入 */ }
}

// outputs
{
  "middleware_output": "completed"
}

// metadata
{
  "middleware_name": "UserProfileMiddleware"
}
```

### 3.4 树形关系示例

一轮典型对话的 Run 树结构：

```
chain: "天气助手" (root, parent_run_id=NULL)          ← trace 的根节点
├── middleware: "UserProfileMiddleware"                ← 用户画像注入
├── llm: "qwen-max" (第1次LLM调用)                    ← AI决定调用工具
│   └── inputs.messages: [system, human]
│   └── outputs.tool_calls: [get_weather]
├── tool: "get_weather"                               ← 工具执行
│   └── inputs: {city, date}
│   └── outputs: {weather, temp_high, ...}
├── llm: "qwen-max" (第2次LLM调用)                    ← AI根据工具结果生成回答
│   └── inputs.messages: [system, human, ai, tool_result]
│   └── outputs.content: "明天北京晴..."
└── middleware: "MemoryUpdateMiddleware"               ← 记忆更新
```

含 SubAgent 的场景：

```
chain: "综合助手" (root)
├── llm: "qwen-max" (第1次)                           ← AI决定委托SubAgent
├── agent: "天气查询Agent"                             ← SubAgent 执行
│   ├── llm: "qwen-flash" (SubAgent第1次LLM)
│   ├── tool: "get_weather"
│   └── llm: "qwen-flash" (SubAgent第2次LLM)
└── llm: "qwen-max" (第2次)                           ← 主Agent汇总回答
```

## 四、TraceCollector 采集服务

### 4.1 定位

新增 `common/common_service/trace_collector.py`，作为独立服务负责 trace/run 数据的 DB 读写。

设计原则：
- **职责单一**：只做 trace 数据的 CRUD，不做 WebSocket 推送、不做事件解析
- **异步非阻塞**：所有 DB 操作使用 async，不阻塞对话流式推送
- **可复用**：后续监控面板直接调用此服务的查询方法

### 4.2 核心接口

```python
class TraceCollector:
    """追踪数据采集服务 — 负责 trace/run 的实时写入和查询"""

    # ==================== Trace 生命周期 ====================

    async def start_trace(
        self,
        trace_id: str,
        session_id: str,
        scene_id: str,
        user_id: str,
        metadata: dict | None = None,
    ) -> None:
        """创建一条 trace 记录（status=running）。

        在每轮对话开始时（Phase 3 之后、Phase 4 之前）由 chat_service 调用。
        """

    async def end_trace(
        self,
        trace_id: str,
        status: str = "completed",
    ) -> None:
        """结束一条 trace 记录。

        更新 end_time、duration_ms、status，并聚合所有子 run 的 token 统计。
        在每轮对话结束时（Phase 5）由 chat_service 调用。
        """

    # ==================== Run 生命周期 ====================

    async def start_run(
        self,
        run_id: str,
        trace_id: str,
        parent_run_id: str | None,
        run_type: str,
        name: str,
        inputs: dict | None = None,
        metadata: dict | None = None,
    ) -> None:
        """创建一条 run 记录（status=running）。

        在各类事件 start 回调中由 event_handler 调用。
        """

    async def end_run(
        self,
        run_id: str,
        status: str = "completed",
        outputs: dict | None = None,
        error: str | None = None,
        token_usage: dict | None = None,
    ) -> None:
        """结束一条 run 记录。

        更新 end_time、duration_ms、status、outputs、error、token_usage。
        在各类事件 end 回调中由 event_handler 调用。
        """

    # ==================== 查询接口 ====================

    async def get_traces_by_session(
        self,
        session_id: str,
    ) -> list[dict]:
        """按 session_id 查询所有 trace 记录（按 start_time 升序）。

        返回 trace 列表，每条包含基本信息和聚合统计。
        供调试页面的 trace 列表使用。
        """

    async def get_runs_by_trace(
        self,
        trace_id: str,
    ) -> list[dict]:
        """按 trace_id 查询所有 run 记录（按 start_time 升序）。

        返回扁平的 run 列表，前端根据 parent_run_id 自行构建树形结构。
        供调试页面的调用树使用。
        """
```

### 4.3 Token 聚合逻辑

`end_trace` 中需要聚合该 trace 下所有 `run_type = 'llm'` 的 run 的 token 统计：

```python
async def end_trace(self, trace_id: str, status: str = "completed") -> None:
    # 1. 查询该 trace 下所有 llm 类型 run 的 token_usage
    # 2. 累加 input_tokens、output_tokens、total_tokens
    # 3. UPDATE trace_records SET
    #    end_time = now(),
    #    duration_ms = (now - start_time) * 1000,
    #    status = status,
    #    total_input_tokens = sum_input,
    #    total_output_tokens = sum_output,
    #    total_tokens = sum_total
```

### 4.4 异常安全

采集服务的 DB 操作不应影响对话流程。所有方法内部需要 try-except，失败时记录 logger.error 但不抛异常：

```python
async def start_run(self, ...):
    try:
        # DB INSERT
    except Exception as e:
        logger.error(f"[TraceCollector] start_run 写入失败: {e} | run_id: {run_id}")
        # 不抛异常，不影响对话流程
```

## 五、event_handler 集成改造

### 5.1 改造思路

当前 `event_handler.py` 中的调试数据收集逻辑（`call_chain_nodes` 列表、`llm_calls` 列表）将被 TraceCollector 的实时写入替代。

**核心变更**：

| 原逻辑 | 新逻辑 |
|--------|--------|
| `call_chain_nodes = [] if is_debug else None` | 移除，不再在内存收集 |
| `llm_calls = [] if is_debug else None` | 移除，不再在内存收集 |
| `if is_debug and call_chain_nodes is not None: call_chain_nodes.append(...)` | 改为 `await trace_collector.start_run(...)` |
| `node["status"] = "completed"; node["end_time"] = ...` | 改为 `await trace_collector.end_run(...)` |
| `return {"content": ..., "call_chain_nodes": ..., "llm_calls": ...}` | 改为 `return final_content`（trace 数据已实时写入 DB） |

### 5.2 process_events 签名变更

```python
# 旧签名
async def process_events(
    event_stream, send_func, cancel_event=None,
    is_debug=False, debug_context=None,

# 新签名

    event_stream, send_func, cancel_event=None,
    is_debug=False, debug_context=None,
```

### 5.5 chain 根节点的处理

`chain` 类型的根节点不是由 LangGraph 事件触发的，需要在 `process_events` 开始和结束时手动创建：

```python
async def process_events(...):
    # 开始时创建 chain 根节点
    chain_run_id = str(uuid.uuid4())
    if trace_collector and trace_id:
        await trace_collector.start_run(
            run_id=chain_run_id,
            trace_id=trace_id,
            parent_run_id=None,  # 根节点
            run_type="chain",
            name=agent_name or "main_agent",
            inputs={"user_message": user_message},
            metadata={"agent_name": agent_name, "model": model_name},
        )

    if is_debug and trace_collector and trace_id:
        await send_func({
            "action": "trace_run_start",
            "data": {
                "run_id": chain_run_id,
                "trace_id": trace_id,
                "parent_run_id": None,
                "run_type": "chain",
                "name": agent_name or "main_agent",
                "status": "running",
                "start_time": start_time_iso,
                "inputs": {"user_message": user_message},
            }
        })

    # ... 处理事件流 ...

    # 结束时关闭 chain 根节点
    if trace_collector and trace_id:
        await trace_collector.end_run(
            run_id=chain_run_id,
            status="completed",
            outputs={"ai_message": final_content},
        )

    if is_debug and trace_collector and trace_id:
        await send_func({
            "action": "trace_run_end",
            "data": {
                "run_id": chain_run_id,
                "status": "completed",
                "outputs": {"ai_message": final_content},
                "end_time": end_time_iso,
                "duration_ms": duration_ms,
            }
        })
```

> **注意**：所有子节点（llm / tool / agent / middleware）的 `parent_run_id` 需要正确指向 `chain_run_id` 或 SubAgent 的 run_id。当前已有 `call_stack` 机制可以复用。

### 9.5 数据获取方式

调试页面的 CallChainTab 数据来源从两个渠道，两个渠道使用**同一个 TraceRun 数据结构**：

**渠道 1：实时 WebSocket 事件（对话进行中）**

对话进行中，前端通过 WebSocket 接收 `trace_run_start` / `trace_run_end` 事件（仅 `is_debug=True` 时推送），在 `useDebugStore` 中维护内存中的 TraceRun 列表，实时更新调用树。

这两个事件的 `data` 结构与 API 返回的 TraceRun 完全一致，前端无需做格式转换。

**渠道 2：API 查询（历史回放）**

对话结束后或切换历史会话时，调用 `POST /debug/trace/list` 和 `POST /debug/trace/detail` 从 DB 获取持久化数据。

**切换策略**：

- 当前活跃会话（WebSocket 连接中）：使用渠道 1 的实时数据
- 历史会话（从会话列表点入）：使用渠道 2 的 API 数据
- 对话完成后：自动从渠道 1 切换到渠道 2（触发一次 API 查询，用 DB 权威数据替换内存数据）

## 十、数据流总览

### 10.1 全量采集数据流（对话进行中）

```
WebSocket 消息到达
  ↓
chat_service.handle_chat
  ├─ Phase 3: assembler.assemble
  ├─ trace_collector.start_trace(...)                    ← INSERT trace_records
  ├─ Phase 4: event_handler.process_events
  │   ├─ 创建 chain 根节点
  │   │   └─ trace_collector.start_run(chain)            ← INSERT trace_runs
  │   ├─ on_chain_start (middleware)
  │   │   └─ trace_collector.start_run(middleware)       ← INSERT trace_runs
  │   ├─ on_chain_end (middleware)
  │   │   └─ trace_collector.end_run(middleware)         ← UPDATE trace_runs
  │   ├─ on_chat_model_start
  │   │   └─ trace_collector.start_run(llm)             ← INSERT trace_runs
  │   ├─ on_chat_model_stream (多次)
  │   │   └─ 仅 WebSocket 推送，不写 DB
  │   ├─ on_chat_model_end
  │   │   └─ trace_collector.end_run(llm)               ← UPDATE trace_runs
  │   ├─ on_tool_start
  │   │   └─ trace_collector.start_run(tool)            ← INSERT trace_runs
  │   ├─ on_tool_end
  │   │   └─ trace_collector.end_run(tool)              ← UPDATE trace_runs
  │   └─ 关闭 chain 根节点
  │       └─ trace_collector.end_run(chain)             ← UPDATE trace_runs
  └─ trace_collector.end_trace(...)                      ← UPDATE trace_records (聚合token)
```

### 10.2 调试页面数据流（历史回放）

```
用户在调试页面选择历史会话
  ↓
前端调用 POST /debug/trace/list { session_id }
  ↓
admin_backend → trace_collector.get_traces_by_session()
  ↓
返回该会话所有轮次的 trace 列表
  ↓
用户展开某一轮
  ↓
前端调用 POST /debug/trace/detail { trace_id }
  ↓
admin_backend → trace_collector.get_runs_by_trace()
  ↓
返回该轮所有 runs（扁平列表）
  ↓
前端 buildRunTree() 构建树形结构
  ↓
渲染调用树（第一层），点击节点 → 钻入详情页（第二层），返回键回到调用树
```

## 十一、与其他模块的数据边界

| 边界 | 说明 |
|------|------|
| **debug_snapshot** | trace 数据从 debug_snapshot 中完全剥离，debug_snapshot 仅保留配置快照（prompt、tools、skills、user_profile） |
| **message_events** | message_events 表负责消息事件持久化（用于会话回放），trace 表负责链路追踪，两者独立 |
| **sessions** | trace_records 通过 session_id 关联 sessions 表，不新增外键 |
| **checkpoints** | trace 数据与 checkpoint 独立，checkpoint 负责对话历史持久化，trace 负责执行链路追踪 |
| **scheduled_tasks** | 定时任务触发的对话同样采集 trace，通过 metadata.is_scheduled 和 metadata.scheduled_task_id 标识 |
