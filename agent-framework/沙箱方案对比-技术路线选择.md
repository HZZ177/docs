# Skill 沙箱执行方案 — 技术路线对比

> 本文档对比两套技术路线，供小组讨论决策使用。
> 两套方案是**平等的技术选择**，各有适用场景。

---

## 方案速览

| 维度 | 方案 A：DeerFlow 风格 | 方案 B：Agent Sandbox + OSS |
|------|---------------------|---------------------------|
| **一句话** | 1 thread = 1 Pod，空闲超时后销毁 | 共享 Pod 池，通过 OSS 路径识别用户身份 |
| **生命周期** | 空闲超时级（首次调用创建，10 分钟空闲后销毁） | 池级（Pod 与用户无绑定，可复用） |
| **绑定粒度** | thread_id（确定性哈希 → sandbox_id） | 无绑定（OSS 路径决定身份） |
| **代码执行方式** | Pod 内 HTTP 服务 + NodePort 暴露 | Pod 内 HTTP 服务 |
| **参考项目** | 字节跳动 DeerFlow | K8s SIG Apps Agent Sandbox |
| **存储方式** | hostPath（宿主机磁盘） | emptyDir + OSS 对象存储 |
| **Pod 调度** | 固定到特定节点 | 任意节点 |
| **预热池** | 无（首次 Skill 调用时按需创建） | 有（SandboxWarmPool） |
| **资源消耗模型** | 与活跃 thread 数线性相关 | 与池大小相关（相对固定） |
| **跨进程恢复** | 文件锁 + 状态文件 + 后端 discover | Controller 管理 |
| **CRD** | 无（自定义 Provisioner） | 4 个标准 CRD |
| **Python SDK** | kubernetes（自己封装） | k8s-agent-sandbox（官方） |
| **gVisor 支持** | 无 | 原生支持 |

## 架构对比图

### 方案 A 数据流

```
Agent Backend → AioSandboxProvider → Provisioner → K8s API → Pod(HTTP 服务 :8080)
                 (三层一致性检查)                              │         ↑
                                                         hostPath 挂载  │
                                                              │    HTTP API 调用
                                                         宿主机磁盘    │
                                                         (数据持久)    │
                                              Agent Backend ──────────┘
                                              (通过 NodePort 直接调用 Pod)
```

### 方案 B 数据流

```
Agent Backend → SandboxClient → K8s API → Agent Sandbox Controller
                                               │
                                          预热池分配 Pod
                                               │
                                    ┌──────────┼──────────┐
                                    │          │          │
                                 Init容器   主容器     Sidecar
                                 OSS恢复   执行Skill  OSS同步
                                    │                     │
                                    └──────── OSS ────────┘
```

## 核心维度对比

### 0. 生命周期模型（两套方案最本质的区别）

| 维度 | 方案 A | 方案 B |
|------|--------|--------|
| **Pod 与用户关系** | 1 thread = 1 Pod（强绑定） | 无绑定（共享池） |
| **Pod 创建时机** | 首次 Skill 调用时按需创建（lazy_init） | 预热池提前创建 |
| **Pod 销毁时机** | 空闲超时（默认 10 分钟）或进程关闭 | 执行完归还池（或销毁后池自动补充） |
| **用户身份识别** | thread_id → sha256 → sandbox_id（确定性哈希） | OSS 路径（sessions/{user_id}/{session_id}/） |
| **代码执行方式** | Pod 内 HTTP 服务 + NodePort，backend 通过 HTTP API 调用 | Pod 内 HTTP 服务 |
| **同一对话内多次执行** | 复用同一 Pod（进程内缓存命中，HTTP API 调用） | 可能分配到不同 Pod |
| **跨进程恢复** | 文件锁 + sandbox.json + 后端 discover | Controller 管理 |
| **Pod 数量** | = 活跃 thread 数（10 个活跃 thread = 10 个 Pod） | = 池大小（固定，如 5 个） |
| **资源弹性** | 线性增长，但空闲 Pod 会超时释放 | 池大小可调，超出排队等待 |
| **空闲开销** | 空闲 Pod 存活最多 10 分钟后自动回收 | 执行完归还，不存在空转 |

```
方案 A 的时间线（以 thread-A 为例，基于 DeerFlow 源码）：

  ──首次Skill调用──Skill2──Skill3────(空闲)──────超时10min──
  │  创建 Pod-A    │       │         │           销毁 Pod-A │
  │  ┌─────────── Pod-A 常驻(HTTP服务) ────────┐           │
  │  │ HTTP调用    HTTP调用  HTTP调用   (idle)   │           │
  │  └─────────────────────────────────────────┘           │

  空闲期间 Pod 也在运行，但最多存活 10 分钟
  同一 thread 再次调用时会重新创建 Pod + 挂载同一 hostPath

方案 B 的时间线（以用户 A 为例）：

  ──请求 Skill1──────请求 Skill2──────请求 Skill3──
  │  取出 Pod-X      │  取出 Pod-Y      │  取出 Pod-X   │
  │  OSS 恢复        │  OSS 恢复        │  OSS 恢复     │
  │  执行             │  执行             │  执行          │
  │  归还 Pod-X      │  归还 Pod-Y      │  归还 Pod-X   │

  用户空闲期间不占用任何 Pod
  每次执行前需要从 OSS 恢复数据（额外延迟）
```

### 1. 性能

| 指标 | 方案 A | 方案 B |
|------|--------|--------|
| 沙箱创建 | 3-10 秒（实时创建 Pod） | ~100ms（预热池分配） |
| 工作区恢复 | 0 秒（hostPath 直接挂载） | 1-30 秒（从 OSS 下载） |
| 文件读写 | 本地磁盘 IOPS | emptyDir IOPS（同级别） |
| 产物持久化 | 0 秒（已在磁盘上） | 异步同步（30 秒间隔） |
| **首次执行总延迟** | **3-10 秒** | **1-31 秒**（含 OSS 恢复） |
| **再次执行（Pod 存活）** | **< 1 秒** | **< 1 秒** |

分析：
- 方案 A 的首次瓶颈是 Pod 创建（无预热池），但同一 thread 内后续执行命中进程内缓存、直接 HTTP API 调用，极快
- 方案 B 的首次瓶颈是 OSS 恢复（即使预热池秒级分配 Pod，Init 容器仍需下载）
- **关键差异**：方案 A 空闲超时前 Pod 常驻，「再次执行」是零开销 HTTP 调用；方案 B 每次执行可能分配不同 Pod，都需要经历 OSS 恢复
- 方案 A 适合「一次对话多轮 Skill 调用」场景；方案 B 更适合「短时高并发、执行完就走」场景

### 2. 扩展性

| 指标 | 方案 A | 方案 B |
|------|--------|--------|
| 单节点上限 | 受限于单机资源 | 不适用 |
| 多节点支持 | 需要节点亲和性 | 原生支持 |
| 节点扩容 | 需要数据迁移或 NFS | 加节点即可 |
| 存储扩容 | 受限于本地磁盘 | OSS 理论无限 |
| **推荐并发上限** | **< 200** | **> 1000** |

### 3. 可用性

| 指标 | 方案 A | 方案 B |
|------|--------|--------|
| 节点故障 | 数据可能丢失 | 数据在 OSS，不影响 |
| 磁盘故障 | 数据丢失 | 数据在 OSS，不影响 |
| OSS 故障 | 不影响 | 无法创建新沙箱 |
| Controller 故障 | 不适用 | 已有 Pod 不影响，新建受阻 |

### 4. 安全性

| 指标 | 方案 A | 方案 B |
|------|--------|--------|
| 基础隔离 | Pod 级 | Pod 级 |
| SecurityContext | 手动配置 | 手动配置 |
| gVisor | 需自行集成 | runtimeClassName 一行搞定 |
| Kata Containers | 需自行集成 | runtimeClassName 一行搞定 |
| hostPath 风险 | 有（需严格限制路径） | 无（使用 emptyDir） |

### 5. 开发运维

| 指标 | 方案 A | 方案 B |
|------|--------|--------|
| 代码量 | ~300 行 Provisioner | ~200 行 Service + OSS 镜像 |
| 需自行开发 | Provisioner 服务 | OSS Sync 镜像 |
| 需安装组件 | 无额外组件 | Agent Sandbox Controller + OSS |
| 调试难度 | 低（SSH 看文件） | 中（需查看 OSS + Pod 日志） |
| 学习成本 | 低（K8s 基础） | 中（CRD + OSS SDK） |
| 维护成本 | 低 | 中 |

### 6. 成本

| 项目 | 方案 A | 方案 B (OSS 内网) | 方案 B (MinIO) |
|------|--------|------------------|---------------|
| 计算资源 | ¥8000/月 | ¥8000/月 | ¥8000/月 |
| 本地存储 | ¥400/月 | ¥200/月 | ¥200/月 |

## 讨论要点

### 必须回答的问题

1. **用户规模预期**
   - < 200 并发 → 方案 A 足够
   - > 500 并发 → 方案 B 更合适
   - 中间地带 → 取决于增长预期

2. **数据丢失容忍度**
   - 沙箱数据可重新生成 → 方案 A
   - 必须保证数据不丢失 → 方案 B

3. **团队 K8s 经验**
   - 初级（只会 kubectl） → 方案 A
   - 中级（了解 CRD、Operator） → 均可
   - 高级（有 Operator 开发经验） → 方案 B

4. **是否需要执行不可信代码**
   - 只执行团队编写的 Skill → 方案 A 的隔离足够
   - 需要执行 LLM 生成的代码 → 方案 B 的 gVisor 更安全

5. **单节点 vs 多节点部署**
   - 单节点 → 方案 A 天然适合
   - 多节点 → 方案 B 更合适

6. **Alpha 阶段是否可接受**
   - 需要稳定、可预测 → 方案 A（完全自控）
   - 愿意承担 API 变动风险 → 方案 B

### 混合方案可能性

如果两套方案各有吸引力，可以考虑混合方案：

```
混合方案 C：自定义 Provisioner + OSS

- 使用方案 A 的 Provisioner 服务（不引入 Agent Sandbox CRD）
- 使用方案 B 的 OSS 持久化逻辑（Init + Sidecar）
- 获得多节点支持，但不引入 Alpha 阶段的 CRD
- 失去预热池，但保持架构可控
```
